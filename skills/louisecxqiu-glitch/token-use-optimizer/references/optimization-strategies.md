# Token 优化策略详解

> token-optimizer Skill 的完整操作手册。SKILL.md 负责调度，本文件负责"怎么做"。

---

## 基线计算公式

```
固定开销/轮 ≈ (always-on rules 总行数 × 1.3)
             + (memory 条目数 × 平均行数 × 1.3)
             + (system prompt 固定开销 ~2000)
```

> 1.3 是中文内容的 token/字符经验系数（英文约 1.0）。

手动盘点清单：

```
□ Always-on Rules 数量和总行数
□ Requestable Rules 数量（确认不会误触）
□ Memory 条目数量和估算 token 数
□ 知识库入口文件行数
□ 常驻 Skill 指令长度
```

## L1: Rules 优化策略

### Always-on vs Requestable 判断标准

**核心原则：只有"身份铁律"级别的规则才配 always-on。**

| 规则类型 | always-on？ | 理由 |
|---------|:-----------:|------|
| 身份定义（我是谁、核心行为） | ✅ | 每轮都需要 |
| 自我进化协议 | ✅ | 每次对话开头必跑 |
| 模式切换（工作/闲聊） | ✅ | 需要实时判断 |
| 任务完成通知 | ❌ | 只在完成实质任务时需要 |
| 产出物归档 | ❌ | 只在生成产出物时需要 |
| 双写同步 | ❌ | 只在写 memory 时需要 |
| 部署规则 | ❌ | 只在部署时需要 |

### 降级为 Requestable 的操作

修改 `.mdc` 文件的 frontmatter：

```yaml
---
description: >
  [规则职责一句话]。
  当需要 [触发场景] 时触发。
alwaysApply: false
---
```

**description 写法要点**：第一句说职责，第二句写触发条件（"当需要…时触发"），包含 3-5 个关键词帮助自动匹配。

### 规则合并策略

当两条规则内容重叠 >50% 时：
1. 确定主规则（内容更完整）
2. 合并另一条的独有内容
3. 删除冗余规则文件
4. 验证合并后功能完整

### 规则内容瘦身技巧

| 技巧 | 操作 | 典型收益 |
|------|------|---------|
| 示例外迁 | 详细示例移到 references/，规则只留引用 | -20~40% |
| 压缩合并 | 多个短节合并为 bullet list | -15~30% |
| 表格替代 | 用表格替代段落式描述 | -10~20% |
| 引用替代 | "详见 `XXX.md`" 替代内联长段 | -50~70%（该节） |

### 检查项汇总

| 检查项 | 判断标准 | 处理方式 |
|-------|---------|---------|
| 重复规则 | 内容重叠 >50% | 合并为一条 |
| 过长规则 | 单条 >150 行 | 拆分：核心留 rule，详情移 references/ |
| 可降级规则 | 非每次对话必需 | 改 requestable，加 description |
| 僵尸规则 | 创建后从未触发 | 评估删除或合并 |

## L2: Memory 优化策略

### Token 成本结构

每条 Memory 每轮注入 system prompt：
- 事实性（1 行）≈ 30-50 tokens
- 结构化经验（4 行）≈ 100-200 tokens
- 长描述（5+ 行）≈ 200-400 tokens

15 条 memory 估算：15 × 100 ≈ 1,500 tokens/轮

### 清理决策树

```
这条 Memory 还需要吗？
├── 内容已被 Rule 完整覆盖？ → YES → 删除（双重计费）
├── 一次性事件记录？ → YES → 删除（无行动价值）
├── 引用的工具/配置还存在吗？ → NO → 删除（过时）
├── 用户明确要求的红线？ → YES → 保留
├── 2+ 次试错的经验教训？ → YES → 保留（结构化格式）
└── 当前仍有效的事实？ → YES → 保留（精简为 1 句）
```

### 格式规范

**事实性 Memory**（1 句话）：
```
项目 X 的唯一运行环境路径是 ~/CodeBuddy/project-x/
```

**经验教训 Memory**（结构化 4 段）：
```
[经验] 标题
- 场景：什么情况下遇到的
- 根因：为什么会发生
- 解法：下次怎么做
- 适用范围：哪些场景通用
```

**红线**：
- ❌ 超过 5 行的 Memory → 拆分或移到 Rule/Knowledge
- ❌ 与 Rule 重复的内容
- ❌ 日志型记录 → 放 daily log

### 目标

Memory 条目控制在 ~15 条以内，每条不超过 3 行。

## L3: Knowledge 优化策略

### 巨型文件拆分模式

**Before**：一个 800+ 行的入口文件包含路由表 + 联想 + 索引 + 示例。

**After**：
```
入口文件.md (≤200 行) ← 纯路由表/索引
  ├── 详细内容A.md  ← 场景A时按需加载
  ├── 详细内容B.md  ← 场景B时按需加载
  └── 各域文件/*.md ← 各域详细文件按需加载
```

**拆分标准**：
- 入口文件控制在 200 行以内
- "触发条件不同"的内容块拆为独立文件
- 文件名即触发词（如 `BOOK-INDEX.md` 只在推荐书籍时加载）

### "两跳"加载模式

```
用户问题 → Step 1: 读路由表（≤200行，~300 tokens）
         → Step 2: 按路由读具体域文件（按需）
```

避免一次性加载整个知识库。

## L4: 运行时行为优化

### 精准读取 vs 全量读取

```python
# ❌ 全量读取 2000 行文件（~3000 tokens）
read_file("large_file.py")

# ✅ 只读需要的 50 行（~75 tokens）
read_file("large_file.py", offset=100, limit=50)
```

适用：确认文件头尾、读已知位置函数、检查配置特定字段。

### 主动 /compact

| 对话轮数 | 建议 |
|---------|------|
| < 5 轮 | 正常对话 |
| 5-10 轮 | 留意上下文大小 |
| > 10 轮 | **主动建议 /compact** |
| > 15 轮 | 强烈建议拆分新会话 |

### Subagent 分流

大规模代码搜索优先使用 `code-explorer` subagent：
- 搜索结果不进入主 agent 上下文
- 大幅减少主对话 token 消耗
- 适合：代码结构理解、跨文件搜索、大范围重构规划

### Skill 按需加载

- 不同时加载多个 Skill
- 用完的 Skill 不会自动卸载，注意上下文膨胀
- 特定步骤需要的 Skill，在该步骤再加载

## 优化收益计算模板

```
优化前：
  Always-on Rules:  [A] tokens/轮
  Memory:           [B] tokens/轮
  System Prompt:    ~2,000 tokens/轮
  ────────────────────────
  固定开销合计:     [A+B+2000] tokens/轮

优化后：
  Always-on Rules:  [A'] tokens/轮
  Memory:           [B'] tokens/轮
  System Prompt:    ~2,000 tokens/轮
  ────────────────────────
  固定开销合计:     [A'+B'+2000] tokens/轮

  节省:    [(A+B) - (A'+B')] tokens/轮
  节省率:  [节省/原总] × 100%
  10轮累计: [节省×10] tokens
```

## 守护机制：防止开销回弹

1. **Rule 创建审批**：新建 always-on rule 前先问"真的每次对话都需要？"
2. **Memory 定期清理**：每月清理过时 Memory
3. **月度 Token 审计**：运行 `token_audit.py` 检查膨胀
4. **进化协议守护**：在 self-evolution 机制中持续关注 Token 优化
