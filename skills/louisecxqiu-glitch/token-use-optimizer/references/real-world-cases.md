# Token 优化实战案例

> 可选参考。展示一个真实 AI Agent 项目的完整优化过程，帮助理解四层诊断框架的实际应用。

---

## 案例概况

| 维度 | 说明 |
|------|------|
| 项目类型 | CodeBuddy IDE 中的 AI Agent 体系 |
| 架构 | Rules + Memory + Knowledge + Skills |
| 项目阶段 | 运行 3 个月，从未系统性清理 |
| 优化目标 | 降低每轮固定 Token 开销 |

## 优化前诊断

| 组件 | 数量 | 估算 tokens/轮 |
|------|------|:-------------:|
| Always-on Rules | 7 条 | ~5,500 |
| Memory 条目 | 20+ 条 | ~4,000 |
| System Prompt | — | ~2,000 |
| **合计** | | **~11,500** |

发现的六类问题：
1. 两条 Rule 内容重叠 >80%
2. 三条 Rule 并非每次对话都需要，但设为 always-on
3. Memory 与 Rule 双重覆盖同一内容
4. 存在一次性事件记录型 Memory
5. 知识库入口文件 863 行，每次全量加载
6. Rules 中内联大量示例和关联表

## 第一轮：Rules + Memory 优化

| 操作 | 手法 | 收益 |
|------|------|:----:|
| 合并重复 Rule | 删除内容已被另一条覆盖的冗余规则 | -800 tokens/轮 |
| 3 条 Rule 降级 | 从 always-on 改为 requestable，加 description 触发词 | -1,200 tokens/轮 |
| Memory 精简 | 删除 10 条（被 Rule 覆盖/一次性/过时） | -1,500 tokens/轮 |
| **小计** | | **-3,500 tokens/轮** |

## 第二轮：Knowledge + 内容瘦身

| 操作 | 手法 | 收益 |
|------|------|:----:|
| 知识库入口拆分 | 863→200 行路由表 + 按需加载的详细文件 | 知识调用 -60% |
| Rule A 瘦身 | 示例外迁 + 段落合并为 bullet（125→93 行，-26%） | -200 tokens/轮 |
| Rule B 瘦身 | 机制去重 + 合并压缩（172→119 行，-31%） | -400 tokens/轮 |
| **小计** | | **-600 tokens/轮 + 知识调用 -60%** |

## 最终结果

| 指标 | 优化前 | 优化后 | 变化 |
|------|:-----:|:-----:|:----:|
| Always-on Rules | 7 条 | 4 条 | -3 |
| Memory 条目 | 20+ 条 | ~13 条 | -7 |
| 每轮固定开销 | ~11,500 | ~8,000 | **-30%** |
| 10 轮累计 | ~115,000 | ~80,000 | **省 ~35,000** |
| 知识库入口 | 863 行 | 200 行 | -77% |

## 沉淀的核心原则

1. **身份铁律 always-on，行为指南 on-demand** — 判断 Rule 级别的第一性原理
2. **Memory 和 Rule 不重复** — 同一内容只在一处，优先放 Rule
3. **知识库"路由表 + 按需加载"** — 入口轻量，详情按需
4. **精准读取 > 全量读取** — 已知文件用 offset/limit
5. **主动 /compact > 被动溢出** — >10 轮主动压缩

## 经验教训

1. **温水煮青蛙**：每次加一条 rule/memory 都觉得无所谓，3 个月累积效应惊人。需定期审计。
2. **双重计费最隐蔽**：Memory 和 Rule 覆盖同一内容是最常见浪费，因两者由不同场景创建。
3. **降级 ≠ 删除**：Requestable rule 在需要时仍能被触发词唤回，不丢失功能。
4. **大文件拆分 ROI 最高**：800 行拆成 200 行路由表 + 按需加载，效果最显著。
5. **守护机制必须建立**：优化完不建守护，几个月后又回原点。
