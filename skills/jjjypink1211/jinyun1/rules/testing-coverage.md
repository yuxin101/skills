# Go Test Coverage Analysis

**Impact: HIGH** | **Category: testing** | **Tags:** go, test, coverage, unit-test, ci/cd

## Overview

This rule defines standards for analyzing Go test coverage data generated by the official `go test` tool, ensuring adequate test coverage for critical business logic and identifying untested code paths.

## Why This Matters

- Low test coverage increases risk of undiscovered bugs in production
- Comprehensive coverage ensures core business logic is validated
- Standardized coverage analysis enables consistent quality across Go projects
- Coverage data helps prioritize test writing for high-risk code

## Required Command

Agent MUST execute this command to generate coverage data:

```bash
go test "./..." \
    -v \
    -coverprofile="coverage.out" \
    -covermode=count \
    -gcflags=-l \
    -json > test-report.json
```
### Command Explanation

Flag	Purpose

-v	Verbose output (includes test names and results)

-coverprofile="coverage.out"	Generate line-by-line coverage data

-covermode=count	Track number of times each line is executed (enables branch coverage analysis)

-gcflags=-l	Disable inlining to get accurate coverage for small functions

-json	Output test results in JSON format (easy to parse)

### Coverage Metrics to Analyze

1. Line Coverage

Definition: Percentage of code lines executed by tests

Calculation: (number of covered lines) / (total executable lines) * 100

Critical code paths (HTTP handlers, business logic, database operations) require ≥80% coverage

2. Function Coverage

Definition: Percentage of functions with at least one test

Calculation: (number of covered functions) / (total functions) * 100

All exported functions (uppercase) require ≥90% coverage

3. Branch Coverage

Definition: Percentage of code branches (if/else, switch, for) executed by tests

Calculation: (number of covered branches) / (total branches) * 100

Core conditional logic requires ≥75% coverage

### Coverage Risk Levels
| Risk Level | Line Coverage | Function Coverage | Action |

| CRITICAL | < 60% | < 60% | Block merge; require immediate test additions |

| HIGH | 60% - 79% | 60% - 79% | Fix before merge; focus on core logic coverage |

| MEDIUM | 80% - 89% | 80% - 89% | Accept with TODO; improve edge case coverage |

| LOW | ≥ 90% | ≥ 90% | Accept; optional minor improvements | 

❌ Incorrect (Low Coverage)
```go
运行
// service/order.go (55% line coverage)
func calculateTotal(price float64, quantity int, tax float64) float64 {
    if quantity <= 0 { // Uncovered branch
        return 0
    }
    subtotal := price * float64(quantity)
    if tax > 0 { // Uncovered branch
        subtotal += subtotal * tax
    }
    return subtotal
}
// Only tests happy path (quantity=5, price=10, tax=0.1)
// No tests for quantity ≤0 or tax=0
```
✅ Correct (High Coverage)
```go
运行
// service/order.go (95% line coverage)
func calculateTotal(price float64, quantity int, tax float64) float64 {
    if quantity <= 0 {
        return 0
    }
    subtotal := price * float64(quantity)
    if tax > 0 {
        subtotal += subtotal * tax
    }
    return subtotal
}

// Test covers all branches
func TestCalculateTotal(t *testing.T) {
    tests := []struct {
        name     string
        price    float64
        quantity int
        tax      float64
        want     float64
    }{
        {"happy path", 10.0, 5, 0.1, 55.0},
        {"zero quantity", 10.0, 0, 0.1, 0.0},
        {"no tax", 10.0, 5, 0.0, 50.0},
        {"negative quantity", 10.0, -2, 0.1, 0.0},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got := calculateTotal(tt.price, tt.quantity, tt.tax)
            if got != tt.want {
                t.Errorf("calculateTotal() = %v, want %v", got, tt.want)
            }
        })
    }
	
}
```

### Test Failure Analysis

** Extract from test-report.json: ** 

	1.Failed test names and packages
	2.Error messages and stack traces
	3.Specific line numbers of test failures
	3.Recommendations to fix failing tests
	
### Best Practices
	Focus coverage on business logic: Prioritize testing core domain logic over utility functions
	Table-driven tests: Use Go's table-driven test pattern to cover multiple scenarios
	Mock external dependencies: Use testify/mock to mock databases/APIs for unit tests
	Integration tests for critical paths: Combine unit tests with integration tests for HTTP handlers
	Coverage in CI/CD: Enforce minimum coverage thresholds in CI pipelines
	Exclude non-critical code: Use //go:noinline or build tags to exclude boilerplate code from coverage
