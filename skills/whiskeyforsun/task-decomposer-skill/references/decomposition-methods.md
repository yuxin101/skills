# 任务拆解方法论

## 一、拆解原则

### 1. 最小可执行原则
- 每个任务应该是一个人可以在短时间内独立完成的
- 理想时间：2-5 分钟
- 避免：需要多人协作、需要长时间（>1小时）的任务

### 2. 清晰输入输出原则
- 每个任务必须有明确的输入
- 每个任务必须有明确的输出
- 避免：输入输出模糊的任务

### 3. 独立可验收原则
- 每个任务可以独立测试和验收
- 不依赖其他任务完成才能验证
- 避免：需要等待其他任务才能验证的任务

### 4. 无循环依赖原则
- 任务之间不应该有循环依赖
- A → B → C 是可以的
- A → B, B → A 是不行的

## 二、拆解方法

### 1. 按业务流程拆解

适用于：业务逻辑复杂、流程清晰的需求

```
业务流程：用户下单 → 库存检查 → 支付 → 发货

拆解结果：
1. 创建订单
2. 检查库存
3. 调用支付
4. 触发发货
```

### 2. 按技术层次拆解

适用于：分层架构的系统

```
技术层次：Controller → Service → Repository → Entity

拆解结果：
1. 开发 Controller 层
2. 开发 Service 层
3. 开发 Repository 层
4. 创建 Entity 类
```

### 3. 按模块/领域拆解

适用于：大型系统、多模块项目

```
模块划分：用户模块、订单模块、支付模块

拆解结果：
1. 开发用户模块
2. 开发订单模块
3. 开发支付模块
```

### 4. 按变更类型拆解

适用于：功能扩展、重构、优化

```
变更类型：新增、修改、删除、优化

拆解结果：
1. 新增功能 A
2. 修改现有功能 B
3. 删除废弃功能 C
```

## 四、时间估算

### 1. 经验估算法

基于历史数据进行估算：
- 记录每个类似任务的实际耗时
- 计算平均值作为估算基准
- 根据复杂度进行调整

### 2. 三点估算法

```
最可能时间 (M) = 正常情况下的时间
乐观时间 (O) = 一切顺利的时间
悲观时间 (P) = 最坏情况的时间

估算时间 = (O + 4M + P) / 6
```

### 3. 功能点估算法

根据功能复杂度评估工作量：
- 简单功能：1-2 分
- 一般功能：3-5 分
- 复杂功能：6-10 分
- 每分 = 0.5-1 小时

## 五、依赖管理

### 1. 依赖类型

**硬依赖（必须完成）：**
- A 必须在 B 之前完成
- A 未完成，B 无法开始

**软依赖（建议完成）：**
- A 建议在 B 之前完成
- A 未完成，B 也可以开始但效率降低

### 2. 依赖可视化

```
任务依赖图：
┌─────┐     ┌─────┐     ┌─────┐
│ 任务1 │ ──→ │ 任务2 │ ──→ │ 任务3 │
└─────┘     └─────┘     └─────┘
     ↑           ↓
     │     ┌─────┐
     └──── │ 任务4 │
           └─────┘

关键路径：任务1 → 任务2 → 任务3
```

### 3. 并行化分析

识别可以并行执行的任务：
- 没有依赖关系的任务可以并行
- 依赖同一个前置任务的任务可以并行
- 并行任务越多，效率越高

## 六、质量检查

### 1. 任务完整性检查

- [ ] 每个任务都有清晰的描述
- [ ] 每个任务都有明确的输入输出
- [ ] 每个任务都有验收标准
- [ ] 所有任务加起来覆盖完整需求

### 2. 任务粒度检查

- [ ] 每个任务都能在 2-5 分钟内完成
- [ ] 没有任务需要超过 1 小时
- [ ] 没有任务太简单（<1分钟）

### 3. 依赖合理性检查

- [ ] 没有循环依赖
- [ ] 依赖关系清晰明确
- [ ] 关键路径长度合理

### 4. 优先级合理性检查

- [ ] 核心功能优先级最高
- [ ] P0 任务不超过总任务的 30%
- [ ] 优先级与业务价值匹配
