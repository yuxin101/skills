## 一、概述

OpenNexum 是一个运行在 OpenClaw 上的多 agent 编排框架，专为**单人指挥、多 agent 并行执行**的工作模式设计。

它的核心理念是：**合约驱动执行，知识持续沉淀**。每项工作在启动前先确立可验证的完成标准（Contract），完成后自动将经验写回系统供后续复用。随着时间推移，系统变得越来越了解你的项目和工作习惯。

### 设计背景

OpenNexum 在早期 agent 编排实践的基础上，吸收了 OpenAI 和 Anthropic 在智能体工程上的核心经验：

- **Repo 是唯一的 source of truth**：agent 看不到的信息等于不存在，所有知识必须结构化存储在项目里
- **Generator 与 Evaluator 分离**：做的人不应该评判自己的输出，独立 evaluator 是质量的真正保障
- **harness 组件应当可裁剪**：每个组件都是对模型能力不足的补偿假设，随模型进化可按需删减

## 二、定位与边界

### 2.1 目标用户

**OpenNexum 专为个人开发者设计**，不是通用 multi-agent 平台。

适用场景：
- 个人独立工作，无需多人协作模式
- 项目代码量 ≤ 20 万行，单次开发周期 ≤ 1 个月
- 使用 OpenClaw 作为主控层，调度 Claude Code / Codex / Google CLI 等 CLI agent
- 本机单机运行（Mac Mini），使用 tmux 管理 agent 进程

### 2.2 任务类型

OpenNexum 支持三类工作，共享同一套编排机制：

| 类型 | type 枚举值 | 示例 |
|------|------------|------|
| **开发类** | `coding` | 功能开发、Bug 修复、重构、测试、部署 |
| **日常任务类** | `task` | 数据整理、研究报告、信息汇总、文档翻译 |
| **创作类** | `creative` | 写作、内容生成、脚本创作、设计描述 |

> **Phase 1 范围**：仅支持 `coding` 类型。`task` 和 `creative` 类型的 Contract 规范和 evaluator 策略在 Phase 4 实现。

### 2.3 非目标

- 不支持多人团队协作
- 不支持分布式多机部署
- 不追求"通用万能"，有明确的用户边界

## 三、核心设计

### 3.1 Contract-Driven 执行

所有工作在启动前必须建立 **Task Contract**：在任何内容被生成之前，先对齐"什么叫完成"。

Contract 定义：
- **Scope**：要做什么、改哪些文件、明确禁止修改的范围
- **Deliverables**：具体交付物清单
- **Eval Strategy**：用什么方式验证
- **Criteria**：逐条可验证的通过标准

Contract 是 generator 和 evaluator 共同可见的完成定义，不是 orchestrator 单方面写的 prompt。**Contract 一旦创建即为静态，不在执行过程中修改**（运行态信息存储在 active-tasks.json）。

### 3.2 Generator ≠ Evaluator（GAN 原则）

执行 agent（generator）和评估 agent（evaluator）严格分离：

- generator 专注完成任务
- evaluator 独立验证，不参与执行
- evaluator 被显式调校为"怀疑主义"：任务是找问题，而非确认没问题
- fail 时必须给出具体文件路径、函数名、期望行为 vs 实际行为，不允许模糊结论

### 3.3 知识持续沉淀

**对 agent 来说，context 里看不到的信息等于不存在。**

OpenNexum 通过两层结构确保知识可见、可演化：

**项目层**：每次任务完成后，agent 自动将遇到的问题和发现写入 `docs/lessons/`；harvest 流水线将 lessons 整理回 `AGENTS.md`，让下一个任务的 agent 能看到。

**Skill 层（OpenClaw）**：跨项目的通用规范和反模式库存储在 skill 内部；项目初始化时自动注入到项目结构中；gardener agent 定期将多个项目的经验蒸馏回 skill 层。

### 3.4 渐进式披露（Progressive Disclosure）

AGENTS.md 是目录，不是说明书，保持约 100 行。

> "给 agent 一张地图，而不是一本 1000 页的手册"

详细内容通过 AGENTS.md 的索引按需加载。Gardener agent 的核心职责之一是维护 AGENTS.md 在 ~100 行以内——低频使用的条目会被降级到 `docs/lessons/` 详细文档，AGENTS.md 只保留索引。

### 3.5 机械约束优于 Prompt 约束

对于不变的规则（提交格式、文件命名、架构层级），用 linter/CI 机械强制执行，error message 里直接写修复指令。规则一旦编码，立即对所有 agent 生效，无需反复在 prompt 中叮嘱。

## 四、系统架构

### 4.1 分层结构

```
┌───────────────────────────────────────────────────────┐
│  人类（Telegram / 终端）                               │
│  提需求 · 审核 Contract · 处理升级问题                 │
└────────────────────┬──────────────────────────────────┘
                     │
┌────────────────────▼──────────────────────────────────┐
│  OpenClaw — Orchestrator（编排层）                     │
│                                                        │
│  • 接收需求，制定 Task Contract                        │
│  • 决定 eval strategy，dispatch generator              │
│  • 处理 evaluator 结果（pass/fail 循环）               │
│  • 管理知识流：lesson 写入 → harvest → 同步            │
│  • Telegram 通知人类关键节点                            │
└──┬──────┬──────┬──────┬──────┬──────┬─────────────────┘
   │      │      │      │      │      │
┌──▼──┐┌──▼──┐┌──▼───┐┌─▼──┐┌──▼──┐┌──▼──────┐
│plan ││cod  ││cod   ││cc- ││eval ││gardener │
│     ││ex-1 ││ex-   ││fr- ││     ││         │
│规划 ││后端 ││front ││ont ││评估 ││知识维护 │
│     ││     ││内部  ││外部││     ││         │
└─────┘└─────┘└──────┘└────┘└─────┘└─────────┘
                              cc-writer（Phase 4）
```

### 4.2 Agent 角色

| Agent | 默认模型 | 职责 |
|-------|---------|------|
| `plan` | claude-opus | 规划、制定/审核 Contract、拆解任务 |
| `codex-1` | gpt-5.4 | 后端、算法、脚本、部署 |
| `codex-frontend` | gpt-5.4 | 内部工具、admin 类前端（对样式无高要求） |
| `cc-frontend` | claude-sonnet | 外部用户可见的前端 UI（质量优先） |
| `cc-writer` | claude-sonnet | 写作、创作、文档类任务（Phase 4） |
| `eval` | gpt-5.4 | 独立评估，不参与执行；主观/创作任务可配置为 claude-sonnet |
| `gardener` | claude-sonnet | 定期 harvest lessons，维护知识库（Phase 3） |

**模型路由默认建议（可通过配置覆盖）：**

| 任务类型 | 默认 Agent | 理由 |
|----------|-----------|------|
| 后端 / 算法 / 脚本 | `codex-1` | token 经济 |
| 内部 admin / 后台 UI | `codex-frontend` | 样式要求低，节省 token |
| 外部用户可见 UI | `cc-frontend` | 质量优先 |
| 写作 / 创作（Phase 4） | `cc-writer` | 语言能力 |
| eval（客观验证） | `eval`（gpt-5.4） | 速度 + 经济 |
| eval（主观/创作） | `eval`（claude-sonnet） | 批判性思维 |
| 规划 / Contract 制定 | `plan`（claude-opus） | 全局理解能力 |

> ⚠️ 以上为**建议默认值**。所有 agent 的模型和 CLI 工具均可在项目配置中独立覆盖，系统不强制绑定任何模型。

Skill 级默认值存储在 `skill/references/defaults.json`，项目级配置（`nexum/config.json`）覆盖其中任意字段。可通过 `nexum config --show` 查看当前生效的完整合并配置。

### 4.3 执行容器

使用 **tmux** 作为 agent 进程容器：
- 轻量，单机无额外依赖
- 随时 attach 查看 agent 实时状态
- 支持多 session 并行
- 通过 dispatch 脚本统一管理生命周期

### 4.4 AGENTS.md 与 CLAUDE.md 同步

项目根目录同时维护两个文件以兼容不同 CLI：

- `AGENTS.md`：Codex CLI 自动加载
- `CLAUDE.md`：Claude Code 自动加载

**`AGENTS.md` 是唯一 source of truth**，`CLAUDE.md` 由 dispatch.sh 在 force-commit 阶段显式同步（`cp AGENTS.md CLAUDE.md`）。git hook 中**不做任何 commit 操作**，避免递归触发。

> 规则：永远不要直接编辑 `CLAUDE.md`。

## 五、Task Contract

### 5.1 Coding 类 Contract 格式

```yaml
# docs/nexum/contracts/TASK-001.yaml
id: TASK-001                    # 格式：[A-Z]+-[0-9]+
name: "实现用户登录页面"
type: coding
created_at: "2026-03-27T09:00:00Z"

scope:
  files:
    - src/pages/login.tsx
    - src/api/auth.ts
  boundaries:
    - "不修改 src/components/ui/ 下的基础组件"
    - "不修改数据库 schema"
  conflicts_with: []            # 与哪些 task id 存在语义冲突，需串行

deliverables:
  - "登录表单，支持邮箱 + 密码"
  - "调用 POST /api/auth/login"
  - "错误状态展示"
  - "登录成功跳转 /dashboard"

eval_strategy:
  type: e2e
  tools: [playwright]
  local_url: "http://localhost:3000"   # e2e 必填
  criteria:
    - id: C1
      desc: "表单提交后正确调用 /api/auth/login"
      method: "playwright: intercept network request"
      threshold: pass
    - id: C2
      desc: "错误密码时展示错误提示"
      method: "playwright: check error element visible"
      threshold: pass
    - id: C3
      desc: "登录成功后 URL 变为 /dashboard"
      method: "playwright: check URL after submit"
      threshold: pass

generator: codex-frontend
evaluator: eval
max_iterations: 3
depends_on: []
```

### 5.2 Creative/Task 类 Contract 格式（Phase 4）

```yaml
# docs/nexum/contracts/WRITE-001.yaml
id: WRITE-001
name: "撰写 PolyGo 产品介绍文档"
type: creative
created_at: "2026-03-27T09:00:00Z"

scope:
  files:
    - docs/product-intro.md
  boundaries:
    - "不修改其他文档"
  conflicts_with: []

deliverables:
  - "产品背景和核心功能介绍（500-800字）"
  - "面向技术用户，清晰描述系统架构"
  - "包含使用场景举例"

eval_strategy:
  type: review
  evaluator_model: claude-sonnet   # 创作类覆盖为 claude-sonnet
  criteria:
    - id: C1
      desc: "内容完整性：所有 deliverables 均已覆盖"
      method: "review: checklist"
      threshold: pass
    - id: C2
      desc: "表达清晰度"
      method: "review: score"
      threshold: score
      min_score: 7
    - id: C3
      desc: "原创性：无明显 AI 模板化痕迹"
      method: "review: score"
      threshold: score
      min_score: 6

generator: cc-writer
evaluator: eval
max_iterations: 3
depends_on: []
```

### 5.3 Eval Strategy 选择矩阵

| 任务特征 | 推荐策略 | 执行方式 | 默认 Evaluator 模型 |
|----------|----------|----------|-------------------|
| 纯算法/逻辑，有明确输入输出 | `unit` | 运行项目 test suite，解析结果 | gpt-5.4 |
| 涉及数据库/API 集成 | `integration` | 启动服务，调用 API，验证响应 | gpt-5.4 |
| 有前端 UI，用户可交互 | `e2e`（需 `local_url`） | Playwright 驱动浏览器执行 criteria | gpt-5.4 |
| 架构/代码质量/安全性 | `review` | evaluator 读代码，逐条 criteria 给 pass/fail | gpt-5.4 |
| 多维度复杂功能 | `composite` | 多个子策略顺序执行，全部 pass 才 pass | gpt-5.4 |
| 写作/创作类 | `review`（评分制） | evaluator 读内容，criteria 用 score threshold | claude-sonnet |
| 日常任务/研究报告 | `review`（完成度核查） | evaluator 核对 deliverables 是否齐全 | gpt-5.4 |

### 5.4 并行安全规则

**file scope 是并行的必要条件，但不是充分条件。**

允许并行的条件（同时满足）：
1. `scope.files` 集合与所有 running 任务无交集
2. `conflicts_with` 中未列出任何 running task id
3. 无语义依赖（不修改被其他 running 任务依赖的接口/类型）

当并行条件不满足时，通过 `depends_on` 串行化（而非运行时检测）。**Evaluator 是并行冲突的最终兜底**——eval 阶段会发现 scope 冲突导致的实际问题。

### 5.5 Contract 制定流程

```
orchestrator 起草 contract draft（含 conflicts_with 声明）
    ↓
（如需代码探索）plan agent 审核并补充技术细节
    ↓
orchestrator 最终确认，写入 contracts/TASK-ID.yaml
    ↓
dispatch generator（同时记录 base_commit）
    ↓
generator 完成后，evaluator 拿 contract 逐条验证
    ↓
全部 pass → mark done
有 fail → feedback 回 generator，进入下一轮
超过 max_iterations → escalate to human
```

### 5.6 Evaluator 调校要求

evaluator prompt 必须包含：
1. **怀疑主义指令**："你的工作是找问题，不是夸奖输出"
2. **逐条验证**：对每条 criterion 必须给出 pass/fail + 具体证据，不允许整体判断
3. **可操作的 fail 反馈**：包含文件路径、函数名、期望行为 vs 实际行为
4. **禁止模糊通过**："整体看起来没问题"不构成通过标准

## 六、知识管理

### 6.1 Lesson 格式

agent 在完成任务过程中，遇到非预见问题或值得记录的模式时，写入：

```markdown
<!-- docs/lessons/YYYY-MM-DD-TASK-ID.md -->
## Lesson: [简短标题]
- trigger: [什么情况触发了这个问题]
- wrong: [错误做法或错误假设]
- right: [正确做法]
- affected: [具体文件/函数/API]        ← 有此字段 = 技术层 lesson
- agent_note: [orchestrator 下次注意]  ← 有此字段 = 流程层 lesson
- tags: [pg, auth, typescript, ...]
```

`affected` 和 `agent_note` 互斥：
- `affected` 有代码路径 → **技术层**，最终进入 AGENTS.md
- `agent_note` 描述 orchestrator 行为 → **流程层**，最终进入 skill 内置知识

> **Phase 1**：agent 写入 lesson 文件，但 harvest 流水线在 Phase 3 实现。Phase 1-2 期间 lesson 文件只做积累，由人工定期整理。

### 6.2 Harvest 流水线（Phase 3）

每次批次完成后自动触发：

```
on-complete（全部任务 done）→ 触发 harvest
    ├── 读取新增 lesson 文件（自上次 harvest 以来）
    ├── 按字段分流：affected → 技术层；agent_note → 流程层
    ├── 技术层去重（同一 affected path + 相似 tags → 认为重复）
    ├── 互斥检查（同一 trigger 下 right 字段方向相反 → 认为冲突）
    └── 根据结果分流：
        ├── 纯新增 + 无冲突 → 写入 AGENTS.md nexum:lessons 区间 + Telegram 通知
        └── 有冲突 or 有歧义 → 生成 diff + Telegram 通知，等人工确认

AGENTS.md 维护规则：
  - 只修改 <!-- nexum:lessons:start --> 和 <!-- nexum:lessons:end --> 之间的内容
  - Gardener 负责维持 AGENTS.md 总行数 ~100 行
  - 超出时：低频条目降级到 docs/lessons/，AGENTS.md 只保留索引链接
  - harvest 检测到 nexum:lessons 区间有人工修改时，停止自动合并，生成三方 diff 通知人工处理
```

### 6.3 知识层级

| 层级 | 存储位置 | 内容 | 更新方式 |
|------|----------|------|----------|
| 技术层 | `project/docs/lessons/` → `AGENTS.md` | 代码规范、反模式、踩坑 | Phase 3 harvest 自动触发 |
| 流程层 | `skill/references/lessons/` | orchestrator 行为优化 | harvest 更新 skill |

### 6.4 Gardener Agent（Phase 3）

每批次完成后**自动触发**（也可手动触发）：
- 扫描所有 lesson 文件，识别过时/失效条目（referenced 文件已不存在）
- 合并重复/相似内容
- 维护 AGENTS.md 总行数 ~100 行（低频条目降级）
- 输出变更摘要，人工审核后提交

## 七、项目初始化

### 7.1 命令

```bash
nexum init --project <name> --tech "<stack>" --type <coding|mixed>
```

`--type` 是项目维度的大类（`coding` = 纯开发项目；`mixed` = 开发 + 写作/分析混合），与 Contract 里的 task `type` 枚举值是不同维度的分类。

### 7.2 执行内容

1. 创建标准目录结构
2. 从 skill 模板生成 `AGENTS.md`（≈100 行目录 + 通用工程规范）
3. 自动生成 `CLAUDE.md`（与 AGENTS.md 内容一致，后续由 dispatch.sh 维护同步）
4. 从 skill 反模式库（`skill/references/defaults.json` + `skill/references/lessons/`）提取与 tech stack 相关的条目，注入 AGENTS.md
5. 初始化 `docs/` 目录（design / lessons / nexum/contracts）
6. 生成 `docs/lessons/TEMPLATE.md`（lesson 写入模板）
7. 初始化 `nexum/active-tasks.json`（空批次）
8. 生成 `nexum/config.json`（从 skill 级默认值填充）
9. 安装 git hooks

### 7.3 标准项目结构

```
project/
├── AGENTS.md                    ← source of truth，≈100 行目录
├── CLAUDE.md                    ← 由 dispatch.sh 自动同步，勿手动编辑
├── ARCHITECTURE.md              ← 架构概览（首次由 plan agent 生成）
├── docs/
│   ├── design/                  ← 技术设计文档
│   ├── lessons/                 ← 踩坑记录（agent 自动写入）
│   │   └── TEMPLATE.md          ← lesson 写入模板
│   └── nexum/
│       └── contracts/           ← Task Contract 存档（YAML，静态不变）
└── nexum/
    ├── active-tasks.json        ← 唯一运行态主存储
    ├── config.json              ← 项目级配置（覆盖 skill 级默认值）
    ├── events.jsonl             ← 持久化事件日志（append-only）
    ├── runtime/
    │   ├── eval/                ← evaluator 结果（YAML，持久化）
    │   └── screenshots/         ← e2e 截图（.gitignore，不纳入版本控制）
    └── history/                 ← 已归档批次
```

### 7.4 AGENTS.md 内置学习协议

生成的 AGENTS.md 必须包含：

```markdown
## 错误学习协议

遇到非预见错误或值得记录的模式时：

1. 修复或完成任务
2. 写入 docs/lessons/YYYY-MM-DD-<TASK-ID>.md（使用 docs/lessons/TEMPLATE.md）
3. 不要直接修改 AGENTS.md 的 nexum:lessons 区间 — 由 gardener 统一维护
4. 不要直接编辑 CLAUDE.md — 由 dispatch.sh 自动同步

这是强制要求，不是可选项。
```

## 八、任务生命周期

### 8.1 状态机

```
                    ┌─────────────────────────┐
                    ▼                         │（depends_on 重新 blocked）
blocked ──→ pending ──→ running ──→ evaluating ──→ done
                           │             │
                           ▼             ▼
                         failed ←── eval_fail（未超次数）
                           │
                           ▼（超过 max_iterations）
                        escalated ──→ 通知人类，等待决策
                           │
                           ▼（人类处理后）
                          done / cancelled

cancelled：任何状态均可转入（人类主动中止）
  → 对应 tmux session 被终止
  → 下游 blocked 任务转为 escalated（等人类决定是否继续）
```

`blocked`：depends_on 中存在未 done 的 task 时的等待态。所有依赖 done → 自动转 pending。若依赖 task 最终为 `escalated` / `cancelled` → 当前 blocked task 同样转为 `escalated`，通知人类决策。

### 8.2 完整执行循环

```
1. orchestrator 制定 Task Contract（静态，写入 contracts/TASK-ID.yaml）
2. 注册任务到 active-tasks.json（status: pending，记录 contract_path）
3. dispatch generator：
   a. 记录 base_commit 到 active-tasks.json
   b. status → running，updated_at
4. generator 执行，git commit（只 add scope.files）
5. post-commit hook → 写 nexum/events.jsonl + push
6. on-complete（generator 完成）：
   a. 校验 commit scope（diff base_commit..HEAD vs contract.scope.files）
      - 违规 → status: failed，reason: scope_violation，冻结现场，Telegram 通知人类
      - 正常 → 继续
   b. status → evaluating，updated_at
   c. dispatch evaluator（根据 eval_strategy）
   d. 写 nexum/events.jsonl
   e. openclaw system event → 唤醒 orchestrator
7. evaluator 完成：
   a. 结果写入 nexum/runtime/eval/TASK-ID-iter-N.yaml
   b. on-complete 读取结果：
      - pass → status: done，解锁下游，通知，触发 harvest（批次全 done 时）
      - fail + iteration < max_iterations → iteration++，feedback 回 generator，status: running
      - fail + iteration >= max_iterations → status: escalated，Telegram 紧急通知
8. 所有任务 done → 汇报 → 自动触发 harvest + gardener（Phase 3）
```

### 8.3 重启恢复

orchestrator 启动时执行 reconcile：
1. 读取 `active-tasks.json`，找出所有 `status: running` 或 `evaluating` 的任务
2. 检查对应 tmux session 是否存活
3. session 不存活 → status: failed，reason: session_dead，通知人类
4. session 存活 → 视为正常继续（heartbeat 机制会检测是否卡死）
