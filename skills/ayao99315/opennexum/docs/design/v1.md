## 一、设计概览

### 1.1 系统边界

```
┌─────────────────────────────────────────────────────────────────┐
│  OpenClaw（宿主）                                                │
│                                                                  │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │  OpenNexum Skill                                           │ │
│  │                                                            │ │
│  │  orchestrator (AI)  ←→  scripts/  ←→  nexum/              │ │
│  │         ↕                                                  │ │
│  │  tmux sessions ──→ [codex-1] [cc-frontend] [eval] ...     │ │
│  └────────────────────────────────────────────────────────────┘ │
│                                                                  │
│  project repo (git)                                             │
│  ├── AGENTS.md / CLAUDE.md                                      │
│  ├── docs/lessons/  docs/nexum/contracts/                       │
│  └── nexum/ (active-tasks.json, events.jsonl, runtime/ ...)    │
└─────────────────────────────────────────────────────────────────┘
```

OpenNexum 作为 OpenClaw skill 存在，由三部分组成：
- **scripts/**：可执行脚本（dispatch、on-complete、harvest、init 等）
- **references/**：prompt 模板、schema 定义、skill 级默认值
- **orchestrator**：AI（OpenClaw 主 session）按本文档的规则做决策

### 1.2 核心数据流

```
需求输入
  │
  ▼
[orchestrator] 制定 Contract → docs/nexum/contracts/TASK-ID.yaml（静态，创建后不变）
  │
  ▼
[dispatch.sh] 记录 base_commit + 注册运行态 + 启动 generator
  │
  ▼
[tmux: generator] 执行 → git add scope.files → git commit
  │
  ▼
[post-commit hook] 写 nexum/events.jsonl + push
  │
  ▼
[on-complete.sh] scope 校验 → dispatch evaluator → 写 events.jsonl → openclaw system event
  │
  ▼
[tmux: evaluator] 读 Contract + 执行 eval → 写 nexum/runtime/eval/TASK-ID-iter-N.yaml
  │
  ▼
[on-complete.sh] 读 eval 结果 → pass: mark done + 解锁下游
                              → fail: feedback loop / escalate
  │
  ▼（all done）
[harvest.sh] + [gardener] 知识沉淀（Phase 3）
```

## 三、核心数据结构

### 3.1 Task Contract（YAML，静态）

Contract 创建后**不再修改**。运行态字段（status、iteration、commit 等）存储在 active-tasks.json。

```yaml
# docs/nexum/contracts/TASK-001.yaml
id: TASK-001
name: "实现用户登录页面"
type: coding
created_at: "2026-03-27T09:00:00Z"

scope:
  files:
    - src/pages/login.tsx
    - src/api/auth.ts
  boundaries:
    - "不修改 src/components/ui/ 下的基础组件"
  conflicts_with: []            # 与哪些 task id 存在语义冲突，需串行

deliverables:
  - "登录表单，支持邮箱 + 密码"
  - "调用 POST /api/auth/login"
  - "错误状态展示"
  - "登录成功跳转 /dashboard"

eval_strategy:
  type: e2e
  tools: [playwright]
  local_url: "http://localhost:3000"
  criteria:
    - id: C1
      desc: "表单提交后正确调用 /api/auth/login"
      method: "playwright: intercept network request"
      threshold: pass
    - id: C2
      desc: "错误密码时展示错误提示"
      method: "playwright: check error element visible"
      threshold: pass

generator: codex-frontend
evaluator: eval
max_iterations: 3
depends_on: []
```

### 3.3 Eval Result（YAML，持久化）

evaluator 写入 `nexum/runtime/eval/TASK-ID-iter-N.yaml`，路径由 on-complete.sh 按规则构造，无需 Contract 记录。

```yaml
# nexum/runtime/eval/TASK-001-iter-1.yaml
task_id: TASK-001
iteration: 1
evaluator_model: gpt-5.4
evaluated_at: "2026-03-27T09:30:00Z"

verdict: fail                   # pass | fail | error | inconclusive

# 系统层面的异常（app 未启动、工具不可用等）
system_errors: []

# 分层结果（composite 时有多个 strategy）
strategy_results:
  - strategy_type: e2e
    verdict: fail
    criteria_results:
      - id: C1
        result: pass
        evidence: "Network request to /api/auth/login observed"
      - id: C2
        result: fail
        evidence: "Element #login-error not visible after wrong password"
        detail: "src/pages/login.tsx:87 — setError() called but CSS condition wrong"
        score: null             # review/score 类 criteria 使用，e2e 时为 null

# 给 generator 的 feedback（fail 时必填）
feedback: |
  C2 失败：错误提示元素存在但未显示。
  位置：src/pages/login.tsx:87，setError() 被调用但 CSS 条件判断有误。
  期望：API 返回 401 时 #login-error 应变为 visible。

# 截图路径（e2e 时）
artifacts:
  - type: screenshot
    path: "nexum/runtime/screenshots/TASK-001-C2-fail.png"
```

## 四、核心脚本设计

### 4.1 dispatch.sh

**职责**：派发 generator 或 evaluator 的统一入口，管理任务状态转换。

```bash
dispatch.sh <agent> <task_id> --role <generator|evaluator> \
  [--prompt-file <path>] <cli_command...>
```

执行流程：

```
1. 验证 task_id 格式（whitelist regex：^[A-Z]+-[0-9]+$）
2. 读取 Contract YAML，根据 --role 校验：
   - generator：contract.generator == agent
   - evaluator：contract.evaluator == agent
3. 检查 tmux session（nexum-<agent>）存在；不存在则创建
4. 检查 agent 是否 busy（active-tasks.json 中已有 status:running 且 tmux_session==nexum-<agent>）
   - busy → 拒绝，报告冲突
5. 原子更新 active-tasks.json（flock + tmpfile + rename）：
   - generator：status → running，base_commit，tmux_session，started_at，updated_at
   - evaluator：eval_tmux_session，updated_at
6. 写 nexum/events.jsonl（task_started 或 eval_started）
7. 构建 prompt（详见 4.1.1）
8. 在 tmux session 中执行 CLI（heredoc，不做 shell 插值）
9. generator 完成后执行 force-commit（仅 add scope.files）
10. 调用 on-complete.sh（cleanup trap 兜底）
```

**4.1.1 Prompt 构建**

```
读取 prompt 模板（references/prompt-generator-coding.md）
  + 注入 contract.scope（files + boundaries）
  + 注入 contract.deliverables
  + 注入 contract.eval_strategy.criteria（让 generator 了解验证标准）
  + 注入项目相关 lessons（从 docs/lessons/ 按 tags 筛选）
  + 预写 git commit 命令（指定 scope.files，不用 git add -A）
  + 预写 CLAUDE.md 同步步骤（cp AGENTS.md CLAUDE.md）
写入 /tmp/nexum-prompt-<task_id>.txt（可丢弃）
```

**4.1.2 Force-commit 兜底**

```bash
# 仅 add Contract 中 scope.files 列出的文件
if [ -n "$(git status --porcelain)" ]; then
  git add -- ${SCOPE_FILES[@]}
  git commit -m "chore(nexum): force-commit for $TASK_ID [skip ci]"
fi

# 同步 CLAUDE.md
if git diff HEAD~1 --name-only 2>/dev/null | grep -q "^AGENTS.md$"; then
  cp AGENTS.md CLAUDE.md
  git add CLAUDE.md
  git commit -m "chore(nexum): sync CLAUDE.md from AGENTS.md" --no-verify
fi
```

**注意**：CLAUDE.md 同步在 dispatch.sh 显式处理，**不在 post-commit hook 中执行**，避免递归 commit。

### 4.2 on-complete.sh

**职责**：generator 或 evaluator 完成后的统一回调。

```bash
on-complete.sh <task_id> <exit_code> --role <generator|evaluator>
```

**场景 A：generator 完成**

```
1. exit_code != 0 → status: failed，last_error，写 events.jsonl，通知人类，退出
2. Scope 校验：
   git diff <base_commit>..HEAD --name-only 与 contract.scope.files 对比
   - 有 scope 外文件 → status: failed，reason: scope_violation
     冻结现场（不 revert），写 events.jsonl
     Telegram 通知：⚠️ TASK-001 scope 违规，冻结，附违规文件列表
     提示：执行 nexum revert TASK-001 可回滚（人类决定是否执行）
     退出
3. 更新 active-tasks.json：status → evaluating，commit_hash，updated_at
4. 写 nexum/events.jsonl（eval_started）
5. dispatch evaluator（dispatch-evaluator.sh）
6. openclaw system event → 唤醒 orchestrator
7. Telegram 通知（compact：generator done，正在 eval）
```

**场景 B：evaluator 完成**

```
1. 按约定路径读取 eval result：nexum/runtime/eval/TASK-ID-iter-N.yaml
   - 文件不存在 → status: failed，reason: eval_result_missing，通知人类
2. 解析 verdict：
   pass：
     a. status: done，completed_at，updated_at
     b. 写 nexum/events.jsonl（task_done）
     c. 扫描 active-tasks.json，所有 blocked 任务中 depends_on 全 done 的 → pending
     d. openclaw system event
     e. Telegram 完成通知（含 iteration 数、耗时、token）
     f. 检查是否批次全 done → 触发 harvest（Phase 3）

   fail + iteration < max_iterations：
     a. iteration++，updated_at
     b. 构建 feedback prompt（含 eval_result.feedback 全文）
     c. dispatch generator（重试），status: running
     d. 写 nexum/events.jsonl

   fail + iteration >= max_iterations：
     a. status: escalated，updated_at
     b. 写 nexum/events.jsonl
     c. openclaw system event
     d. Telegram 🚨 紧急通知（含完整 eval feedback + 操作建议）

   error / inconclusive：
     a. 视为 fail，同 fail 分支处理
     b. 通知中附加 system_errors 内容
```

### 4.3 dispatch-evaluator.sh

**职责**：根据 Contract eval_strategy 选择 evaluator prompt，通过 dispatch.sh 派发。

```bash
dispatch-evaluator.sh <task_id>
```

执行流程：
```
1. 读取 Contract，获取 eval_strategy.type
2. 构造 eval_result_path：nexum/runtime/eval/TASK-ID-iter-N.yaml（N 从 active-tasks 读）
3. 选择 prompt 模板：
   - unit/integration/review → references/prompt-evaluator.md
   - e2e → references/prompt-evaluator-e2e.md（含 Playwright 操作指引）
   - composite → 按 sub-strategies 依次构建，合并结果
4. 注入：
   - contract.deliverables
   - contract.eval_strategy.criteria（逐条，含 method 和 threshold）
   - eval_result_path（告知 evaluator 结果写到哪里）
   - e2e 模式：额外注入 local_url
5. 调用 dispatch.sh <evaluator-agent> <task_id> --role evaluator <cli_command>
```

### 4.4 update-task-status.sh

所有对 `active-tasks.json` 的写操作通过此脚本，保证原子性：

```bash
update-task-status.sh <task_id> <status> [--field key=value ...]
```

内部实现：
```bash
LOCK_FILE="$TASK_FILE.lock"
(
  flock -x 200
  python3 -c "
import json, sys, os
with open('$TASK_FILE') as f:
    data = json.load(f)
# 更新对应 task
for t in data['tasks']:
    if t['id'] == '$TASK_ID':
        t['status'] = '$STATUS'
        t['updated_at'] = '$(date -u +%Y-%m-%dT%H:%M:%SZ)'
        # 额外字段...
        break
# 原子写入
tmp = '$TASK_FILE.tmp'
with open(tmp, 'w') as f:
    json.dump(data, f, indent=2, ensure_ascii=False)
os.replace(tmp, '$TASK_FILE')
"
) 200>"$LOCK_FILE"
```

task 不存在时返回 exit 2（调用方据此判断是 task-not-found 还是其他错误）。

### 4.5 post-commit hook

hook 中**不做任何 commit 操作**，只做两件事：

```bash
#!/bin/bash
# 1. 写事件信号
TASK_ID="${NEXUM_TASK_ID:-unknown}"
echo "{\"event\":\"commit\",\"task_id\":\"$TASK_ID\",\"hash\":\"$(git rev-parse HEAD)\",\"ts\":\"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"}" \
  >> nexum/events.jsonl

# 2. 推送（失败只记录，不阻塞）
git push origin main 2>/dev/null || \
  echo "{\"event\":\"push_failed\",\"task_id\":\"$TASK_ID\",\"ts\":\"$(date -u +%Y-%m-%dT%H:%M:%SZ)\"}" \
  >> nexum/events.jsonl
```

CLAUDE.md 同步和 force-commit 均在 **dispatch.sh** 中处理，hook 不介入。

### 4.6 nexum-init.sh

```bash
nexum-init.sh --project <name> --tech "<stack>" --type <coding|mixed>
```

执行内容：
```

2. 生成 AGENTS.md：
   a. 读取 references/agents-md-template.md
   b. 从 skill/references/lessons/ 筛选与 tech stack 相关的 lessons
   c. 写入项目名、tech stack、日期、错误学习协议
3. cp AGENTS.md CLAUDE.md
4. 初始化 nexum/config.json（从 references/defaults.json 填充）

9. 生成 docs/lessons/TEMPLATE.md
```

### 4.7 health-check.sh

```bash
health-check.sh [--project <path>]
```

执行：
```
1. 读 active-tasks.json，找所有 status: running 或 evaluating 的任务
2. 对每个任务，检查 tmux_session / eval_tmux_session 是否存活
3. 存活：检查 updated_at，若超过 15 分钟未更新 → 认为卡死
   - 通知：⚠️ TASK-001 agent 疑似卡死（nexum-codex-frontend 已 N 分钟无响应）
4. 不存活：
   - status: failed，reason: session_dead，通知人类
5. 检查后输出健康摘要
```

### 4.8 nexum-revert.sh

人类主动触发，回滚某个 task 的所有提交：

```bash
nexum-revert.sh <task_id>
```

执行：
```
1. 读 active-tasks.json，获取 task.base_commit
2. 确认 task.status == failed（只允许对 failed 任务执行回滚）
3. git log --oneline <base_commit>..HEAD（展示将被回滚的 commits）
4. 询问确认（交互模式）
5. git revert <base_commit>..HEAD --no-commit
6. git commit -m "revert: roll back TASK-001 commits"
7. 更新 active-tasks.json：status: cancelled
8. 通知完成
```

## 五、Evaluator 设计

### 5.1 通用 Evaluator Prompt 结构

```
## Role
你是一个独立代码/内容评审员。你的工作是找问题，不是找夸奖的理由。
每条 criterion 必须给出明确的 pass 或 fail，以及具体证据。
"整体看起来没问题"不是有效结论。

## Task Context
[注入 contract.deliverables]
[注入 contract.scope.files]

## Criteria to Verify
[逐条注入 criteria，含 id、desc、method、threshold]

## Evaluation Rules
1. 逐条验证，不允许跳过任何一条
2. pass 需要：具体证据（文件名/行号/输出内容）
3. fail 需要：文件路径 + 行号（如能确定）+ 期望 vs 实际
4. score 类 criterion：给出 1-10 分 + 理由，低于 min_score 即 fail
5. 全部 criteria pass → verdict: pass；任意 fail → verdict: fail
6. 结果写入：[eval_result_path]（YAML 格式，按 schema）

## Output Schema
[注入 eval-result.yaml schema]
```

### 5.2 E2E Evaluator 扩展（Phase 2）

### 6.2 Harvest 冲突检测（Phase 3）

```python
# 伪代码
def check_lesson_conflict(new_lesson, existing_lessons):
    for existing in existing_lessons:
        # 1. 重复检测：same affected + 高度相似 right 字段
        if (new_lesson.affected == existing.affected and
            similarity(new_lesson.right, existing.right) > 0.8):
            return ConflictType.DUPLICATE, existing

        # 2. 互斥检测：same trigger + same affected + right 方向相反
        if (new_lesson.trigger == existing.trigger and
            new_lesson.affected == existing.affected and
            contradicts(new_lesson.right, existing.right)):
            return ConflictType.CONTRADICTION, existing

    return None
```

冲突分流：
- `DUPLICATE` → 生成 diff，Telegram 通知人工选择保留哪条
- `CONTRADICTION` → 生成三方 diff，Telegram 通知人工仲裁
- 无冲突 → 自动写入 `nexum:lessons` 区间 + Telegram 告知"已自动更新"

harvest 检测到 `nexum:lessons` 区间有人工修改（通过 git blame 判断）时，停止自动合并，生成三方 diff 通知人工处理。

## 七、重启恢复设计

### 7.1 Reconcile 逻辑

orchestrator 启动（或手动触发 `nexum reconcile`）时执行：

```
1. 读取 active-tasks.json，找所有非终态任务（非 done/cancelled）
2. 对每个 running/evaluating 任务：
   a. 检查 tmux_session 是否存活（tmux has-session -t nexum-xxx）
   b. 存活 + updated_at < 15分钟前 → 视为正常，heartbeat 继续监控
   c. 存活 + updated_at > 15分钟前 → 疑似卡死，通知人类
   d. 不存活 → status: failed，reason: session_dead，通知人类
3. 读取 nexum/events.jsonl，找上次已记录的最后事件
   - 补发可能丢失的 openclaw system event
4. 对 blocked 任务：检查 depends_on 是否都 done，是则转 pending
5. 对 pending 任务：通知 orchestrator 可以 dispatch
```

### 7.2 数据恢复优先级

重启后的状态以 **active-tasks.json 为准**（因为它通过 flock+atomic-write 维护，是最可靠的）。events.jsonl 用于补充事件流，不作为状态的 source of truth。
