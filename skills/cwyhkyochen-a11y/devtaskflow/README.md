# DevTaskFlow

**不会写代码，也能把想法做成软件。**

> 用自然语言发起开发任务，AI 驱动的开发流水线 —— 需求 → 方案 → 代码 → 审查 → 修复 → 部署 → 发布，一条链路跑通。

---

## 它怎么工作

你只需要做一件事：**用大白话说清楚你想做什么。**

跟 OpenClaw 聊天框里直接说：

> 帮我做一个PPT制作工具，我提需求，你拆解页面并生成PPT

DevTaskFlow 会自动分析你的需求，输出一份完整的分析报告：

  <br/>
  <em>输入需求 → 自动生成功能拆解、任务清单、技术方案、风险评估</em>
</p>

确认方案后，剩下的事情自动推进：

| 步骤 | 做什么 | 你需要做什么 |
|------|--------|-------------|
| 生成代码 | 调用 AI 写出完整项目代码 | 预览确认 |
| 代码审查 | 自动检查代码质量 | 看结果 |
| 自动修复 | 发现问题立刻修复再复审 | 无 |
| 本地运行 | 启动项目让你实际看看 | 确认上线 |
| 部署上线 | 推到服务器 | 无 |
| 封版归档 | 保存版本快照 | 无 |
| GitHub 发布 | 创建 Release | 无 |

你不需要记命令，不需要手动切换步骤，系统根据当前状态自动推进。

## 为什么做这个

很多人有业务理解、有场景、有产品直觉，甚至比工程师更清楚问题出在哪里。但现实是：

- 不会写代码，就很难自己把工具做出来
- 找外包、排开发、反复沟通，时间和成本都很高
- 即使有 AI，对普通人来说也常常只是"会聊天"，离真正落地还差很远

DevTaskFlow 要解决的，不是"再做一个代码生成器"，而是**把软件开发变成一条普通人也能走通的流程**。

你不需要先学会 React、数据库、部署、代码审查。你只需要先把需求说清楚。

> 技术不该只服务会写代码的人。技术应该成为更多人的杠杆。

---

## 核心能力

**从自然语言到可运行软件的完整链路：**

- 用大白话描述需求，自动分析并拆任务
- 调用 LLM 生成完整项目代码
- 结构化代码审查，发现问题自动修复
- 支持本地运行、部署、封版、GitHub 发布
- 多项目看板，进度一目了然

**对非技术用户友好：**

不需要懂技术，像这样说就够了：

> 我想做一个客户管理工具，给销售团队用。需要录入客户资料、搜索、跟进状态、备注，界面尽量简洁，最好手机也能用。

**给 AI 加上护栏：**

- 状态机推进，每一步都有记录
- 写入前可预览
- 审查 / 修复闭环，避免越修越乱
- 路径安全限制，不会写到项目外面

## 适合谁

- **不会写代码但想把想法做出来的人** — 产品经理、运营、创业者、业务负责人
- **会写代码但想提升效率的人** — 快速验证想法、标准化开发流程
- **需要"流程可追踪"的团队** — 每一步都知道发生了什么，有版本记录可回溯

## 工作流

```text
提需求 → 分析 → 确认 → 生成代码 → 审查 → 修复 → 本地运行 → 部署 → 封版 → 发布
```

每一步都有状态，每一步都能继续 / 恢复，每一步都能被看见。

## 快速开始

### 1. 安装

DevTaskFlow 以 OpenClaw Skill 的形式运行。安装后，直接在 OpenClaw 对话中使用。

### 2. 发起新项目

在 OpenClaw 聊天中直接说：

> 新建一个项目，叫 crm-lite，我想做一个给销售团队使用的客户管理工具，需要录入客户资料、搜索、跟进状态、备注，界面简洁，手机也能用。

系统会自动创建项目目录、分析需求、生成任务拆分，并推进后续开发状态。

### 3. 查看项目状态

在 OpenClaw 中说：

> 看下项目进度

或者通过可视化看板查看所有项目的当前阶段。

## 常用操作

所有操作都通过自然语言交互完成，不需要记命令。以下是一些示例：

```
"帮我分析一下这个需求能不能做"     → 启动分析
"方案没问题，开始写代码"           → 确认并生成代码
"先别部署，我想看看效果"           → 本地运行预览
"可以了，上线吧"                   → 部署
"发布到 GitHub"                    → 封版并创建 Release
"改成蓝色主题"                     → 反馈修改
```

也支持 CLI 模式，适合开发者批量操作：

```bash
dtflow setup
dtflow start --new-project --name NAME --idea "你的需求"
dtflow board
dtflow board --serve  # 可视化看板 → http://localhost:8765
```

## 使用示例

**场景：** 你是运营负责人，想做一个内部小工具。

> 我想做一个内容发布管理台，支持账号管理、发布记录、草稿保存，先做 Web 版，团队内部使用。

**DevTaskFlow 会怎么推进：**

1. 把需求变成结构化开发方案
2. 拆成可执行任务，生成项目文件和版本目录
3. 调用模型生成代码，自动审查
4. 有问题进入修复流程
5. 通过后本地运行验证，部署上线
6. 封版归档，发布到 GitHub

你面对的不是抽象的"AI 对话"，而是一条可以继续推进、可以回看、可以交付的开发链路。

## 架构

模块化设计，核心流程与具体模型、部署方式解耦，未来换底层不影响工作流。

```text
  OpenClaw / 自然语言交互层
            │
  ┌─────────┴─────────┐
  │   pipeline core   │  开发主流程（analyze → write → review → fix → deploy → seal）
  │   orchestrator    │  调度层（本地 LLM / OpenClaw 子 agent）
  │   state + board   │  项目状态与看板
  │   adapters        │  对接外部能力（LLM、部署、归档、GitHub）
  └───────────────────┘
```

详细架构说明见 `docs/` 目录。

## 安全与约束

- 写入路径限制在项目目录内
- 配置和密钥走环境变量 / 本地配置
- 状态文件记录最近动作、错误、摘要
- 写入前可预览（dry-run）
- 审查与修复分阶段执行
- 项目与版本分层管理

目标：**让错误更可见、可诊断、可恢复。**
