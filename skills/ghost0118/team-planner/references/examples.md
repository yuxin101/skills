# Team Planner - 使用示例

## 完整对话示例

### 示例 1: 电商系统开发

**用户提问:**
> "我想做一个电商平台，包含商品展示、购物车、订单管理、支付功能，怎么组织 Agent？"

**技能输出:**
```
# 电商平台开发 - 团队架构方案

## 1. 任务概要
- **目标**: 构建完整电商平台
- **复杂度**: 复杂 (强烈建议多 Agent 协作)
- **推荐方案**: 7 人团队

## 2. 团队成员

### researcher
- **角色**: 技术调研专家
- **职责**: 
  - 调研电商技术栈选型
  - 分析竞品技术架构
  - 评估开源方案
- **依赖**: 无
- **交付物**: 技术调研报告

### architect
- **角色**: 系统架构师
- **职责**:
  - 设计系统架构图
  - 制定数据库方案
  - 定义 API 规范
- **依赖**: researcher 的调研报告
- **交付物**: 架构设计文档

### backend-dev
- **角色**: 后端开发
- **职责**:
  - 实现商品模块 API
  - 实现订单模块 API
  - 实现支付模块对接
  - 数据库设计与实现
- **依赖**: architect 的架构设计
- **交付物**: 后端代码仓库

### frontend-dev
- **角色**: 前端开发
- **职责**:
  - 实现商品展示页面
  - 实现购物车功能
  - 实现用户中心
  - 对接后端 API
- **依赖**: architect 的 API 规范
- **交付物**: 前端代码仓库

### tester
- **角色**: 测试工程师
- **职责**:
  - 编写测试用例
  - 集成测试
  - 性能测试
  - 编写测试报告
- **依赖**: backend-dev 和 frontend-dev 的交付物
- **交付物**: 测试报告

### devops
- **角色**: 运维工程师
- **职责**:
  - 搭建 CI/CD 流水线
  - 配置服务器环境
  - 部署上线
- **依赖**: 完整代码库
- **交付物**: 部署文档

### documenter
- **角色**: 文档工程师
- **职责**:
  - 编写用户手册
  - 编写 API 文档
  - 编写开发指南
- **依赖**: architect 的架构文档
- **交付物**: 完整文档

## 3. 协作流程

### Phase 1: 调研与架构 (Week 1)
**成员**: researcher, architect
- researcher 完成技术选型报告
- architect 基于报告完成架构设计
- 团队评审架构方案

### Phase 2: 核心开发 (Week 2-3)
**成员**: backend-dev, frontend-dev
- 前后端并行开发
- tester 同步编写测试用例
- 每日站会同步进度

### Phase 3: 集成与测试 (Week 4)
**成员**: backend-dev, frontend-dev, tester
- 前后端集成
- 全面测试
- Bug 修复

### Phase 4: 部署上线 (Week 5)
**成员**: devops, documenter
- 部署到生产环境
- 完成所有文档
- 用户验收测试

## 4. 通信策略
- **广播**: 项目启动、阶段完成、重大决策
- **直发**: 具体任务分配、代码审查反馈
- **关闭通知**: 各成员完成工作时
```

### 示例 2: 数据分析报告

**用户提问:**
> "帮我分析一下我们公司的用户数据，产出用户画像报告"

**技能输出:**
```
# 用户数据分析 - 团队架构方案

## 1. 任务概要
- **目标**: 用户数据分析与画像报告
- **复杂度**: 中等
- **推荐方案**: 3 人团队

## 2. 团队成员

### data-collector
- **角色**: 数据收集工程师
- **职责**:
  - 连接数据源
  - 清洗数据
  - 数据预处理
- **交付物**: 清洗后的数据集

### analyst
- **角色**: 数据分析师
- **职责**:
  - 统计分析
  - 用户分群
  - 行为分析
- **依赖**: data-collector 的数据集
- **交付物**: 分析报告

### reporter
- **角色**: 报告撰写师
- **职责**:
  - 可视化图表制作
  - 报告撰写
  - 关键洞察提炼
- **依赖**: analyst 的分析结果
- **交付物**: 最终用户画像报告

## 3. 协作流程

### Phase 1: 数据准备
**成员**: data-collector

### Phase 2: 分析
**成员**: analyst

### Phase 3: 报告
**成员**: reporter
```

## 常见问题

### Q: 什么时候不需要多 Agent？
- 简单问答
- 单个文件操作
- 快速翻译
- 简单计算

### Q: 团队多大合适？
- **2-3人**: 简单任务，好管理
- **4-6人**: 中等复杂，可并行
- **7人以上**: 复杂任务，需要专职协调

### Q: 如何避免 Agent 之间冲突？
1. 明确每个 Agent 的职责边界
2. 设置清晰的依赖关系
3. 建立统一的协调机制
