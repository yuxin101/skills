# Deep Dive Scan Protocol

> Load on demand: only read when a scan request is received.

## When to Scan
- Product-lead sends a Deep Dive request via inbox
- New product added to shared/products/_index.md
- After major code changes (L4 incremental)
- Periodic health checks (L3, requested by product-lead)

## Execution Protocol

1. **Read the request** from inbox: product name, code directory, scan level, focus areas
2. **Enter the project directory** (read-only unless explicitly told to modify code)
3. **Detect tech stack** automatically:
   - `composer.json` → Laravel/PHP
   - `package.json` → Node/React/Vue (check framework field, dependencies)
   - `requirements.txt` / `pyproject.toml` → Python/Django/FastAPI
   - `go.mod` → Go | `Cargo.toml` → Rust
   - Multiple? Note it's a monorepo or full-stack project
4. **Execute scan commands** per stack (see below)
5. **Write knowledge files** to `shared/products/{product}/`
6. **Log the scan** in `shared/products/{product}/changelog.md`
7. **Notify product-lead** via inbox when done

## Per-Stack Scan Strategies

### Laravel/PHP
- L0: tree, composer.json, .env.example
- L1: migrations → database.md, route:list → routes.md + api.md, Models → models.md
- L2: Services/Actions → services.md, Middleware/Policies → auth.md, Jobs/Listeners → jobs-events.md, Console/Kernel → scheduled tasks, config/services.php → integrations.md
- L3: grep TODO/FIXME/HACK, test coverage, Dockerfile, CI, security scan

### React/Vue/Frontend
- L0: tree src/, package.json
- L1: components → frontend.md, router config, state management stores
- L2: API client layer → data-flow.md, form validation, auth/token handling, i18n

### Python/Django/FastAPI
- L1: models.py → database.md, urls.py → routes.md, serializers → api.md
- L2: views.py → services.md, middleware/permissions → auth.md, celery → jobs-events.md

### General (any stack)
- git log --oneline -30, find *.md, grep TODO/FIXME/HACK, tree -L 3
- .env.example → config-env.md, Dockerfile/CI → devops.md, test dirs → test-coverage.md

## Content Quality Standards

Knowledge files must capture:
1. **Facts** — what exists (tables, routes, classes)
2. **Relationships** — how things connect
3. **Rationale** — why it's built this way
4. **Implicit rules** — undocumented business logic buried in code
5. **Gotchas** — surprises for a new developer
6. **Cross-module coupling** — where changing A silently breaks B
7. **Performance notes** — N+1 queries, missing indexes, bottlenecks

## Output Format

```
# {Title} - {Product Name}
> Auto-generated by the dual-dev Deep Dive flow.
> Last scan: YYYY-MM-DD | Level: L{n} | Stack: {stack}
> Code directory: {path}
```

## L4 Incremental Protocol

1. `git log --oneline --since={last scan date}`
2. `git diff --stat` → map changed files to knowledge files
3. Read only changed files, update only affected knowledge files
4. Log in changelog.md

## Large Projects (>500 files)
Spawn Claude Code for the scan: provide scan level, output directory, content standards. Read-only mode.
