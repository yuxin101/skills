# ProcessOn Diagrams (AI 绘图助手)

本技能旨在将用户意图或代码逻辑转化为专业、美观的图形。它能够识别多种图形类型，并针对每种类型应用特定的优化规则，以确保生成的图形质量最高。

## 核心能力

### 1. 图形类型智能识别
**逻辑**：根据用户的描述或选中的代码上下文，自动识别应生成的图形类型。
- **流程图 (Flowchart)**：适用于逻辑判断、步骤流转、业务流程。
- **时序图 (Sequence Diagram)**：适用于系统间交互、接口调用顺序。
- **架构图 (Architecture)**：适用于系统组成、模块划分、云架构。
- **网络拓扑图 (Network Topology)**：适用于服务器部署、网络连通性、设备互联。
- **ER 图 (Entity Relationship)**：适用于数据库建模、数据结构关系。
- **组织结构图 (Org Chart)**：适用于团队结构、层级关系。
- **时间轴/时间线 (Timeline)**：适用于历史演变、项目进度。
- **信息图/金字塔图 (Infographic/Pyramid)**：适用于等级划分、知识总结。
- **草图重绘 (Image-to-Diagram)**：如果用户上传了图片（如手绘草图、截图）并要求“重绘”或“转成流程图”，必须先识别图片中的节点、文字和连接关系，然后按照图形类型提取结构化数据，最后调用生成接口。

**主动询问**：如果用户意图不明确（例如只说“画个图”），必须主动询问用户：“您想生成哪种类型的图？（如流程图、架构图、时序图等）”。

### 2. 内容完整性校验与多轮交互 (需求澄清)
**逻辑**：在确定了图形类型后，必须评估用户提供的描述是否足以支撑该图形的生成。**绝不盲目猜测或生成逻辑严重缺失的图形。**
- **关系/结构不明确**：例如用户要生成“网络拓扑图”或“架构图”，但未说明核心组件间的连接关系、通信协议或层级划分。
- **流程/逻辑有断层**：例如“流程图”判断条件后缺少分支走向，或“时序图”缺少关键的参与者和返回值。
**交互要求（非常重要）**：
当发现信息不足或关系模糊时，必须暂停生成，并与用户进行多轮对话交互：
1. **指出模糊点**：明确告诉用户当前缺少哪些关键信息（例如：“您的描述中没有明确 Web 层和数据层之间的网络拓扑关系”）。
2. **提供启发式建议**：根据行业最佳实践或代码上下文，为用户提供合理的思路或默认方案供其选择（例如：“通常我们会加入一个负载均衡层（如 Nginx/ALB），并设置内网隔离，您看需要按照这种标准的架构来画吗？”或“针对这个报错处理分支，通常的逻辑是重试或直接返回 500，您倾向于哪种？”）。
3. **确认后执行**：在用户澄清、补充信息或直接确认使用您提供的建议方案后，再进入下一步。

### 3. Prompt 优化规则
在调用生成接口前，必须根据识别到的图形类型，在用户原始需求基础上增加专业描述，以获得更好的视觉效果：
- **通用规则**：要求“专业风格”、“布局清晰”、“颜色协调”、“避免线条交叉”。
- **流程图**：增加“明确开始和结束节点”、“使用标准菱形表示决策点”。
- **时序图**：增加“标注清晰的参与者 (Participants)”、“显示具体的调用消息”。
- **架构图**：增加“按层级划分（展示层、逻辑层、数据层）”、“标注组件间的通信协议”。
- **网络拓扑图**：增加“标注设备IP、子网划分、网关及防火墙策略”、“突出核心网络设备与终端的连通性”。
- **ER 图**：增加“标注主外键关系”、“显示字段类型”。

### 4. 生成与渲染流程
1.  **分析上下文**：提取关键实体、动作、判断条件。
2.  **构建优化 Prompt**：结合“图形类型 + 用户意图 + 优化规则”生成最终 Prompt。
3.  **优先调用 MCP 服务**：如果当前宿主已安装并暴露了名为 `processon-diagrams` 的 ProcessOn MCP 服务，优先通过该 MCP 调用图形生成能力，以便宿主直接消费结构化 `content` 并尽可能渲染图片。MCP 模式下的服务地址、密钥和鉴权方式由用户在宿主中自行配置，本技能不负责代填凭证，也不尝试触发输入框。
4.  **回退到本地脚本**：如果 MCP 服务未安装、不可用、不可达，或当前环境无法调用 MCP，则调用工作区内的 Python 脚本 `scripts/processon_api_client.py`（或在命令行执行 `python3 <skill_dir>/scripts/processon_api_client.py "<prompt>" --title "<short-title>"`）进行图形生成。API 回退模式需要用户自行设置 `PROCESSON_API_KEY`。
5.  **脚本模式输出要求**：脚本模式下应保留当前 `content` 输出，同时自动将 `image/png` 落盘到当前工作区内。文件名标题不要由脚本自行从 prompt 猜测，而应由大模型根据用户输入先总结成一个简短标题，再通过 `--title` 传给脚本；脚本默认使用 `标题-时间戳.png` 作为文件名，只有同一次生成多张图时才为后续图片追加序号。脚本只负责使用该标题生成文件名并返回保存路径、首图路径与“可内联展示”的元数据。若宿主或当前对话支持图片渲染，拿到本地路径后应优先直接展示图片，同时明确告诉用户保存路径；且不要额外输出 Mermaid 文本内容，以免与图片内容不一致影响观看。

## 执行策略
- **优先识别**：始终先确定图形类型。
- **先显式告知**：在实际调用 MCP 或脚本前，先用一句简短提示明确告知用户正在使用 `processon-diagrams` 技能处理当前请求，例如“这次我会用 `processon-diagrams` 技能来生成这张图”。这样用户能在对话中清楚感知技能已被触发。
- **优先 MCP，脚本兜底**：当存在名为 `processon-diagrams` 的 MCP 服务时优先走 MCP；无该 MCP 或不可用时走本地脚本兜底。不要反过来。
- **MCP 命名稳定**：本 skill 约定的 MCP 服务名固定为 `processon-diagrams`。如果后续接入多个平台，请尽量保持这个服务名不变；工具名也应保持稳定，避免 skill 无法可靠判断或调用。固定名字不是 MCP 协议强制要求，但对 skill 的可维护性和跨平台触发稳定性非常重要。
- **凭证职责清晰**：MCP 模式下，凭证完全由宿主或用户自行配置；脚本回退模式下，用户必须显式设置 `PROCESSON_API_KEY`。本技能会返回清晰的配置示例，并尽量通过结构化 `credential/actions` 元数据提示宿主弹出凭证输入或快捷操作；如果宿主不支持该类交互，则退化为手动配置指引。本技能不在聊天内收集密钥，也不尝试修改用户环境变量。
- **架构分析策略**：当用户要求分析项目架构时，**核心目标是梳理出项目各个业务/技术模块之间的逻辑关系和调用链路，而不是简单地画出文件夹目录树（Directory Tree）。** 请主动阅读项目的核心配置文件（如 `package.json` / `pom.xml`）、入口文件及核心路由，提取出**模块间的依赖关系、数据流向和层级架构**后，再生成逻辑层面的架构图。
- **静默执行**：除前置的技能触发提示外，在生成 Prompt 后直接调用工具，无需向用户展示中间的优化过程，直接展示结果。
- **错误处理兜底 (Fallback)**：如果 MCP 和 API 脚本都失败，必须先向用户展示明确的失败原因，并明确提示如何配置 `PROCESSON_API_KEY`，附上可直接复制的示例命令；不要再自动输出 Mermaid 作为兜底图。若脚本已成功生成并保存 PNG，则优先展示图片并附上本地路径。

## 配置示例

### MCP 模式
如果宿主已支持并安装了 `processon-diagrams` MCP 服务，请由用户在宿主配置中自行完成服务地址和凭证设置。示例：

```json
{
  "mcpServers": {
    "processon-diagrams": {
      "type": "http",
      "url": "https://smart.processon.com/mcp",
      "headers": {
        "Authorization": "Bearer <your-processon-api-key>"
      }
    }
  }
}
```

### API 回退模式
如果当前宿主没有安装 MCP，技能会自动回退到本地脚本。此时请用户自行设置环境变量：

```bash
export PROCESSON_API_KEY="<your-processon-api-key>"
```

其中 `PROCESSON_API_KEY` 为 API 回退模式必填。

## 示例优化 Prompt
> **用户意图**：帮我画一个登录流程。
> **优化后**：请生成一张专业的**流程图**，描述用户登录注册流程。包含：前端校验、后端鉴权、数据库查询、Token 发放。要求：布局美观，使用标准的流程图符号，逻辑闭环，配色方案采用商务蓝。
