# Follow-Up Checkliste — Nachbereitung nach Brainstorming

**Use:** 48h, 1 Woche, 4 Wochen nach Session. Stellt sicher dass Ideen zu Action werden.

---

## 1. 48-Stunden-Check (Sofort nach Session)

### 1.1 Dokumentation (Max 30 Min nach Session)

```
□ Dokumentation erstellt (output-template.md ausgefüllt)
□ Top 3 Ideen klar definiert
□ Action Items mit Ownern und Deadlines
□ Session-Report verschickt (Email/Slack)
□ Offene Fragen dokumentiert
□ Nächste Session geplant (falls nötig)
```

### 1.2 Stakeholder-Update (24-48h nach Session)

```
□ Ergebnis-Summary an Teilnehmer geschickt
□ Action Items in Project-Tool (Jira/Asana/etc.) übertragen
□ Stakeholder ohne Teilnahme informiert (falls relevant)
□ Feedback-Request verschickt (optional 3-Question Survey)
```

### 1.3 Direktes Feedback (48h nach Session)

```
□ Teilnehmer gefragt: "Wie wars?"
□ Verbesserungsvorschläge notiert
□ Schwierigkeiten im Prozess identifiziert
□ Facilitator Self-Review:
  - Was lief gut?
  - Was würde ich anders machen?
  - Welche Technik hat funktioniert/nicht funktioniert?
```

---

## 2. 1-Woche-Follow-Up

### 2.1 Action-Item-Tracking

```
□ Action Item 1: [Wer] hat begonnen? Status?
□ Action Item 2: [Wer] hat begonnen? Status?
□ Action Item 3: [Wer] hat begonnen? Status?

Falls nicht gestartet:
□ Blockaden identifiziert?
□ Owner an Deadline erinnert?
□ Scope angepasst (falls nötig)?
```

### 2.2 Eskalations-Tracking

```
□ Offene Fragen aus Session geklärt?
□ Offene Fragen noch offen? → Wer kümmert sich?
□ Neue Stakeholder benötigt für Entscheidung?
□ Sponsor/Entscheider einbezogen?
```

### 2.3 Team-Engagement

```
□ Team noch aligned mit Top-Ideen?
□ Neue Informationen/Constraints eingetreten?
□ Anpassung der Priorisierung nötig?
□ Sprint/Roadmap aktualisiert?
```

### 2.4 Kommunikations-Update

```
□ Wöchentliches Update an Stakeholder?
□ Entscheidungen kommuniziert?
□ Fortschritt sichtbar gemacht?
□ nächste Schritte klar?
```

---

## 3. 4-Wochen-Review

### 3.1 Impact-Messung

```
□ Metriken definiert? (aus Metriken-Template)
□ Baseline etabliert? (Vorher-Zustand)
□ Erste Messungen durchgeführt?
□ Trend positiv/negativ/neutral?
□ Korrekturmaßnahmen nötig?
```

### 3.2 ROI-Bewertung

```
□ Kosten für Session + Follow-Up: [Betrag]
□ Erwarteter ROI: [Betrag/X%]
□ Realisierter ROI bisher: [Betrag/X%]
□ Aufwand vs. Nutzen: [Verhältnis]
□ Lohnt sich die Investition?
```

### 3.3 Lessons Learned

```
□ Was hat funktioniert?
□ Was würden wir anders machen?
□ Welche Ideen wurden umgesetzt? Welche nicht?
□ Was hat der Prozess gekostet (Zeit, Geld)?
□ Nächste Session: Was ändern wir?
```

### 3.4 Retrospektive (optional)

```
□ Kurze Retro mit Kern-Team (30 Min)?
□ Was lief gut → behalten
□ Was lief schlecht → ändern
□ Konkrete Improvements für nächste Session
```

---

## 4. Decision-Review (falls Entscheidungen getroffen)

### 4.1 Entscheidungs-Tracking

```
Entscheidung: [Beschreibung]
Getroffen am: [Datum]
Erwartetes Ergebnis: [Konkret]
Verantwortlich: [Name]

Check nach 1 Woche:
□ Umsetzung gestartet?
□ Erwartetes Ergebnis auf Kurs?
□ Anpassung nötig?

Check nach 4 Wochen:
□ Ergebnis eingetreten?
□ Gemessen/Kalkuliert?
□ Learnings für nächste Entscheidung?
```

### 4.2 Entscheidungs-Eskalation

```
□ Entscheidung braucht Review? → Wer entscheidet?
□ Entscheidung veraltet? → Neue Session?
□ Annahmen nicht eingetreten? → Anpassung?
□ Entscheidung war falsch? → Reversion oder Anpassung?
```

### 5.1 Von Idee zu Backlog-Item

```
Idee: [Name]
Kategorie: [New Feature / Improvement / Tech Debt / Research]

Backlog Item erstellt:
□ Ticket/Story erstellt in [Tool]
□ Description ausgefüllt mit Kontext

□ Schätzung: [T-Shirt/Story Points/Stunden]
□ Priorität: [P0/P1/P2/P3]
□ Sprint zugeordnet: [Sprint/Quarter]
□ Abhängigkeiten identifiziert?
```

### 5.2 Projekt-Setup (falls Idea becomes Project)

```
□ Projekt/Epic erstellt?
□ Meilensteine definiert?
□ Ressourcen zugewiesen?
□ Budget freigegeben?
□ Stakeholder-Sponsor benannt?
□ Regelmäßiges Status-Update geplant?
```

## 7. Follow-Up-Cadence

```
SESSION-TAG (0):
  □ Dokumentation erstellen
  □ Email verschicken
  □ Action Items in Jira/Tool

48 STUNDEN:
  □ Stakeholder-Update
  □ Feedback einholen
  □ Facilitator Self-Review

1 WOCHE:
  □ Action Items Status-Check
  □ Eskalationen verfolgen
  □ Team-Engagement prüfen
  □ Slack/Email Update

2 WOCHEN:
  □ Fortschritt messen
  □ Anpassungen vornehmen
  □ Neue Inputs integrieren

4 WOCHEN:
  □ Impact-Messung
  □ ROI-Kalkulation
  □ Lessons Learned
  □ Retrospective (optional)
  □ Roadmap/Backlog Update

3 MONATE (optional):
  □ Langzeit-Impact check
  □ Entscheidungs-Review
  □ Archivierung/Publication
```

## 8. Eskalations-Pfade

### 8.1 Wenn Action Items nicht starten

```
Level 1 (Tag 1-2 nach Deadline):
→ Sanfte Erinnerung an Owner

Level 2 (Tag 3-5 nach Deadline):
→ Direktes Gespräch: "Brauchst du Support?"

Level 3 (Tag 7+ nach Deadline):
→ Eskalation an Sponsor/Stakeholder
→ Decision: Streichen, verschieben, oder andere Owner?
```

### 8.2 Wenn Scope Creep passiert

```
Signal: Neue Ideen für gleiche Session werden diskutiert

Response:
"Okay, das ist eine gute Idee für [Topic/Phase].
Lass uns das im Parking Lot sammeln - das ist für [aktuelle Session] 
fokus auf [Original-Ziel].

Für [neue Idee] machen wir eine separate Session."

→ Nicht mischen, Fokus behalten
```

### 8.3 Wenn Entscheidung sich als falsch herausstellt

```
Schritt 1: Fakten sammeln
- Was wurde entschieden?
- Was war die Annahme?
- Was ist eingetreten vs. erwartet?

Schritt 2: Korrektur-Maßnahme
- Reversion möglich? (Cost/Benefit)
- Anpassung nötig?
- Neue Session für alternatives Vorgehen?

Schritt 3: Lernen dokumentieren
- Warum war die Entscheidung falsch?
- Was hätten wir besser machen können?
- Wie verhindern wir in Zukunft?
```
