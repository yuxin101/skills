# Clarify Workflow

Use this workflow to identify the real question before trying to solve it.

## When to use

Use this workflow when:

- the user's question is vague
- key terms are undefined
- the user is mixing goal and method
- a tactical question is hiding a strategic one
- the problem is emotionally charged or poorly framed
- the issue cannot be examined because nothing has been defined cleanly

## Core question

What is the user actually trying to decide, understand, or achieve?

## Process

### Step 1: Restate the surface question

Write the question as the user currently frames it.

### Step 2: Identify vague or overloaded language

Look for terms such as:

- better
- professional
- scalable
- necessary
- good
- worth it
- risky
- simple
- successful
- high quality

Ask what each one means in this specific context.

### Step 3: Separate goal from method

Check whether the user is naming:

- the real outcome
- a preferred method
- a fear
- a constraint
- a proxy metric
- a social expectation

Do not assume the user's preferred method is the real objective.

### Step 4: Surface initial assumptions

Identify the first layer of hidden assumptions in the framing.

Examples:
- an app is necessary
- more complexity means more quality
- competitors define the right structure
- the bottleneck is already known

### Step 5: Rewrite the problem cleanly

State the sharpest version of the actual question.

## Output format

- Surface question
- Ambiguous terms
- Hidden assumptions
- Real goal
- Reframed question

## Quality bar

A good clarification should make the problem narrower, more precise, and easier to evaluate.
