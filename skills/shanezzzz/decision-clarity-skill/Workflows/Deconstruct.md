# Deconstruct Workflow

Use this workflow to reduce a problem into basic facts, constraints, mechanics, and causal structure.

## When to use

Use this workflow when:

- the issue feels large or fuzzy
- a problem is being treated as a black box
- cost, difficulty, or slowness is being accepted without decomposition
- a decision depends on understanding the real mechanism
- the current framing may be inherited from habit or industry convention

## Core question

What is this problem actually made of?

## Process

### Step 1: State the real goal

Write the actual outcome that matters.

### Step 2: Break the issue into constituent parts

Reduce the problem into components such as:

- users
- tasks
- costs
- steps
- incentives
- dependencies
- time requirements
- decision points
- risks
- constraints

### Step 3: Identify basic facts

List only what is currently supported by evidence, observation, logic, or hard boundaries.

### Step 4: Classify constraints

Separate:

- hard constraints
- soft constraints
- open assumptions

### Step 5: Map causal structure

Explain what seems to be causing what.

Examples:
- why a workflow is slow
- why growth is expensive
- why the product feels bloated
- why the user is confused

### Step 6: Highlight false complexity

Show where the default story is larger or more mystical than the actual structure.

## Output format

- Real goal
- Problem components
- Basic facts
- Hard constraints
- Soft constraints
- Causal structure
- False complexity or misleading framing

## Quality bar

A good deconstruction should make the issue feel more concrete, more measurable, and less inherited.
