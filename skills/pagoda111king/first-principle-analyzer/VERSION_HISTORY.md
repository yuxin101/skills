## v0.1.0 (2026-03-27) - 初始版本

### 灵感来源

#### 核心灵感 1：Elon Musk 的第一性原理思维

**来源**：Elon Musk 多次公开访谈（TED、Wired、YouTube）

**核心思想**：
> 「我们倾向于用类比思维生活——做别人正在做的事，或者稍微改进一下。但第一性原理要求你从物理学角度思考：什么是真正正确的？什么是可能的？然后从那里往上推理。」

**提取模式**：
- **情境**：面对复杂问题或「不可能」的目标
- **决策**：拒绝类比思维（「别人怎么做」），回归基本真理（「物理上可能的是什么」）
- **方法**：
  1. 识别并质疑所有假设
  2. 分解到不可再分的基本真理
  3. 从基本真理重新构建解决方案
- **经典案例**：
  - SpaceX 火箭成本分析：不是「火箭一直都很贵」，而是分解为原材料成本→发现原材料只占火箭成本的 2%→决定自研自产
  - Tesla 电池成本分析：不是「电池组一直都很贵」，而是分解为组成元素的市场价格→发现成本来自组织方式→重新设计电池架构

**对本技能的启示**：
- 必须强制用户质疑假设，不能跳过这一步
- 分解必须到达「基本真理」层级（不可再分）
- 重构阶段必须与类比方案对比，凸显创新价值

---

#### 核心灵感 2：Linux Kernel 开发流程文档

**来源**：torvalds/linux - Documentation/process/development-process.rst

**核心思想**：
> 「这个社区的工作方式可能对不熟悉 Linux 内核开发（或自由软件开发）的人来说难以理解。本文档试图记录这个过程，帮助开发者以最小的挫折感与开发社区合作。」

**提取模式**：
- **情境**：复杂系统需要多人协作
- **决策**：将过程文档化、结构化、分层级
- **方法**：
  1. 按角色分层（新开发者/研究者/维护者/硬件厂商）
  2. 按阶段分层（早期/编码/提交/跟进/高级主题）
  3. 每个阶段有明确的验收标准
- **关键洞察**：
  - 文档与代码同等重要
  - 过程透明化降低协作成本
  - 分层让不同角色快速找到所需信息

**对本技能的启示**：
- 第一性原理分析必须有清晰的流程文档
- 按问题类型分层（技术问题/商业问题/人生决策）
- 每个分析阶段有明确的输出物

#### 核心灵感 3：物理学的第一原理

**来源**：亚里士多德物理学、笛卡尔方法论

**核心思想**：
> 「第一原理是一个基本的、不证自明的命题或假设，不能被从任何其他命题或假设中推导出来。」—— 亚里士多德

**提取模式**：
- **情境**：需要建立可靠的知识体系
- **决策**：从不可质疑的基本真理开始
- **方法**：
  1. 系统性怀疑（笛卡尔）：质疑一切可质疑的
  2. 分解到不可再分
  3. 从基本真理演绎推理
- **经典案例**：
  - 牛顿力学：从三大定律演绎出整个经典力学
  - 欧几里得几何：从 5 条公设演绎出整个几何体系

**对本技能的启示**：
- 必须定义什么是「基本真理」（不可再分、不证自明）
- 必须有验证机制确认已到达基本真理层级
- 重构阶段必须是演绎推理，而非归纳

### 需求定义

#### 用户需求（原始）

「我要用第一性原理思考问题，而不是类比思维。」

#### 本质需求（第一性原理分析后）

**分解**：
1. 什么是「用第一性原理思考」？
   - 识别假设 → 质疑假设 → 分解问题 → 到达基本真理 → 重构方案

2. 为什么需要工具帮助？
   - 人类大脑默认使用类比思维（认知捷径）
   - 第一性原理需要刻意练习和结构化流程
   - 需要外部视角质疑隐含假设

3. 工具的核心价值是什么？
   - 强制性质疑流程（不能跳过）
   - 结构化分解框架（避免遗漏）
   - 基本真理性验证（避免伪分解）
   - 重构方案生成（从基本真理演绎）

**本质需求**：
> 一个结构化的思维框架，强制用户完成「质疑 - 分解 - 验证 - 重构」的完整流程，并提供外部视角挑战隐含假设。

### 设计决策

#### 决策 1：采用分阶段流程，而非单步分析

**情境**：设计技能的核心工作流

**选项**：
- A) 单步分析：用户输入问题，直接输出完整报告
- B) 分阶段流程：逐步引导用户完成质疑 - 分解 - 验证 - 重构

**选择**：B) 分阶段流程

**理由**：
- 第一性原理思考的核心价值在于**过程**，而非结果
- 用户需要参与质疑和分解过程，才能理解思维模式
- 分阶段允许用户在每个阶段反思和调整

**权衡**：
- 放弃：快速出结果
- 得到：用户真正学会第一性原理思维

---

#### 决策 2：强制性质疑流程，不能跳过

**情境**：用户可能想跳过假设质疑阶段

**选项**：
- A) 允许用户跳过，提高灵活性
- B) 强制完成质疑，才能进入下一阶段

**选择**：B) 强制完成

**理由**：
- 类比思维的根源在于未质疑的假设
- 跳过质疑 = 回到类比思维
- 这是第一性原理的核心，不能妥协

**权衡**：
- 放弃：部分追求速度的用户
- 得到：保证分析质量

---

#### 决策 3：使用 Markdown 输出报告

**情境**：选择报告格式

**选项**：
- A) JSON（机器可读）
- B) Markdown（人类可读）
- C) 两者都提供

**选择**：B) Markdown（v0.1.0），C) 是未来方向

**理由**：
- v0.1.0 优先服务人类用户
- Markdown 易于阅读和分享
- 未来可添加 JSON 输出供其他技能调用

## v1.0.0 (愿景)

### 终极目标

成为「思维操作系统」的核心组件——任何复杂决策都可以调用 first-principle-analyzer 进行深度分析，输出可执行、可验证、创新的方案。

1. **多领域支持**：技术/商业/人生/学术/政治
2. **专家模式库**：整合 50+ 顶尖专家的思维模式
3. **可视化分解**：思维图谱，展示假设 - 分解 - 重构的全链路
4. **记忆与进化**：记录用户的思维模式，提供个性化指导
5. **协作分析**：多人共同参与质疑和分解过程
6. **API 化**：可被其他技能调用，作为底层思维引擎

## 附录：使用示例

### 示例 1：技术问题

**输入**：「如何创建一个更好的技能系统？」

**输出**：
```markdown
# 第一性原理分析报告

## 问题分类
技术架构设计

## 识别的假设
1. 「技能系统」必须是集中式的
2. 技能必须由人类创建
3. 技能发现必须通过搜索

## 假设质疑
1. 为什么技能系统必须是集中式的？
   → 历史原因（npm/pip 模式），非技术必然
2. 为什么技能必须由人类创建？
   → 当前 AI 能力限制，非永恒真理
3. 为什么技能发现必须通过搜索？
   → 信息组织方式限制，非最优解

## 分解到基本真理
1. 技能的本质是什么？
   → 可复用的能力单元
2. 为什么需要技能？
   → 避免重复造轮子
3. 技能的核心属性？
   → 原子性、可组合性、可发现性

## 重构方案
方案 A：基于能力图谱的自动发现系统
方案 B：AI 自动生成技能的系统
方案 C：去中心化的技能市场

## 与类比方案对比
传统方案：参考 npm/pip/cargo 的设计
创新方案：从第一原理出发的能力图谱系统
```
