# Cron Templates — Nightly Extraction & Morning Brief Configuration

Complete documentation for the two cron job configs generated by `setup_ops.sh` and activated via `openclaw cron add`.

---

## Overview

Two scheduled jobs keep your agent learning:

1. **Nightly Extraction** — Runs ~11 PM (customizable), extracts facts from daily notes
2. **Morning Brief** — Runs ~8 AM (customizable), delivers curated summary

Both are JSON configs that you activate manually for safety: `openclaw cron add < config.json`

---

## Nightly Extraction Cron

### Example Config

```json
{
  "name": "nightly-extraction-agent-1",
  "schedule": "0 23 * * *",
  "timezone": "America/New_York",
  "agent_id": "agent-1",
  "description": "Consolidate daily conversations into atomic facts",
  "command": "echo 'Nightly extraction running' && extract_facts.sh",
  "model": "anthropic/claude-haiku-4-5",
  "memory_path": "memory/",
  "output_path": "life/"
}
```

### Field Reference

#### `name` (string, required)
Unique job identifier. Used in logs and for enable/disable.

**Convention:** `nightly-extraction-[agent-id]`

Example: `nightly-extraction-agent-1`

#### `schedule` (cron expression, required)
When to run. Format: `minute hour day-of-month month day-of-week`

**Common examples:**
- `0 23 * * *` — Every day at 11 PM
- `0 0 * * *` — Every day at midnight
- `0 22 * * 1-5` — Weekdays at 10 PM (skip weekends)

**Default:** `0 23 * * *` (11 PM daily)

#### `timezone` (string, required)
Timezone for cron schedule interpretation.

**Format:** IANA timezone database (e.g., `America/New_York`, `Europe/London`, `Asia/Tokyo`)

**Importance:** Cron schedules are timezone-aware. `0 23` means 11 PM local time, not UTC.

#### `agent_id` (string, required)
Which agent runs this job. Used for logging and multi-agent deployments.

**Convention:** Same as in IDENTITY.md

Example: `agent-1`, `keats`, `cortex-primary`

#### `description` (string, recommended)
Human-readable description. Shows up in logs and job listings.

Example: `"Consolidate daily conversations into atomic facts"`

#### `command` (string, optional)
Shell command to execute. If provided, runs before the OpenClaw internal extraction logic.

**Use cases:**
- Call a pre-extraction script: `./scripts/preprocess.sh && extract_facts.sh`
- Log to file: `echo "Extraction starting $(date)" >> logs/extract.log`
- If empty/omitted: Uses OpenClaw's built-in extraction logic

#### `model` (string, required)
Which model to use for extraction. Should be cheap and fast.

**Recommended:** `anthropic/claude-haiku-4-5` ($0.25/1M tokens)

**Why Haiku:** Structured extraction is a routine task. Haiku handles it perfectly. (See MemPO research: self-managed memory reduces token cost 67–73%.)

**Not Opus:** Opus ($15/1M tokens) is 60× more expensive for the same task.

#### `memory_path` (string, required)
Where to find daily notes for extraction.

**Default:** `memory/`

**Format:** Relative path from workspace root. Contains files like `memory/daily/2026-03-17.md`, `memory/daily/2026-03-16.md`, etc.

#### `output_path` (string, required)
Where to write extracted facts (items.json).

**Default:** `life/`

**Structure:** `life/projects/[project]/items.json`, `life/areas/[area]/items.json`, etc. (PARA structure)

## Morning Brief Cron

### Example Config

```json
{
  "name": "morning-brief-agent-1",
  "schedule": "0 8 * * *",
  "timezone": "America/New_York",
  "agent_id": "agent-1",
  "description": "Morning priorities and overnight summary",
  "model": "anthropic/claude-haiku-4-5",
  "memory_path": "memory/",
  "knowledge_graph_path": "life/",
  "delivery_channel": "telegram",
  "delivery_target": "123456789",
  "include_risks": true,
  "include_goals": true
}
```

### Field Reference

#### `name` (string, required)
Unique job identifier.

**Convention:** `morning-brief-[agent-id]`

Example: `morning-brief-agent-1`

#### `schedule` (cron expression, required)
When to run. Typically in the morning.

**Common examples:**
- `0 8 * * *` — Every day at 8 AM
- `0 7 * * 1-5` — Weekdays at 7 AM (skip weekends)
- `30 8 * * *` — Every day at 8:30 AM

**Default:** `0 8 * * *` (8 AM daily)

**Tip:** Schedule 15–30 minutes after you typically wake up. You want the brief ready to read with coffee.

#### `timezone` (string, required)
Timezone for schedule interpretation.

**Same as nightly extraction timezone.** If both jobs are in the same workspace, use same timezone.

#### `agent_id` (string, required)
Which agent runs this job.

**Convention:** Same as in IDENTITY.md, matching nightly extraction agent_id

#### `description` (string, recommended)
Human-readable description.

Example: `"Morning priorities and overnight summary"`

#### `model` (string, required)
Which model to use for synthesis.

**Recommended:** `anthropic/claude-haiku-4-5` ($0.25/1M tokens)

**Why:** Synthesis from facts is a structured task. Haiku is sufficient and cheap.

#### `memory_path` (string, required)
Where daily notes live (for context).

**Default:** `memory/`

#### `knowledge_graph_path` (string, required)
Where extracted facts live (items.json in PARA structure).

**Default:** `life/`

#### `delivery_channel` (string, optional)
How to deliver the brief. If omitted, prints to stdout.

**Valid values:**
- `telegram` — Send to Telegram chat
- `slack` — Send to Slack channel
- `email` — Send via email
- `stdout` — Print to console (default)
- `file` — Write to file (requires `delivery_target`)

#### `delivery_target` (string, optional)
Where to deliver (format depends on channel).

**Examples:**
- Telegram: `"123456789"` (chat ID)
- Slack: `"#general"` (channel name)
- Email: `"user@example.com"` (email address)
- File: `"logs/brief.txt"` (file path)

**How to find Telegram chat ID:**
```bash
# Send a message to your bot, then:
curl https://api.telegram.org/bot[BOT_TOKEN]/getUpdates
# Look for "chat": { "id": 123456789 }
```

#### `include_risks` (boolean, optional)
Whether to highlight risks in the brief.

**Default:** `true`

**Rationale:** Morning context should surface potential blockers. Keep risk-aware.

#### `include_goals` (boolean, optional)
Whether to highlight goals in the brief.

**Default:** `true`

**Rationale:** Morning context should remind you of targets. Start aligned.

## Multi-Agent Setup

If you have multiple agents in one workspace:

```json
{
  "name": "nightly-extraction-agent-1",
  "agent_id": "agent-1",
  "schedule": "0 23 * * *",
  ...
}
```

```json
{
  "name": "nightly-extraction-agent-2",
  "agent_id": "agent-2",
  "schedule": "0 23 * * *",
  ...
}
```

Each agent has its own extraction job. They run independently, maintain separate items.json files (or the same one with agent_id as provenance).

## Activation & Management

### Add a cron job

```bash
openclaw cron add < nightly-extraction-cron.json
```

### List active cron jobs

```bash
openclaw cron list
```

### Edit a cron job

```bash
# Export current job
openclaw cron export nightly-extraction-agent-1 > nightly-extraction-cron.json

# Edit the file
vim nightly-extraction-cron.json

# Re-add (or update)
openclaw cron update < nightly-extraction-cron.json
```

### Remove a cron job

```bash
openclaw cron remove nightly-extraction-agent-1
```

### Manually trigger a job (for testing)

```bash
openclaw cron trigger nightly-extraction-agent-1
```

## Cost Estimation

**Monthly cost for nightly + morning (using Haiku):**

Assumptions:
- Nightly extraction: ~500 tokens input, ~300 tokens output
- Morning brief: ~1000 tokens input, ~200 tokens output
- Days per month: 30

**Calculation:**
- Nightly: 30 × (500 + 300) = 24,000 tokens/month
- Morning: 30 × (1000 + 200) = 36,000 tokens/month
- **Total:** 60,000 tokens/month @ $0.25/1M = **$0.015/month** (~$0.18/year)

**With Opus instead:**
- Same tokens @ $15/1M = **$0.90/month** (~$11/year)

**Savings:** 60× cheaper with Haiku.

## Troubleshooting

### Job isn't running

1. Check timezone: `echo $TZ` vs config timezone
2. Check schedule syntax: Use `cronline.io` to validate cron expression
3. Check agent availability: `openclaw status`
4. Check logs: `openclaw cron logs nightly-extraction-agent-1`

### Extraction output looks wrong

1. Check `memory_path` points to daily notes: `ls memory/`
2. Check `output_path` exists: `ls life/`
3. Manually trigger for logs: `openclaw cron trigger nightly-extraction-agent-1 --verbose`

### Brief not delivering to Telegram

1. Check `delivery_channel` is `"telegram"`
2. Check `delivery_target` is correct chat ID (should be numeric string)
3. Verify bot token is configured: `openclaw config list`
4. Test manually: `openclaw cron trigger morning-brief-agent-1 --verbose`

## Advanced Customization

### Custom extraction schedule per weekday

Two separate jobs:

```json
{
  "name": "nightly-extraction-weekdays",
  "schedule": "0 23 * * 1-5",
  "agent_id": "agent-1",
  ...
}

{
  "name": "nightly-extraction-weekends",
  "schedule": "0 10 * * 0,6",
  "agent_id": "agent-1",
  ...
}
```

### Different models for different times

High-stakes morning brief (Sonnet), cheap nightly extraction (Haiku):

```json
{
  "name": "nightly-extraction",
  "model": "anthropic/claude-haiku-4-5",
  ...
}

{
  "name": "morning-brief",
  "model": "anthropic/claude-sonnet-4-6",
  ...
}
```

### Store brief to file instead of Telegram

```json
{
  "name": "morning-brief",
  "delivery_channel": "file",
  "delivery_target": "logs/briefs/morning-$(date +%Y-%m-%d).txt",
  ...
}
```

## Research Backing

**MemPO (arXiv:2603.00680):** Cost-conscious operations via cheap Haiku models (self-managed memory reduces token cost 67–73%).

**SuperLocalMemory (arXiv:2603.02240):** Local-first storage with provenance tracking (who/when/what). Cron jobs write to local `items.json` with full source metadata.

**Retrieval Bottleneck (arXiv:2603.02473):** Brief synthesis focuses on hot/warm facts, not dense summaries. Cheap writes via cron, smart retrieval via morning job.
