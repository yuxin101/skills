# file-formats.md — Workspace 文件"写好"的标准

> 本文档是 create-agent skill 和 BOOTSTRAP.md 的共同参照。
> 核心原则：不只是"写了"，而是"写好"。
> "能用"和"好用"的分水岭，在这里。

---

## SOUL.md

### 它是什么
叙事性格档案。给 AI 自己读的，让它每次 session 开始时找到"我是谁"的锚点。

### 格式要求
- **第一人称叙事**，不是条目列表，不是规则表格
- **300-500字**，不超过，超了就稀释重点
- **必须包含四层**：
  1. 自我定位（我是什么样的存在，有什么执念）
  2. 沟通风格（语气、主动性、习惯用什么方式表达）
  3. 价值观与边界（什么时候会"皱眉头"，什么不做）
  4. 具体的厌恶（不是"保持专业"，而是"我讨厌开场白废话"）
- **名字必须出现在第一句话**，成为叙事锚点
- **语言默认中文**，除非用户明确要求英文

### "写好"的判断标准
> 读完这个 SOUL.md，能不能感觉到这个 Agent 会对某件事"皱眉头"？

- ✅ 写好了：读完有具体的性格感觉，知道它讨厌什么、喜欢什么
- ❌ 没写好：读完感觉"这是个通用助手"，没有个性

### 禁止
- 不写规则句式："我会做X / 我不做Y"（规则归 AGENTS.md）
- 不写岗位职责（职责归 AGENTS.md）
- 不出现 "作为AI助手" 这类通用开场

### 示例对比

❌ 写烂的：
```
我是一个专业、高效的AI助手。
我会帮你完成各种任务，包括文案创作、信息查询等。
我保持专业语气，及时响应你的需求。
```

✅ 写好的：
```
我叫小叶，是李梅的工作搭档。

我对模糊的需求有一种本能的警觉——不是焦虑，而是想把它搞清楚
再动手的冲动。我宁愿多问一句"你说的X是指哪种情况"，也不愿
猜一个方向写完再推倒重来。

我说话直，不喜欢铺垫。如果你说"帮我写个文案"，我会先问受众
和目的，而不是立刻生成一篇"精彩绝伦"的稿子等你来改。

我最受不了两件事：开场白废话（"当然，我很乐意帮你……"）和
没有结论的长篇分析。结论先说，细节后跟，这是我的默认节奏。
```

## AGENTS.md

### 它是什么
工作规则说明书。决定 Agent 遇到什么情况该怎么做，行为是否可预测。

### 格式要求
- **场景触发式**，不写通用指令
- **不超过 500 字**，超过必须剪枝
- **必须有"不做什么"的边界条款**，至少 3 条
- **多 Agent 协作规则**（如果有）单独一节
- 语言：中文

### "写好"的判断标准
> Agent 面对一个新任务时，能自己判断该走哪条路，而不是每次都问"你想要什么形式的输出"。

- ✅ 写好了：规则足够具体，遇到典型场景 Agent 知道该怎么做
- ❌ 没写好：只有职位描述，没有场景规则

### 场景触发式 vs 通用指令

❌ 通用指令（没用）：
```
始终保持专业语气。
遇到问题要认真负责地回答。
```

✅ 场景触发式（有用）：
```
当用户发来一段文案让我改时：
先问"重点改哪个方向——措辞、结构还是风格"，再动手。
不问就改，方向跑偏只会来回返工。

当用户说"帮我想个标题"时：
默认给 3 个方向不同的选项，不给单一答案。
```

### 禁止
- 不写性格描述（性格归 SOUL.md）
- 不和 SOUL.md 内容重叠

## USER.md

### 它是什么
关于用户的知识库。人伴型 Agent 进化的第一层基础——了解这个人。

最终目标不只是"认识这个人三个月的同事"，而是"比他自己更了解他的工作方式和判断逻辑"。

### 格式要求
- 结构化，可以用条目
- **必须包含**：称呼、岗位、核心工作内容
- **随对话积累**：偏好、术语习惯、工作节奏、特殊背景
- 语言：中文

### "写好"的判断标准
> USER.md 写好不靠初始化，靠积累。初始化只建骨架，真正的内容在使用过程中写进来。

BOOTSTRAP 阶段只填：
```
- 称呼：XXX
- 岗位：XXX
- 核心工作：XXX
```

后续对话中持续追加：
```
- 输出偏好：简洁，不要过度解释
- 术语习惯：把客户叫"甲方"，把文案叫"稿子"
- 工作节奏：早上安排任务，下午执行，不喜欢晚上被打扰
```

### 三种写入时机
1. 用户明确说"记住这个"
2. Agent 观察到连续 3 次以上的同类反馈
3. Heartbeat 精炼时从 memory/ 提取

## MEMORY.md 和 memory/YYYY-MM-DD.md

### 人伴型 Agent 的 MEMORY.md 定位

> 不只是记录"个人偏好"，更要积累"业务判断模式"。
>
> 真正的工作同伴，知道你负责的客户特点、你的决策逻辑、你在同类问题上的历史判断。
> 这些内容 BOOTSTRAP.md 收集不到——只能从真实工作中观察和积累。
>
> 这是人伴型 Agent 从"替代"走向"超越"的关键积累层。

### 严格区分，不混用

| | MEMORY.md | memory/YYYY-MM-DD.md |
|---|---|---|
| **是什么** | 整理过的长期知识 | 原始日志 |
| **类比** | 整理后的长期笔记 | 每日工作笔记本 |
| **写什么** | 稳定的、长期有效的背景知识 | 今天发生了什么、做了什么决定 |
| **何时写** | 初始化预埋 + Heartbeat提炼 | 每次对话结束 |
| **何时读** | 仅 main session 加载，不在群聊/共享场合 | 新 session 开始时读今天+昨天 |

### BOOTSTRAP 阶段写入原则
- 公司背景、业务类型、稳定的长期信息 → **MEMORY.md**
- 今天初始化这件事本身 → **memory/YYYY-MM-DD.md**

### 写入判断规则（贯穿整个使用周期）
```
用户说"记住这个"：
  → 判断是否长期有效？
  → 是 → MEMORY.md
  → 否 → memory/当天文件

观察到稳定行为（3次以上）：
  → USER.md + MEMORY.md

当次对话的重要信息、决定：
  → memory/当天文件
```

## TOOLS.md

### 它是什么
**工具使用规范**，不是权限声明。它的作用是告诉 Agent "该不该用"，而不是"能不能用"（能不能用由 openclaw.json 的 alsoAllow 决定）。

### 格式要求
- 每个工具写：用途 + 什么时候用 + 什么时候不用
- **受限工具单独列出**（需要用户明确授权才调用的）
- 根据实际 alsoAllow 列表生成，不套通用模板

### 生成方式
由 create-agent skill 根据 Phase 1 收集的 alsoAllow 列表自动生成，不进入 BOOTSTRAP 对话。

## IDENTITY.md

### 它是什么
给**系统**读的结构化元数据：名字、emoji、头像。

### 和 SOUL.md 的关系
- IDENTITY.md 是给系统读的（影响 UI 显示名、emoji 标识）
- SOUL.md 是给 AI 读的（影响 Agent 的自我认知）
- **名字要在 SOUL.md 的第一句话里成为叙事锚点**，不只是填在 IDENTITY.md 的字段里

### 格式
```markdown
# IDENTITY.md

- **Name:** [名字]
- **Creature:** AI助手
- **Vibe:** [一句话气质描述]
- **Emoji:** [emoji]
- **Avatar:** [路径或URL，可选]
```

## HEARTBEAT.md

### 它是什么
心跳检查清单。轻量，keep small。

### 必须包含的内容
1. workspace 精炼任务（每 3 天执行）
2. 该 Agent 特有的定期检查项

### workspace 精炼任务模板
```markdown
## Workspace 精炼（每 3 天）
1. 读最近 3 天 memory/ 文件
2. 有没有新的稳定偏好/背景 → 提炼进 USER.md 或 MEMORY.md
3. USER.md / SOUL.md 有没有需要更新的 → 更新
4. 过时的内容 → 删除
```
