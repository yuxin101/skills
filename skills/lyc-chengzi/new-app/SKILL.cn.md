# Internal App Bootstrap

## 目的

当团队成员想从零开始构建一个新的内部应用，并且希望采用一致、可评审、文档优先的流程，而不是直接跳进代码时，使用这个 skill。

这个 skill 沉淀了 i-skills 从空工作区到可运行全栈应用的构建方法，包括：

- 先做需求澄清，再进入实现
- 先沉淀架构和流程决策
- 前后端分离的项目结构
- 认证规划
- 数据库与 schema 规划
- 分阶段实施
- 验证与运维可用性

## 适用场景

当用户提出以下诉求时使用这个 skill：

- 从零开始做一个新应用
- 构建标准化的内部工具
- 用清晰流程启动一个全栈产品
- 帮团队遵循可复用的应用交付流程
- 把想法落成一个文档和实现对齐的 MVP

不要把这个 skill 用在小范围 bug 修复、单独 UI 微调、或单个后端接口变更上。

## 核心原则

1. 需求优先
2. 架构先于实现
3. 文档先于代码
4. 中英文文档保持同步
5. 里程碑要小且可验证
6. 验证是交付的一部分，不是最后补做

## 可复用规范基线

使用这个 skill 时，要把 `docs/` 下的规范视为可迁移的通用基线，而不是只服务于当前项目的零散说明。

### 按领域划分的文档权威来源

- `docs/api-development.md` 和 `docs/api-development.cn.md` 定义后端契约、路由、schema、迁移、认证与运维诊断规则。
- `docs/ux-development.md` 和 `docs/ux-development.cn.md` 定义导航、表单、反馈、认证体验、响应式和文案规则。
- `docs/coding-standards.md` 和 `docs/coding-standards.cn.md` 定义仓库级结构、工具链、命名、验证和变更纪律。
- `docs/copilot-guide.md` 和 `docs/copilot-guide.cn.md` 定义文档优先工作流，以及任务推进顺序。

### 在新应用启动中应复用的通用规则

- 前端、后端和文档必须围绕同一套产品术语保持一致。
- 公共契约必须显式且稳定。
- 业务规则应放在后端 service，而不是只体现在前端行为里。
- 路由处理函数和页面组件都应保持轻量。
- JavaScript 包管理器必须统一，只选一种。
- 验证、健康检查和诊断能力都属于交付范围的一部分。
- 认证、存储和数据库边界要显式设计，不要依赖隐藏副作用。
- 如果正式环境使用 PostgreSQL，就要显式管理 schema 和 migration。
- 如果 UI 依赖后端数据或权限，必须同时设计兜底态和错误态。
- 如果没有配套文档和验证，任务不能算完成。

## 标准流程

### 1. 先澄清业务问题

先把应用的业务轮廓问清楚。

至少要确认：

- 谁是使用者
- 核心任务是什么
- 应用面向内部还是外部
- 数据是否敏感或受监管
- MVP 必须包含什么
- 明确不做什么

使用 `templates/requirements-template.md` 和 `templates/requirements-template.cn.md` 中的需求模板。

### 2. 明确不可妥协的约束

在提架构方案前，先确认以下约束：

- 认证提供方
- 部署环境
- 数据库目标
- 存储策略
- 是否需要上传能力
- 访问控制与可见性规则
- 应用是否执行用户上传产物，还是只存储/引用它们
- 是否需要移动端适配

### 3. 输出具体方案

需求明确后，再给出完整方案，至少覆盖：

- 前端框架
- 后端框架
- 数据库
- 认证集成方案
- 存储方案
- 推荐的项目结构
- MVP 页面或工作流地图
- 主要风险和取舍

在用户确认方案之前，不要直接开始实现。

### 4. 先更新文档

在改代码前，先补齐或更新中英文文档。

在动手前，先读取 `docs/` 中对应规范，并把它们视为“该记录什么、该验证什么”的权威来源。

一个新的产品或流程，至少应先写清楚：

- API 预期
- UX 预期
- 如果引入了新约束，要更新编码规范
- 如果团队工作流会变化，要更新工作指引

### 5. 建立工作区和工具链

有意识地搭建项目结构。

典型基线：

- 前端放在 `apps/web`
- 后端放在 `apps/api`
- 文档放在 `docs`
- 共享工作流指引放在 `.github`

尽早统一工具链：

- 只选一个 JavaScript 包管理器
- 定义开发任务
- 约定环境变量文件
- 避免混合锁文件和半成品脚手架状态

### 6. 用纵向切片构建 MVP

优先做“端到端可跑通”的薄切片，而不是先堆很多孤立代码。

推荐顺序：

1. 应用壳层和导航
2. 后端模型和核心路由
3. 前端页面接上文档化契约
4. 创建或上传工作流
5. 审核或审批工作流
6. 认证集成
7. 运维与诊断工具

### 7. 有意识地集成认证和身份

如果应用需要认证：

- 区分浏览器端和服务端认证职责
- 不要硬编码 secret
- 记录运行时配置
- 必要时提供最小 profile/debug 页面帮助排查身份问题
- 敏感身份字段由服务端推导

### 8. 把数据库当成产品能力的一部分

如果正式环境使用 PostgreSQL：

- 显式配置数据库和 schema
- 不依赖隐式默认值
- 通过 Alembic 管理 schema 变更
- 提供数据库、schema 和 revision 状态的诊断能力

### 9. 补齐运维辅助能力

在把应用称为“可用”之前，补上：

- 常用开发任务
- 健康检查或诊断命令
- 迁移命令
- 清晰的本地启动说明

### 10. 交付前做验证

验证必须覆盖这次改动涉及的真实表面。

常见检查项：

- 前端构建
- 后端导入或启动健全性检查
- 代表性接口检查
- 如果接了认证，要验证认证流
- 如果涉及数据库，要验证 migration 或 schema 状态

使用 `templates/delivery-checklist.md` 和 `templates/delivery-checklist.cn.md` 中的交付清单。

## 预期交付物

一个完整的应用启动任务，通常应产出：

- 一份需求摘要
- 一份确认过的架构方案
- 一组已更新的中英文文档
- 可运行的前后端基础骨架
- 明确的环境和任务配置
- 已验证的核心流程
- 简洁的变更总结和后续建议

## 推荐输出结构

使用这个 skill 时，建议按下面结构组织结果：

1. 需求摘要
2. 架构方案与 MVP 边界
3. 文档更新
4. 实施计划
5. 代码和配置变更
6. 验证结果
7. 后续建议

## 常见失败模式

避免这些问题：

- 需求没确认就开写
- 跳过相关 `docs/` 文件，只根据现有代码猜规则
- 文档落后于实现
- 把 mock 行为当成正式契约
- 混用多个包管理器
- 接了认证却没有排障路径
- 在共享数据库环境中依赖应用自动改表
- 只停留在脚手架层，没有验证真实工作流
