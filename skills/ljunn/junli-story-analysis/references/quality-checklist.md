# 拆书质检清单

用于 `君黎AI拆书` 在正式交付前做自检。

## 1. 工作区是否可继续使用

- [ ] `workspace/manifest.json`、`chunk-index.md`、`stage-plan.md`、`use-guide.md` 都存在
- [ ] `drafts/` 下 3 份草稿都存在
- [ ] `outputs/书籍画像.md` 和 6 张方法卡都存在
- [ ] `chunks/raw/` 与 `extraction-cards/` 数量一致
- [ ] 如工程断档，先运行 `python3 scripts/story_analysis_pipeline.py repair <工程目录>`

## 2. 提取卡层

- [ ] 每个已处理 chunk 都回答了 5 件事：叙事功能、方法归属、案例候选、书籍画像信号、关键证据
- [ ] 提取卡不是长剧情复述，也不是读后感
- [ ] 关键证据不是空行，且能回指原文
- [ ] 典型程度、是否建卡、是否跨 chunk 合并已明确

## 3. 草稿层

- [ ] `书籍画像草稿` 已形成题材、风格、节奏、扩长方式和适用边界判断
- [ ] `方法候选笔记` 里 6 个模块都不是空壳
- [ ] `案例候选笔记` 已按“值得检索、值得复用、值得对照”筛过强弱
- [ ] 草稿在阶段边界有更新，不是等到最后一次性硬写

## 4. 书籍画像

- [ ] 说明了“这本书最适合参考什么、最不适合照搬什么”
- [ ] 说明了“不适合直接学什么”
- [ ] 强项与弱项判断清楚，没有把所有模块都写成平均强
- [ ] 不是把剧情简介换个标题

## 5. 方法卡

- [ ] 6 张方法卡都已实填，不是模板状态
- [ ] 每张卡都有明确 chunk 证据锚点
- [ ] 每张卡都回答了：这本书怎么做、为什么有效、能学什么、有哪些节奏或结构特征、什么不能照搬
- [ ] 规律来自跨 chunk 归纳，而不是单段桥段强行上纲
- [ ] 没把单本书的题材红利误写成通用铁律

## 6. 案例卡

- [ ] 只保留真正值得建卡的案例，不为了凑数硬做弱案例
- [ ] 每张案例卡都说明了最适合参考什么场景
- [ ] 每张案例卡都写清了关键动作链、为什么有效、不可照搬
- [ ] 标签是可检索的，不是只写泛词
- [ ] 如果案例依赖跨 chunk，已经明确合并范围

## 7. 收束判断

- [ ] 如果还有大量 `extraction-cards/*.md` 未处理，不要急着交付
- [ ] 如果 `outputs/书籍画像.md` 或方法卡仍是模板，先回去补
- [ ] 如果没有正式案例卡，先判断是真的没有强案例，还是筛选没做完
- [ ] 如使用项目脚本，正式交付前至少先跑一次：

```bash
python3 scripts/story_analysis_pipeline.py check <工程目录>
```

## 8. 一条底线

- [ ] 最终交付的是创作参考资产，不是剧情报告、文学赏析或读后感合集
