# 日报生成流程

最短闭环：`读取关注上下文 → 读取原始新闻 → 过滤去重 → 高价值筛选 → 全文阅读 → 生成日报 → 生成早报导读 → 更新关注清单`

## 两类入口

### cron 正式入口

- 默认读取过去 24 小时
- 生成正式日报
- 发到目标群
- cron 回执发给用户本人

### 飞书私聊入口

- 可临时指定时间范围
- 可只试跑不正式发群
- 默认发给当前发起人
- 适合验证模板、语气和筛选效果

## 执行管道（含门控检查点）

### Step 0：加载关注上下文

执行任何筛选之前，先读取两份上下文：

1. `references/watch_chains.md` — 产业链关注图谱（当前在跟什么链路、判断基线是什么）
2. `watch_target.json` 对应的飞书关注清单表 — 读取状态为"待验证"的问题

这两份上下文将作为 Step 3 高价值判断的输入。

### Step 1：读取原始新闻

从飞书原始新闻表读取本轮日报对应的数据。查询层优先按文本字段 `报道窗口键` 过滤（例如 `2026-03-18-0900`）；如果该字段不可用，再退回按发布时间窗口过滤。

**门控 ◆** 检查读取到的记录数 > 0。为 0 说明该窗口无数据 → 停止，报告"该窗口无原始新闻"。

### Step 2：过滤去重

按 `原文链接`、`标题哈希 + 发布时间` 去重，排除时间不可信或明显噪声记录。

**门控 ◆** 检查去重后记录数 > 0。为 0 说明全部是重复或噪声 → 停止，报告原因。

### Step 3：筛选高价值事件

结合 Step 0 的关注上下文，从候选新闻中筛出值得进入正文的高价值事件。

#### 两层筛选

**第一层：快速排除**（基于标题 + 摘要 + 来源）

排除：
- 纯产品促销、广告
- 重复报道同一事件的跟风稿（保留首发或信息最全的一条）
- 非 AI 主题混入
- 来源不可信

**第二层：高价值判断**（基于标题 + 摘要 + 来源 + 关注上下文）

优先入选（按优先级排序）：
1. 命中关注清单里"待验证"问题的新闻 — 昨天说要盯的，今天有了新证据
2. 命中产业链图谱里活跃链路或关注节点的新闻
3. 官方发布（博客、公告、研究论文）
4. 多源交叉验证的事件（≥2 个独立来源报道）
5. 监管 / 安全 / 融资 / 并购 / 重大产品变更

目标数量：3-8 条进入全文阅读。超过 8 条时，按影响面和证据强度排序取 top 8。不足 3 条时，适当放宽标准。

**门控 ◆** 检查筛选出的高价值事件数 ≥ 1。为 0 说明本轮无值得报道的事件 → 停止，报告"本轮无高价值事件"。

### Step 4：阅读全文

对高价值事件逐条打开原文链接阅读全文。这一步不可跳过——没读全文就不能写"判断更新"。

用 `curl -s "https://r.jina.ai/URL"` 读取原文。

### Step 5：生成日报

由大模型按 `references/output-template.md` 的模板生成 signal 化日报，更新当天日报文档。

**飞书文档内容必须和本地 MD 副本完全一致，不得省略任何字段。** 每条 signal 必须包含全部字段：事件描述、问题还原、所属主题、所属链路、所属节点、旧方案、新变化、为什么重要、待验证问题、判断更新、关键原文链接。缺少任何一个字段都不合格。

**门控 ◆** 检查日报文档创建/更新成功，且包含完整的一级结构（今日新增信号、分主题观察、今日判断更新、高优先级待验证问题、关键原文链接）。

生成后，同时保存一份到 `workspace/daily_reports/YYYY-MM-DD.md`。

### Step 6：生成早报导读

基于日报文档内容生成早报导读消息。

### Step 7：更新关注清单

根据本轮日报产出，更新飞书关注清单表（`watch_target.json`）：

1. **新增**：本轮日报产出的"待验证问题"和"高优先级待验证问题"写入清单，每条记录必须填写以下字段：
   - `问题`：待验证问题的完整描述
   - `所属链路`：该问题对应的产业链路名称
   - `来源日报`：本轮日报的飞书文档 URL（不是本地文件路径）
   - `优先级`：`高` 或 `普通`
   - `状态`：设为 `待验证`
2. **回答**：如果本轮某条 signal 回答了清单中"待验证"的问题，必须同时更新两个字段：
   - `状态`：改为 `已回答`
   - `回答摘要`：用 1-2 句话写明本轮 signal 给出的答案要点，不能留空
3. **不删除**：已过期的条目不自动删除，留给人工定期清理

### Step 8：投递 + 过程日志

- cron 模式：发到目标群（chat_id 由 cron 传入），回执发给用户本人
- 飞书私聊模式：只发给当前发起人

#### cron 发群顺序（严格按以下顺序，分三条消息）

1. **第一条：晨间导读**（Step 6 生成的早报导读）
2. **第二条：日报文档链接**（《AI日报｜当天日期》的飞书文档标题和链接）
3. **第三条：原始新闻表链接**（bitable_target.json 对应的多维表格链接）

群里只发成品，不发系统执行摘要、不发过程日志。

#### 飞书知识库（cron 传入目标知识库地址）

日报文档必须成功写入飞书知识库。如果写入失败，必须在回执中明确报错，不能静默跳过。

输出过程日志：

```
日报生成完成：
- 报道窗口键：{window_key}
- 读取原始新闻：{raw_count}
- 去重后保留：{dedup_count}
- 排除噪声/时间不可信：{dropped_count}
- 关注清单命中：{watch_hit_count}（哪些待验证问题被新证据触及）
- 最终进入正文：{signal_count}
- 高价值事件：{selected_event_titles}
- 关注清单更新：新增{new_questions}条待验证，回答{answered}条
```

## signal 最小骨架

每个进入正文的 signal 至少补齐：

- `headline`
- `theme`
- `chain`
- `node`
- `problem_statement`
- `old_solution`
- `what_changed`
- `why_it_matters`
- `open_questions`
- `evidence_links`

## 写作规则

- 日报不要写成新闻列表
- 先筛选，再写作；不要把原始新闻逐条翻成中文就当日报
- 每个 signal 至少说明：属于哪条链/哪个节点、真正想解决的问题、相比旧方案发生了什么变化、还缺哪些关键证据
- 证据不足时，写"待验证问题"，不要写成定论
- 如果没有阅读全文，就不要写"判断更新"
- 如果某条 signal 回答了关注清单里的待验证问题，在 signal 中明确标注

## 日报完成标准

一份合格的日报，至少同时满足：

1. 有 `今日新增信号`
2. 有 `分主题观察`
3. 有 `今日判断更新`
4. 有 `高优先级待验证问题`
5. 有 `关键原文链接`

## 早报完成标准

一条合格的早报导读消息，至少同时满足：

1. 开头包含当天绝对日期，默认地点为 `福州`
2. 语气温和、克制、有一点人文关怀，但不鸡汤
3. 包含 3 条最重要信号，每条解释"为什么今天值得先看"
4. 包含 1 个今天值得继续关注的问题，以及当天日报文档链接
5. 附上原始新闻多维表格链接，方便回查

## 原始表保护规则

- 默认只读原始新闻表
- 不批量更新、批量删除、重置记录
- 不把 signal、主题判断、归并结果回写到原始表
