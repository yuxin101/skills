# 采集常见坑

## 执行器容易进入调试模式

- 典型症状：先读一堆文档，再跑 `python3 -c`、`web_search`、手工造 JSON。
- 正确行为：`agent-reach -> 轻量提取 -> 入表 -> 结束`
- 如果已经成功写入，就不要再补抓、补写、重跑字段转换。

## 时间错乱通常不是”时区”这么简单

- 先看 `原始时间` 是否保真。
- 如果 `原始时间` 已经是完整绝对时间，`发布时间` 应直接由脚本标准化生成。
- 不要用抓取时间、当天零点或模型猜测去兜底。

## LLM 手算 Unix 时间戳必定出错

- 飞书日期时间字段需要 Unix 毫秒时间戳。
- 实测 LLM 手算时间戳经常差 1 年、数天甚至数小时（例如 2026-03-18 算出 2025-03-18）。
- **所有时间解析和时间戳转换必须走 `normalize_agent_reach.py` + `build_source_items.py`**，不要在 JSON 里直接手写 Unix 时间戳数字。
- 如果跳过这两个脚本直接拼 bitable JSON，时间一定是错的。

## 浏览器最容易把采集变成“看页面猜字段”

- 默认 cron 不用浏览器。
- 浏览器更适合后续阅读全文，不适合当原始字段主来源。

## `agent-reach` ≠ `web_fetch` 工具

- `agent-reach` 的通用网页 channel 是 `exec` 运行 `curl -s "https://r.jina.ai/URL"`。
- `web_fetch` 是另一个工具，不走 jina.ai，只拿原始 HTML 静态内容，SPA 页面几乎拿不到任何新闻条目。
- 典型症状：页面返回 200 但 rawLength 只有几十到几百字节，内容只有页面简介或版权脚注。
- 遇到这种情况说明用了错误工具，必须换回 `curl r.jina.ai`。

## `agent-reach` 能拿文本，不等于天然有结构化字段

- 它适合读取列表页文本。
- 字段提取、摘要清洗、时间标准化仍要走本 skill 的规则。

## 摘要为空不一定是 bug

- 如果页面没有独立摘要，或者抓到的是标题复读，`原文摘要` 留空是正确结果。
- 不要为了让表“看起来完整”而把标题重复写进摘要。

## 只抓到单一来源，通常说明链路没跑完整

- 正常主链不该长期只剩一个源。
- 如果只抓到财联社，优先检查：
  - 其他源是否被过滤掉
  - 本轮是否中途进入调试模式
  - 是否只验证了单源而不是全主链
