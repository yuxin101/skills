# 周报 + 深入分析生成流程

三层产出职责分工：
- **日报**（ai_news_reporter）：从每天的信息里提炼信号
- **周报**（本 skill）：review 本周信号，刷新观察框架，识别需要深入分析的课题
- **深入分析**（本 skill）：独立文档，对识别出的课题做全面 research

最短闭环：`读取日报 → 读取观察链 → 信号关联 → 合成趋势 → 生成周报（review） → 识别深入分析选题 → 生成深入分析 → 更新观察链 → 审视关注清单`

## 两类入口

### cron 正式入口

- 默认读取本周一至今天的日报
- 生成正式周报 + 深入分析
- 发到目标群
- cron 回执发给用户本人

### 飞书私聊入口

- 可临时指定日期范围
- 可只试跑不正式发群
- 默认发给当前发起人

## 执行管道（含门控检查点）

### Step 1：读取本周日报

按日期范围读取日报，两个来源按优先级：

1. **本地文件**：`workspace/daily_reports/YYYY-MM-DD.md`
2. **飞书 wiki**：本地没有时，从飞书日报文档读取

**门控 ◆** 检查读取到的日报数 ≥ 3。少于 3 篇说明本周数据不足 → 停止，报告"本周日报不足 3 篇，无法生成有效周报"。

### Step 2：读取观察链上下文

读取三份上下文：

1. `../ai_news_reporter/references/watch_chains.md` — 产业链关注图谱（当前判断基线）
2. 飞书关注清单表（`../ai_news_reporter/watch_target.json`）— 本周待验证问题的状态
3. `references/analysis_framework.md` — 深入分析框架（Step 6 使用）

前两份用于 Step 3 判断信号该归入哪条趋势线、哪些待验证问题本周被回答了。第三份用于 Step 6 深入分析。

### Step 3：提取信号 + 跨天关联

从所有日报中提取 signal，然后寻找跨天关联，合成趋势线。

#### 关联依据（命中任意一条即可关联）

1. **同主题同链路**：属于同一个主题的同一条链路（对照 watch_chains.md）
2. **因果链**：某天的事件是前几天事件的后续反应
3. **同一玩家**：同一家公司/组织本周的多个动作
4. **对立面**：同一问题的正反两面
5. **待验证问题被回答**：某天日报的"待验证问题"在后续日报中被新 signal 回答

不要硬关联。没有明确依据的 signal 放在"本周独立信号"里。

#### 趋势线合成

对每条趋势线：

1. 写一句话概括这条趋势在说什么
2. 列出信号链（日期 + signal + 关键变化）
3. 对照 watch_chains.md 的当前判断基线，写明本周是"加强"、"修正"还是"无变化"
4. 提取本周验证了的问题和下周值得关注的方向

**门控 ◆** 检查至少产出 1 条趋势线或 3 条独立信号。都为 0 说明信号提取失败 → 停止，检查日报格式。

### Step 4：生成周报（Review 文档）

周报的定位是 **review**，不是深入分析的载体。核心任务：

1. **趋势回顾**：本周有哪些趋势线？信号链是什么？
2. **观察框架刷新**：对照 watch_chains.md，本周哪些判断被加强/修正/新增？
3. **识别深入分析选题**：从趋势线中挑出 1-2 个最值得深入分析的课题（见 Step 5）

按 `references/output-template.md` 的"周报模板"生成。

**门控 ◆** 检查周报包含完整一级结构（本周趋势、本周独立信号、本周判断变化、深入分析选题、下周关注清单）。

生成后：
1. 保存到 `workspace/weekly_reports/YYYY-MM-DD.md`（以周报覆盖的最后一天日期命名）
2. 同步到飞书 wiki 文档

**飞书文档内容必须和本地 MD 副本完全一致，不得省略。**

### Step 5：识别深入分析选题

从本周趋势线中选出 **1 个**最值得深入分析的课题。这一步只做选题，不做分析。

#### 选题标准

按以下优先级选择：
1. 本周信号链最长（跨天关联最多）的趋势
2. 判断基线发生实质性变化的趋势
3. 对产业影响面最大的趋势

#### 输出

在周报文档中写明：
- 选题：{课题名称}
- 选题理由：{一句话}
- 涉及趋势线：{哪些}
- 深入分析文档链接：（Step 6 完成后回填）

### Step 6：生成深入分析（独立文档）

这是一个**独立的产出**，不嵌入周报正文。深入分析的目标是对识别出的课题做全面 research，产出一份可独立阅读的分析文档。

#### 执行要求

1. **必须阅读原文**：对该课题涉及的所有 signal，用 `curl -s "https://r.jina.ai/URL"` 逐条阅读原文。这一步不可跳过——没读原文不能写深入分析
2. **采集原文配图**：阅读原文时，从 jina 返回的 markdown 中提取 `![...](url)` 格式的图片，执行以下筛选流程：
   - **排除**：文件名或路径含 `logo`、`icon`、`favicon`、`avatar`、`banner`、`ad` 的图片；URL 中图片尺寸参数明显 ≤ 100px 的装饰图
   - **保留**：数据图表（chart、graph、benchmark）、架构图（diagram、architecture、flow）、产品截图（screenshot、UI）、信息图（infographic）
   - **数量**：每篇原文最多保留 2 张，整个深入分析最多 3 张
   - **格式**：以 `![一句话说明图片内容](原始URL)` 插入到分析的相关段落中（不要集中堆在末尾）
   - 如果某篇原文没有有价值的图片，跳过即可，不强求
3. **按 analysis_framework.md 执行结构化分析**：读取分析框架，按核心工具（玩家拆解、竞争本质、终局判断、自我追问）执行，辅助工具按话题性质选用
4. **自我追问循环**：初稿写完后提 3 个尖锐反问 → 逐一回答 → 修正初稿
5. **观察链回写**：分析完成后检查 `watch_chains.md` 是否需要更新，在 Step 7 中一并执行

#### 输出

按 `references/output-template.md` 的"深入分析模板"生成独立文档。

1. 保存到 `workspace/deep_analysis/YYYY-MM-DD-{topic_slug}.md`
2. 同步到飞书 wiki 文档
3. 回填周报中的深入分析文档链接

**门控 ◆** 检查深入分析包含：选题理由、原文阅读数量、玩家拆解表、竞争本质、终局判断、不确定性。缺少任何一项不合格。

**篇幅**：1200-2000 字（不含图片和表格），不要写成论文，保持可读性。

### Step 7：更新 watch_chains.md

这是观察链判断基线的**主要更新时机**。这一步不可跳过，即使本周判断没有变化也要确认。

基于本周趋势线 + 深入分析结论，对 `watch_chains.md` 做三类操作：

1. **新增链路**：本周出现了图谱中没有的新链路（至少 2 条 signal 支撑），新增并写明初始判断
2. **修改判断**：本周趋势线或深入分析提供了新证据，更新对应链路的"当前判断"和"关注节点"
3. **降级链路**：连续 2 周没有任何 signal 命中的链路，在判断后追加"（近期不活跃）"标注

**门控 ◆** 完成后必须满足以下全部条件，否则不进入下一步：
- `watch_chains.md` 文件头部"最近更新"日期已改为本周周报日期
- 过程日志中已记录：新增 N / 修改 N / 降级 N（即使全为 0 也要写）
- 如果本周有判断变化但 watch_chains.md 未修改 → 不合格，必须回去改

### Step 8：审视关注清单

这一步不可跳过。批量审视飞书关注清单表中的"待验证"问题：

1. **兜底检查**：本周被回答的问题 → 确认已标记为"已回答"且`回答摘要`已填写（日报 Step 7 应该已做，这里兜底）
2. **超期预警**：超过 7 天未被回答的 → 在过程日志中逐条列出（问题、来源日期、已等待天数），建议降级但不自动修改状态
3. **新增写入**：本周趋势线和深入分析产出的新问题 → 写入清单，每条必须填写：`问题`、`所属链路`、`来源日报`（飞书文档 URL）、`优先级`、`状态`（待验证）

**门控 ◆** 过程日志中必须包含：本周回答 N 条 / 超期 N 条 / 新增 N 条（即使全为 0 也要写）。

### Step 9：投递 + 过程日志

- cron 模式：周报链接 + 深入分析链接发到目标群，回执发给用户本人
- 飞书私聊模式：只发给当前发起人

投递顺序（分三条消息）：
1. 第一条：周报（review）文档链接
2. 第二条：深入分析文档链接
3. 第三条：原始新闻表链接

输出过程日志：

```
周报生成完成：
- 覆盖日期：{start_date} ~ {end_date}
- 读取日报：{report_count} 篇（本地 {local_count} / 飞书 {feishu_count}）
- 提取 signal 总数：{total_signals}
- 合成趋势线：{trend_count} 条
- 独立信号：{standalone_count} 条
- 判断更新：{judgment_update_count} 条
- 深入分析选题：{deep_analysis_topic}
- 深入分析：阅读原文 {articles_read} 篇，采集配图 {images_count} 张，自我追问 {questions_count} 轮
- 深入分析回写观察链：{deep_analysis_chain_updates}（如无变化写"无"）
- watch_chains.md 更新：新增 {new_chains} / 修改 {updated_chains} / 降级 {downgraded_chains}
- 关注清单审视：本周回答 {answered} / 超期建议降级 {overdue} / 新增 {new_questions}
```

### Step 10：分析框架自我迭代

在投递完成后，执行 `references/analysis_framework.md` 中定义的"自我迭代机制"：

1. 读取 analysis_framework.md 中"反馈日志"部分，检查是否有未处理的新反馈
2. 如果有新反馈，按照迭代流程（汇总 → 模式识别 → 框架修订 → 记录变更 → 清理日志）执行
3. 如果没有新反馈，跳过本步骤
4. 每次迭代最多改 3 处，避免过度调整

**注意**：群聊互动中收到的反馈应**先记录到反馈日志**，不要当场修改框架。等到周报生成后统一在这一步处理。

## 写作规则

- 周报是 review 文档，不是深入分析的载体——趋势回顾要精炼，深入分析单独出
- 不要把每天的 signal 原样搬过来，要合成、归纳、提炼
- 趋势判断可以比日报更大胆，但要标注证据强度（几条 signal 支撑、来源是否独立）
- 如果本周信号太少（< 5 条），趋势部分可以缩减，多写独立信号
- 对照 watch_chains.md 写判断变化，说清"原判断是什么 → 本周新证据是什么 → 更新后判断是什么"
- 少用"重磅/史诗级/前夜已至"，多说"为什么这几件事放在一起值得注意"
