# Current Baseline

Last verified: March 17, 2026.

This summary is a convenience layer. Agents must still verify official sources live before citing current law or guidance.

## Baseline statements

- The current HIPAA Security Rule remains the operative baseline for PRS assessments.
- PRS stage assignment remains primarily a Security Rule-oriented readiness exercise.
- The HIPAA Privacy Rule and HIPAA Breach Notification Rule remain relevant adjacent regimes and should be flagged separately when the workload and evidence raise those issues.
- The December 27, 2024 Security Rule NPRM remains proposed and does not replace the current rule until finalized.
- HHS/OCR guidance continues to emphasize administrative, physical, and technical safeguards, risk analysis, contingency planning, and periodic technical and non-technical evaluation.
- Health-related apps and devices are not automatically inside HIPAA scope; current HHS guidance still requires role and relationship analysis, especially for consumer apps and patient-directed API access.
- NIST SP 800-66 Rev. 2 is the current NIST implementation guide for the existing Security Rule baseline.
- HHS does not require certification under the Security Rule and warns against misleading `HIPAA compliant`, `HIPAA secure`, and `HIPAA certified` claims.
- HIPAA does not automatically displace all state-law review; more protective state-law obligations may still matter.

## Minimum current-source set

- HHS Security Rule overview
- HHS Security Rule NPRM page
- HHS risk analysis guidance
- HHS Privacy Rule overview when role, use and disclosure, or PHI scope is relevant
- HHS Breach Notification Rule overview when incident or breach readiness is relevant
- HHS misleading marketing claims guidance
- HHS role and preemption FAQs when applicability or state-law questions matter
- HHS business associates guidance where role or contracting matters
- HHS health-app and patient-access API guidance where mobile apps, connected devices, or provider APIs are involved
- NIST SP 800-66 Rev. 2

## Practical implication for PRS

- PRS stages should be mapped to the current Security Rule baseline and current official guidance.
- Privacy Rule, Breach Notification Rule, and state-law issues should be reported as adjacent boundary notes rather than silently ignored.
- Proposed rule deltas may be noted as future work, but they must not be treated as mandatory current criteria unless the user explicitly asks for a proposed-state gap analysis.
