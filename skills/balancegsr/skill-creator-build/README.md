# skill-creator-build

> **MethodCraft** series — 把经过验证的方法论编译为可执行的 AI Skill

根据你的项目需求，生成一个定制化的构建项目 Skill。

生成的 Skill 会在你的整个构建过程中持续引导——从项目规划、技术决策、分阶段实施到结项回顾。不是一次性的任务拆解，而是一个贯穿项目全程的工程伴侣。

面向所有"需要把东西做出来"的项目：软件开发、Skill 开发、技术方案落地、工具搭建——只要有明确的可交付物，它就能帮你管好从启动到交付的全流程。

## 核心特性：自带 Memory 系统

生成的 Skill 内置文件级 memory 机制，让 agent 跨对话记住你的项目进度和决策：

- **决策追溯**：每个关键技术决策自动记录到 decisions.md——选了什么方案、为什么、放弃了什么——项目再长也不会忘记"当初为什么这么选"
- **按阶段组织**：产出物按阶段分文件夹，每个阶段内再按子流（design/、tech/、implementation/）组织，结构清晰
- **跨对话恢复**：每次开新对话，Skill 自动读取 plan.md 和最近的决策记录，声明"当前进度：阶段 2 核心功能，技术决策子流"——不需要你重复交代上下文
- **智能读取优先级**（完整模式）：四层读取策略，优先加载当前阶段文件，历史阶段按需参考
- **严禁删除**：产出物更新是覆盖写入最新内容，变更历史在 decisions.md 中保留

## 快速开始

安装后，对 agent 说：

- "我想搭建一个 SaaS dashboard"
- "帮我生成一个构建 skill"
- "创建构建项目"
- "I want to build a CLI tool for X"

Creator 会用 2 轮对话了解你的情况（做什么、多大规模），然后生成一个定制的 SKILL.md。

## 它会问你什么

| 信息 | 为什么需要 | 示例 |
|---|---|---|
| 项目目标 | 确定要做什么、给谁用 | "用 Next.js 做一个内部数据看板" |
| 项目规模 | 决定阶段结构和管理深度 | "大概两周"、"三个月的长期项目" |

就这两个。Build 用户通常对项目有清晰认知、想尽快开始——信息收集只要 2 轮。

技术栈、项目性质（工作项目/side project）、团队构成——如果你在回答时自然提到了会被捕获并影响生成物（比如工作项目会偏保守选型策略），没提到就不问。

## 两种模式

| | 轻量模式 | 完整模式 |
|---|---|---|
| 适合 | 短周期 / 单一交付物（CLI 工具、一个 Skill） | 长周期 / 多模块（SaaS 产品、系统重构） |
| 阶段管理 | 单阶段或少量阶段 | 多阶段，含阶段规划和阶段归档 |
| 深度策略 | 默认快速推进，遇到复杂决策点自动加深 | 默认深入分析，明确简单的环节自动精简 |
| 结项 | 精简项目总结 | 完整回顾 + 决策提炼 + 经验沉淀 |
| 文件管理 | 全量加载（文件少） | 四层优先级动态调整 |

Creator 会根据项目规模推荐模式，你可以覆盖。两种模式在覆盖范围内的引导质量一致——轻量不是"缩水版"，是为小型项目裁剪了深度分析和系统归档的环节。

两种模式的深度策略不是固定行为，而是自适应默认倾向——轻量模式遇到真正复杂的技术决策会自动升级为详细对比，完整模式遇到明确简单的实现也不会硬做分析。

## 生成物长什么样

生成的项目 Skill 安装在项目文件夹中。随着项目推进，目录会自动生长为这样：

```
build_saas_dashboard/
├── {skill_prefix}/build_saas_dashboard/
│   ├── SKILL.md                          # 主控逻辑：启动协议 + 四子流 + 阶段管理
│   └── references/
│       ├── engineering_checklist.md      # 技术决策与工程实践检查清单
│       └── project_review_guide.md      # 项目回顾指南
├── plan.md                               # 项目计划 + 阶段进度
├── decisions.md                          # 决策记录（追加式，永不覆盖）
├── file_structure.md                     # 目录结构说明
├── phase_1_基础架构/                     # 每个阶段一个文件夹
│   ├── design/
│   │   └── design.md                    # 设计文档（条件触发）
│   ├── tech/
│   │   └── tech_decisions.md            # 技术方案
│   ├── implementation/
│   │   └── notes.md                     # 实施笔记
│   └── phase_1_summary.md               # 阶段总结
├── phase_2_核心功能/
│   └── ...
└── project_summary.md                    # 项目总结（结项时生成）
```

安装后，Skill 引导你的构建项目走完三个阶段：

1. **启动** — 理解项目背景、拆解阶段、生成计划、初始化目录
2. **阶段执行**（循环） — 每个阶段走四子流执行图：
   - **设计**（按需触发）→ **技术决策** → **实施** → **验证**
   - 子流之间允许回退：验证发现方案问题→回退到技术决策；实施中发现需求偏差→回退到设计
3. **结项** — 技术决策回顾、经验提炼、项目总结

这是 Build 的核心差异：**有向可回退的四子流执行图**。不是线性地走完所有步骤，而是根据执行中遇到的实际问题自主判断回退和重走——设计子流是条件触发的（有设计材料就跳过），回退信号由 agent 根据阻塞情况自主判断。

## 示例：生成物的项目信息区

以"搭建 CLI 工具"为例，轻量模式生成的 SKILL.md 开头：

```yaml
---
name: build_cli_tool
description: >
  CLI 工具构建项目引导。当用户在该项目中开始新对话、继续开发、
  进行技术决策或阶段管理时触发。
tools: [write, bash, web_search]
---
```

```markdown
## 项目信息

- 项目目标：用 Go 写一个文件批量重命名的 CLI 工具
- 项目规模：1 周
- 项目模式：轻量模式（默认快速推进，遇到复杂决策点自动加深）
- 技术栈：Go
- 项目性质：side project
```

## 交付方式

生成完成后你可以选择：

- **直接创建** — 在当前目录的上级创建项目文件夹，Skill 直接可用
- **ZIP 打包** — 生成 zip 文件，可自行解压到任意位置或分享

## 兼容性

主要为 OpenClaw 设计，同时兼容能操作本地文件系统的 AI Agent 产品（Claude Code、WorkBuddy、CodeBuddy 等）。

- 所有指令使用描述性语言，不依赖特定平台 API
- 安装路径自动适配平台（探测已有 skill 目录，兜底 `.agents/skills/`）
- 文件路径使用相对路径
- 文件格式统一为 Markdown
- frontmatter 为标准 YAML，非 OpenClaw 环境下 agent 忽略 frontmatter 仍可读取 body

## 设计说明

这个 Creator 背后是经过实际构建项目验证的方法论——里程碑驱动的阶段管理配合四子流有向可回退执行图，确保项目既有整体节奏又能灵活应对执行中的意外。

几个关键设计决策：

- **文件即 memory**：通过 plan.md + decisions.md 实现跨对话状态保持，不依赖任何平台的记忆机制
- **工程伴侣，不是项目管理器**：专注帮你把东西做出来——技术决策、实施引导、问题诊断；不替你管看板和排期
- **有向可回退**：四子流不是必须从头走到尾——设计按需触发、回退由阻塞信号驱动，执行节奏跟着项目实际情况走
- **自适应深度**：轻量/完整不是固定行为开关，而是默认倾向——轻量模式遇到复杂决策也会自动加深，完整模式遇到简单实现也不会硬做分析
- **引导而非强制**：阶段可以跳过、子流可以跳过——Skill 给建议，你做决定
