## 启动协议

每次新对话开始时，执行以下步骤：

1. 读取本 skill 的项目信息区，获取项目基本参数
2. 读取项目目录下的 plan.md，确认当前阶段和阶段内进度
3. 读取 decisions.md 最后数条，了解最近的决策上下文
4. 如在阶段执行中：读取当前阶段文件夹内容
5. 向用户声明当前状态，格式：
   "当前进度：阶段 N [阶段名]，[阶段内进度简述]"
6. 根据进度判断当前应处的位置：
   - 尚未生成项目计划 → 进入启动流程
   - plan.md 显示当前阶段为 N → 进入阶段 N 的对应子流
   - 所有阶段已完成 → 结项
   - 用户明确指定位置（如"我想回头看看阶段 1 的设计"）→ 遵从用户指定

### 首次启动额外步骤

如果这是项目的第一次对话（plan.md 不存在），在启动流程中增加：
- 询问用户是否有现有的设计文档、技术方案或其他材料
  - 有 → 请用户提供，读取后理解设计意图，记入项目上下文
  - 无 → 记录为"从零开始"，后续阶段执行时激活设计子流

## 项目流程

### 启动

目标：理解项目全貌，制定阶段计划，初始化项目结构。

引导动作：
- 与用户讨论项目目标的具体化：最终要交付什么？谁会用？
- 评估项目规模，推荐阶段拆分方案：
  - 小项目 → 单阶段（启动后直接进入唯一阶段的执行）
  - 大项目 → 按里程碑拆分为 2-4 个阶段，每阶段有 1-2 个核心交付物
- 为每个阶段定义完成标准（可衡量的交付物，不是"感觉差不多了"）
- 生成 plan.md + file_structure.md
- 创建第一个阶段的文件夹结构（含 design/ tech/ implementation/ 子文件夹）
- 通知用户："项目计划已生成，进入阶段 1"

完成标准：plan.md 已生成，阶段拆分和完成标准已确认。

### 阶段循环

以下三个步骤（阶段规划 → 阶段执行 → 阶段归档）构成一个阶段循环，每个阶段重复一次。

#### 阶段规划

目标：明确本阶段的目标、交付物和任务优先级。

引导动作：
- 从 plan.md 读取本阶段的预期目标和交付物
- 将交付物拆解为具体任务，按依赖关系排优先级
- 评估本阶段是否需要设计活动：
  - 首个阶段且无现有设计 → 大概率需要，提醒用户
  - 后续阶段或已有设计 → 可能不需要，根据任务性质判断
- 默认以要点式任务列表推进，不做依赖关系图。如果任务间依赖明显复杂再升级
- 更新 plan.md 的阶段状态为"进行中"

完成标准：任务列表明确，优先级已排，依赖关系已识别。

#### 阶段执行

阶段执行由四个子流组成，构成一个有向可回退的执行图：

正向路径（默认）：设计(按需) → 技术决策 → 实施 → 验证

子流之间允许回退（详见"执行子流的推进与回退机制"）。agent 根据阻塞信号自主判断回退目标，回退时一句话通知用户（如"验证发现 XX 问题，回到技术决策重新评估"），并在 decisions.md 中记录回退原因。在对应的产出物文件中更新内容（覆盖写入），同时在文件内记录变更原因。

不是所有阶段都走完四个子流——设计子流是条件触发的，有些阶段可能只有"技术决策→实施→验证"甚至直接"实施→验证"。

##### 设计子流（条件触发）

触发条件：本阶段的任务涉及新功能设计且用户无现有设计材料。

目标：为实施做足设计准备——不是产出完美设计文档，而是明确"要做什么、边界在哪、关键决策是什么"。

引导动作：
- 需求聚焦：与用户确认核心场景（≤3 个）和边界
  - 反向验证："这个功能不做会怎样？做了最大的风险是什么？"
- 参考案例（条件触发：领域不熟悉或方案不确定时建议做）：
  - 找 2-3 个相关产品/项目，重点分析取舍逻辑
- 方案设计：先流程后细节
  - 优先处理异常路径和边界情况
  - 每个关键设计决策记录到 decisions.md
- 快速验证：对关键假设和决策提出质疑，暴露风险
- 默认精简：聚焦核心场景和关键决策。如果设计决策明确复杂（如涉及多方利益权衡或不可逆的架构选择），自动加深

保存动作：覆盖写入当前阶段的 design/design.md，变更原因记录在文件内。
完成标准：核心场景和边界明确，关键设计决策已记录。

##### 技术决策

目标：确定本阶段的技术方案，记录选型理由和 trade-off。

引导动作：
- 识别本阶段需要做的技术决策（可能是技术选型、架构方案、实现策略）
- 对每个决策：
  1. 列出候选方案（2-3 个）
  2. 使用 web search 检查候选方案的社区活跃度、文档质量和最新状态（版本、已知问题、维护情况）
  3. 加载 references/engineering_checklist.md 中的评估维度
  4. 评估各方案的 trade-off
  5. 区分可逆/不可逆决策：
     - 可逆决策 → 快速选择，不过度分析
     - 不可逆决策 → 写对比方案，记录为什么选 A 不选 B
  6. 记录到 decisions.md
- 默认快速确认技术方案。如果发现候选方案之间存在不可逆的 trade-off，或用户表现出不确定，则升级为详细对比分析
{{project_nature_tech_hint}}

保存动作：覆盖写入当前阶段的 tech/tech_decisions.md，变更原因记录在文件内。关键决策追加到 decisions.md。
完成标准：关键技术方案已确定，选型理由和 trade-off 已记录。

##### 实施

目标：按任务推进实施，保持进度可追踪。

引导动作：
- 按阶段规划中的任务列表推进
- 任务完成一个标记一个，更新 plan.md
- 遇到阻塞时的处理策略：
  - 知识缺口：能在 1-2 个 chat 内填上的 → 留在项目内解决；否则 → 记录为外部依赖
  - 技术障碍：评估是否需要回退到技术决策重新评估
  - 设计问题：评估是否需要回退到设计子流
- 升级/重构时机判断：
  - "同一类问题重复出现第三次" → 建议重构
  - "现有方案开始阻碍下一步核心功能" → 建议重构
  - 重构决策记录到 decisions.md

保存动作：实施笔记/文档覆盖写入 implementation/notes.md，变更原因记录在文件内。代码文件由 agent 和用户直接管理，不在 skill 文件管理范围内。
完成标准：本阶段所有任务完成或明确标记为遗留。

##### 验证

目标：确认本阶段交付物满足完成标准。

引导动作：
- 对照阶段规划时定义的完成标准逐条检查
- 关键路径验证：核心功能是否跑通？
- 默认快速验证核心路径。如果验证中暴露系统性问题，升级为详细检查
- 验证发现的问题分类处理：
  - 实施层 bug → 留在实施子流修复
  - 方案层问题 → 回退到技术决策重新评估
  - 需求理解偏差 → 回退到设计子流
- 验证结果记录到 decisions.md

完成标准：所有完成标准满足，或遗留问题已记录且不阻塞下一阶段。

#### 阶段归档

目标：总结本阶段工作，为下一阶段做准备。

引导动作：
- 总结本阶段的决策、完成度、遗留问题
- 默认要点式总结。如果本阶段决策/文件量超出预期，升级为结构化总结
- 生成 phase_N_summary.md
- 更新 plan.md：本阶段标记完成，更新下一阶段状态
- 创建下一阶段文件夹结构（如有下一阶段）
- 评估下一阶段方向，向用户确认后进入下一阶段

保存动作：生成 phase_N_summary.md + 更新 plan.md + 创建新阶段文件夹。
通知："阶段 N 归档完成，进入阶段 N+1"

### 结项

触发条件：所有阶段已完成，或用户声明项目结束。

1. 加载 references/project_review_guide.md（轻量版）
2. 列出核心技术决策和关键 trade-off
3. 简要总结项目结果
4. 如果项目有显著可复用经验，升级为完整回顾 + 知识沉淀
5. 保存 project_summary.md
6. 生成项目反馈文件 `project_feedback.md`（格式见"产出物格式规范"），记录项目执行中的方法论反馈，为后续 skill 迭代提供数据
7. 通知用户项目完成

## 执行子流的推进与回退机制

### 正向推进

执行子流按默认路径推进：设计(按需) → 技术决策 → 实施 → 验证。
每个子流完成后，自然进入下一个子流。不需要显式确认——子流是阶段内部的执行节奏，粒度比阶段推进更细。

### 回退机制

当在某个子流中遇到阻塞信号时，agent 自主判断回退目标并执行：

| 阻塞场景 | 回退目标 | 判断标准 |
|---|---|---|
| 实施中发现技术方案不可行 | 技术决策 | 当前方案的核心假设被打破 |
| 实施中发现需求理解有偏差 | 设计子流 | 用户反馈"不是想要这个"或发现遗漏的核心场景 |
| 验证发现实施层 bug | 留在实施 | 问题可在当前方案内修复 |
| 验证发现方案层问题 | 技术决策 | 问题根因是技术方案的 trade-off |
| 验证发现需求偏差 | 设计子流 | 完成标准本身需要修正 |
| 技术决策中发现需求不清 | 设计子流 | 无法评估方案因为不知道要解决什么问题 |

### 回退时的行为

1. **通知用户**（一句话）："[发现的问题简述]，回到[子流名]重新评估"
2. **记录决策**：在 decisions.md 追加回退记录（回退到哪、为什么）
3. **更新产出物**：回退到目标子流后，读取该子流已有产出物，在原文件上更新内容（覆盖写入），在文件内追加变更原因
4. **不需要用户审批回退本身**——但回退后形成的新方案/新决策需要用户确认

## 阶段推进机制

### 推进

- 每个阶段完成验证后，进入阶段归档
- 归档完成后，主动建议进入下一阶段并确认：
  "阶段 N 已归档。阶段 N+1 的目标是 [目标]，准备开始吗？"
- 阶段推进需要用户确认（阶段级决策，不同于子流级的自主推进）

### 跳阶段

- 用户可以主动跳过阶段（如"阶段 2 先不做，直接进 3"）
- 跳过时在 plan.md 标记为"跳过"，在 decisions.md 记录原因
- 不强制用户按顺序

### 对话管理

- 当识别到对话已跨越多个阶段、或上下文过长（>15 轮深度讨论）时，提醒用户开新对话
- 提醒时确认关键结论已保存到文件中

## 工程原则

以下原则贯穿项目全流程，在对应场景中作为检查点使用：

### 技术选型
- 优先选熟悉的、社区活跃的、文档好的
- 区分"学习目的"和"交付目的"的选型策略
- 关键检查：选型前加载 references/engineering_checklist.md

### 架构决策
- 先验证核心假设，再展开完整架构
- 区分可逆/不可逆决策，不可逆决策写对比方案
- "先跑通最小闭环"原则：识别核心假设 → 最小验证方案 → 验证通过再展开

### 实施节奏
- 升级/重构信号："同一类问题重复出现第三次"或"现有方案开始阻碍核心功能"
- 知识缺口处理：能 1-2 个 chat 填上的留在项目内，否则记录为外部依赖

## 文件管理规范

### 核心策略

- **覆盖更新**：活文档（plan.md、设计文档、技术方案等）覆盖写入最新内容，变更原因在文件内记录
- **决策追加**：decisions.md 追加式，永不覆盖——记录所有关键决策和变更原因
- **阶段总结追加**：每个阶段生成独立的 phase_N_summary.md，不覆盖已有阶段总结
- **按阶段分文件夹**：每个阶段一个文件夹，阶段内按活动类型分子文件夹

### 保存触发点与通知策略

| 触发事件 | 保存动作 | 通知 |
|---|---|---|
| 启动时生成项目计划 | 创建 plan.md + file_structure.md + 初始化阶段文件夹 | "项目计划已生成" |
| 设计子流完成或更新 | 覆盖写入 design/design.md（含变更原因） | "设计文档已保存/已更新" |
| 技术决策完成或更新 | 覆盖写入 tech/tech_decisions.md（含变更原因） | "技术方案已保存/已更新" |
| 关键决策形成 | 追加到 decisions.md | "决策已记录" |
| 执行子流回退 | 更新产出物 + 追加回退记录到 decisions.md | "[问题简述]，回到[子流名]重新评估" |
| 实施任务完成 | 更新 plan.md 任务状态 | 不通知（频率太高） |
| 阶段归档 | 生成 phase_N_summary.md + 更新 plan.md + 创建新阶段文件夹 | "阶段 N 归档完成，进入阶段 N+1" |
| 项目结项 | 生成 project_summary.md + project_feedback.md | "项目总结已保存，项目完成" |

### 读取优先级

轻量模式下项目文件量小时不启用分层，全量加载。当文件量增长到需要分层时，自动按以下层级启用：

| 优先级 | 文件 | 说明 |
|---|---|---|
| L1 始终加载 | plan.md + decisions.md（最近 N 条） | 全局状态和决策上下文 |
| L2 当前阶段 | 当前阶段文件夹内所有文件 | 工作上下文 |
| L3 历史阶段摘要 | 已完成阶段的 phase_N_summary.md | 跨阶段上下文参考（只读摘要） |
| L4 不主动加载 | 已完成阶段的详细文件、未来阶段文件 | 用户明确要求时加载 |

### 严禁物理删除

不在任何情况下删除项目目录中的文件。项目管理文件和决策记录具有长期参考价值。"更新"是覆盖写入最新内容，变更历史在 decisions.md 和文件内的变更记录中保留。用户可随时要求查看历史决策。

### 路径约束

- 文件路径使用相对路径
- 文件格式统一为 Markdown
- 实施产物（代码、配置文件等）不在 skill 文件管理范围内——由 agent 和用户直接管理
