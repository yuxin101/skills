# 技术决策与工程实践检查清单

> 在技术决策子流中加载此文件，逐项作为评估维度使用。
> 不是每次都走完所有项——按当前决策的类型和重要性选择适用项。

---

## 一、技术选型检查点

做技术选型时（框架、库、服务、工具等），逐条评估：

### 核心评估维度

- [ ] **团队熟悉度**：团队（或个人）对该技术的掌握程度如何？
  - 熟悉 → 低风险，可快速推进
  - 不熟悉但想学 → 区分"学习目的"和"交付目的"（见下方）
  - 完全陌生 → 高风险，除非有强理由否则避免

- [ ] **社区活跃度**：该技术的社区是否活跃？
  - GitHub stars/commits 趋势、Stack Overflow 活跃度、Discord/论坛活跃度
  - 关注"最近 6 个月"的活跃度而非累计数字

- [ ] **文档质量**：官方文档是否足够支撑开发？
  - 有清晰的 Getting Started、API Reference、常见问题
  - 有可运行的示例代码

- [ ] **维护成本**：长期维护该技术的成本如何？
  - 版本更新频率和破坏性变更历史
  - 依赖链的复杂度
  - 是否有商业支持或长期维护承诺

### 学习目的 vs 交付目的

- [ ] **明确本次选型的目的**：
  - 学习目的（个人项目、探索性项目）→ 允许尝试新技术，接受学习曲线带来的时间成本
  - 交付目的（有明确交付时间、有用户在等）→ 优先选熟悉的、稳定的、文档好的
  - 混合目的 → 核心路径用熟悉技术保障交付，非关键模块可尝试新技术

## 二、架构决策检查点

做架构级决策时（系统结构、数据模型、核心流程等），逐条评估：

### 可逆性判断

- [ ] **判断决策的可逆性**：
  - 可逆决策（如 UI 库选择、CSS 方案、API 响应格式细节）→ 快速选择，不过度分析。错了改就是
  - 不可逆或高成本可逆决策（如数据库选择、核心数据模型、公开 API 契约）→ 写对比方案，记录为什么选 A 不选 B

### 最小闭环验证

- [ ] **识别核心假设**：这个架构方案的成立依赖哪些假设？
- [ ] **设计最小验证方案**：能否用最小代价验证核心假设？
  - 一个 spike/prototype 能验证吗？
  - 能否在不搭完整架构的情况下验证关键路径？
- [ ] **验证通过再展开**：核心假设未验证前，不要铺开完整实现

### 复杂度控制

- [ ] **当前需求真的需要这个复杂度吗？**
  - "将来可能需要"不是现在引入复杂度的理由
  - 如果不确定，选简单的——等"同一问题重复出现第三次"再升级

---

## 三、实施质量检查点

实施过程中的工程实践检查：

### 阻塞处理

- [ ] **知识缺口评估**：
  - 能在 1-2 个对话内填上的 → 留在项目内解决
  - 需要系统学习的 → 记录为外部依赖，不阻塞当前任务

### 重构时机

- [ ] **重构信号检测**：
  - 同一类问题重复出现第三次 → 信号明确，建议重构
  - 现有方案开始阻碍下一步核心功能 → 信号明确，建议重构
  - 仅仅是"代码不够优雅" → 不是重构的充分理由

### 交付前检查

- [ ] **核心路径跑通**：最关键的用户路径是否端到端可用？
- [ ] **异常路径覆盖**：已知的异常情况是否有处理？
- [ ] **关键决策有记录**：decisions.md 中是否记录了本阶段的关键技术决策？
