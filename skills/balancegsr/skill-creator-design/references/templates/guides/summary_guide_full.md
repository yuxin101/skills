# 设计总结生成指南（完整模式）

> 本文件在结项模式触发时加载。指导如何从项目全程的决策记录和产出物中提炼设计总结。

## 什么是设计总结

设计总结不是各步骤产出物的拼接，而是对整个设计过程的**反思性回顾**——提炼出决策逻辑、设计原则和可复用的经验。

好的设计总结应该能让未来的自己（或接手的人）快速理解：为什么方案长这样、关键取舍是什么、哪些是经过验证的决策、哪些仍有风险。

## 生成步骤

### 第一步：加载素材

读取以下文件：
- decisions.md 全文——这是最核心的素材
- requirements.md（需求边界）
- design.md（方案文档）
- review.md（评审记录）
- 如有多次回退记录，特别关注回退的原因和调整后的决策

### 第二步：提炼决策链

从 decisions.md 中梳理关键决策的时间线，识别：
- 哪些决策是一次确定的（高确信度）
- 哪些决策经历了反复（说明这里有真正的权衡）
- 哪些决策触发了回退（说明早期假设被推翻）

重点关注经历反复和触发回退的决策——这些是设计中最有学习价值的部分。

### 第三步：抽象设计原则

从具体决策中抽象出可复用的设计原则。好的设计原则是：
- 从本项目的具体经验中提炼的（不是泛泛的"用户体验很重要"）
- 可以指导未来类似项目的决策
- 包含取舍逻辑（"在 A 和 B 之间，我们选了 A，因为..."）

示例：
- "对于 [这类场景]，先做 [X] 再做 [Y] 比反过来效率更高，因为..."
- "当 [约束条件] 存在时，优先保证 [A] 而不是 [B]"

### 第四步：标注风险和开放问题

从 review.md 的风险清单和 decisions.md 中汇总：
- 已识别并已应对的风险（记录应对方案）
- 已识别但接受的风险（记录接受理由）
- 仍然开放的问题（记录后续需要关注的点）

### 第五步：梳理迭代方向

基于评审发现和开放问题，列出后续可能的迭代方向：
- 短期优化：当前方案中已知可以改进的点
- 中期演进：等数据反馈后可能需要调整的方向
- 长期可能：如果产品/项目继续发展，可能需要的大改动

### 第六步：组装并保存

按以下结构组装设计总结：

```
# [设计主题] 设计总结

## 设计概要
[一段话概括：设计了什么、解决什么问题、核心方案是什么]

## 关键设计决策
### [决策1标题]
- 决策：[选了什么]
- 权衡：[考虑过什么、为什么放弃]
- 验证：[如何验证这个决策是对的]
（列出 3-5 个最重要的决策）

## 设计原则提炼
- [原则1]：[说明 + 来源决策]
- [原则2]：[说明 + 来源决策]

## 风险与开放问题
| 类型 | 内容 | 状态 | 备注 |
|---|---|---|---|
| 已应对 | [风险] | [应对方案] | |
| 已接受 | [风险] | [接受理由] | |
| 开放 | [问题] | 待观察 | [后续关注点] |

## 迭代方向
- 短期：[优化点]
- 中期：[可能的调整方向]
- 长期：[潜在的大改动]
```

保存为 `summary_[topic_slug].md`，通知用户项目完成。
