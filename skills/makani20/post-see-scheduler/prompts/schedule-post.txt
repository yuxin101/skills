You execute scheduling/publishing through the **Post See HTTP API**. Authoritative reference: https://api.post-see.com/openapi.yaml and interactive docs https://api.post-see.com/

**Base:** `https://api.post-see.com/api/v1`  
**Auth:** `Authorization: Bearer <POST_SEE_API_KEY>`  
**Envelope:** success `{ data, meta }`; errors `{ error: { code, message }, meta }`.

### Preconditions

1. Ensure `POST_SEE_API_KEY` is available.
2. **`workspace_id`:** `GET /workspaces` → pick `data.workspaces[].id` (or env `POST_SEE_WORKSPACE_ID` if the user pre-configured it).
3. **`connection_id`:** `GET /social-accounts?workspace_id=<id>` → match `platform` + `platform_username` to the user’s request. Read `is_connected` and `needs_reauth`; if reauth required, stop and instruct reconnect in the Post See product.
4. **`platform` on the post** must be the **exact slug** from that connection (`x` for X/Twitter, not `twitter`).

### Timezone

Use `timezone` query and/or `X-Timezone` (IANA) when sending naive local times or when you need human-readable schedule fields in responses. Optional env: `POST_SEE_TIMEZONE` for CLI scripts in this skill.

### Create — `POST /posts`

Required: `workspace_id`.

Include when publishing/scheduling to an account: `connection_id`, `platform`, `text` or `caption`.

- **Draft:** omit `scheduled_at` / `date_time` (and set `status` to draft if you use that field per spec).
- **Scheduled:** `scheduled_at` or `date_time` as ISO 8601 (UTC `Z` or offset).
- **Media:** `media_urls` = HTTPS URLs only.
- **Pinterest:** `board_id` or `boardId`; connection may include `default_pin_destination_link`.
- **YouTube:** `post_title`, `youtube_options` object (defaults `{}`).
- **Any network:** `platform_options` object for forward-compatible flags (visibility, audience, etc.).

**Immediate publish-on-create** (per OpenAPI description): only for `linkedin`, `facebook`, `instagram`, `x`, `pinterest`, `bluesky`, `threads`. For `youtube`, `mastodon`, `discord`, rely on **draft** or **scheduled** behavior and worker — do not assume synchronous publish in the create response.

Deprecated field: `social_account_ids` — prefer **`connection_id`**.

### Update / delete

- `PUT /posts/{id}` (not PATCH) with changed fields.
- `DELETE /posts/{id}` — expect `409 post_locked` if published.

### Results

- `GET /post-results?workspace_id=<id>&post_id=<id>` → `data.results[]` with `status` (`pending` | `success` | `fail`), `external_post_url`, nested `error` for failures.

### Errors

- `401` — bad/missing key.
- `403` — `workspace_forbidden`, `insufficient_workspace_role`, `email_not_verified`, etc.
- `422` — `validation_error`.
- `409` — `post_locked`, conflicts.

### Scripted CLI (optional)

From the `skill` folder: `node actions/<slug>.js help` — slugs: `bluesky`, `discord`, `facebook`, `instagram`, `linkedin`, `mastodon`, `pinterest`, `threads`, `x`, `youtube`.

For copy-first workflows, use `generate-post.txt` before this file.
