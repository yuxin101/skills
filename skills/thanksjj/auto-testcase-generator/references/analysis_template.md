# 需求分析简报模板

生成测试用例前，先完成需求分析简报，作为测试设计的基础。简报采用 Markdown 格式输出。

## 简报结构

```
# 需求分析与测试理解简报：{被测功能名称}

## 来源文档
- 列出所有输入文档及版本信息

## 0a. 生效的测试规则
（记录使用的测试规范来源和优先级）

## 0b. 被测主体范围界定（SUT Scope Definition）
（明确测试边界）

## 1. 文档结构覆盖追踪
（逐章节审阅追踪表）

## 2. 需求缺口清单
（未定义行为、隐含规则、边界缺失等）

## 3. 关键规则
（提取的业务规则和约束条件）

## 4. 测试标准
（操作场景与验收标准矩阵）

## 5. 测试理解简报
（系统流程、核心规则、关键风险、测试重点分配）

## 6. 背景知识
（上下游依赖、架构上下文）

## 7. 分析结论
（总体评估、跟进建议、测试策略）
```

## 各章节详细说明

### 0b. 被测主体范围界定

> **注意**：此章节的内容来自"阶段零：测试范围确认"的输出结果。在阶段零中已经与用户完成了范围确认，此处直接引用已确认的结论。

将需求内容划分为三个区域：

| 区域 | 含义 | 测试处理 |
|------|------|---------|
| **Zone 1 — 核心区（直接测试范围）** | 被测功能直接实现的行为 | 完整设计测试用例：提取规则、缺口、标准 |
| **Zone 2 — 交互区（上下游交互）** | 与被测功能有交互的外部系统/模块 | 精简为背景知识，仅在接口边界处设计少量用例 |
| **Zone 3 — 外部区（排除范围）** | 别的团队负责的东西，与本次测试无关 | 不生成测试用例，不分析 |

输出格式：

```
### 被测主体（SUT）
{被测系统/功能名称}

### 已确认的范围划分

| # | 功能/模块 | 区域 | 划分理由 |
|---|----------|------|---------|
| 1 | ... | Zone 1（核心区） | ... |
| 2 | ... | Zone 2（交互区） | ... |
| 3 | ... | Zone 3（外部区） | ... |

### 范围确认状态
- ✅ 已在阶段零与用户确认
- Zone 1 共 X 个模块，后续阶段仅针对这些模块进行深度分析
```

### 1. 文档结构覆盖追踪

以表格形式逐章节追踪：

| 列 | 说明 |
|----|------|
| # | 序号 |
| 一级章节 | 文档一级标题 |
| 二级章节 | 文档二级标题 |
| 覆盖状态 | ✅ 已覆盖 / ⚠️ 部分覆盖 / ❌ 未覆盖 |
| Zone | Zone 1/2/3 |
| 提取的规则 | 对应规则编号（如 BR-001） |
| 备注 | 补充说明 |

### 2. 需求缺口清单

识别以下类型的需求缺口：

- **未定义错误处理**：异常场景缺少处理说明
- **隐含规则**：行为逻辑需推断而非明确定义
- **缺失边界定义**：数值范围、限制条件未明确
- **并发/竞态**：多操作并发场景未定义
- **状态不一致**：文档内自相矛盾的描述

每个缺口标注严重程度（高/中/低）和改进建议。

### 3. 关键规则

按以下类型分类提取：

| 规则类型 | 编号前缀 | 说明 |
|---------|---------|------|
| 数据规则 | DR-xxx | 数据格式、内容、取值范围 |
| 行为规则 | BR-xxx | 系统行为和业务逻辑 |
| 流程规则 | PR-xxx | 操作流程和步骤顺序 |
| 约束规则 | CR-xxx | UI布局、尺寸、技术限制 |
| 安全规则 | SR-xxx | 权限控制、数据安全、安全约束 |

### 4. 测试标准

以矩阵形式定义验收标准：

| 列 | 说明 |
|----|------|
| 操作/场景 | 待测试的操作或场景 |
| 正确性标准 | 功能正确的判定条件 |
| 性能标准 | 响应时间等性能要求 |
| 可靠性标准 | 稳定性和容错要求 |
| 可观测性标准 | 日志、监控等可观测性要求 |

### 5. 测试理解简报

包含四个子章节：

1. **系统流程**：核心操作流程的文字描述
2. **核心规则 Top 5**：最关键的 5 条测试规则
3. **关键风险**：风险领域、描述、潜在影响、等级
4. **测试重点分配**：按优先级（P0/P1/P2）分配测试重点和覆盖深度

### 7. 分析结论

包含：
- **需求文档完整度评估**
- **可测试性评估**
- **建议跟进项**（按优先级排列，标注需产品/开发确认）
- **测试策略**：采用的策略（风险驱动/功能驱动等）

## 补充知识集成

当用户提供额外信息（如口头确认、需求澄清）时，在简报末尾追加"补充知识集成"章节：

- 记录知识来源和日期
- 新增补充规则（按对应类型编号，如 DR-xxx/BR-xxx/PR-xxx/CR-xxx/SR-xxx）
- 更新风险评估
- 调整测试重点

### 报告生成规则

1. **矩阵表必须完整**：覆盖矩阵中列出所有规则/标准，不得省略已覆盖的条目
2. **问题清单务实**：只列出真正需要处理的问题，不人为制造问题
3. **修复记录可追溯**：每次修复都记录影响的用例编号，方便回查
4. **数据一致性**：报告中的用例编号必须与 Excel 文件中的实际编号一致
