# 动态采样实现方案（A）

## 1. 目标
本文件定义 `go-stargazing` 的动态采样、全国粗筛、第二轮精查、区域聚合与调参规则。

它解决的问题是：
- 默认全国查询时，不能依赖固定候选区域池
- 不能一次性全量高密度精查
- 需要在"覆盖率、耗时、可解释性"之间取得平衡

---

## 2. 总体流程
全国 / 大范围查询采用两阶段：

### 阶段 1：云量粗筛
- 根据目标范围面积自动生成采样点
- **第一轮只查云量 `cloud_cover`**
- 用云量门槛快速过滤明显不适合拍星的区域

### 阶段 2：完整精查
- 仅对粗筛通过区域查完整指标
- 用轻量评分体系做区域排序
- 生成最终区域结论

---

## 3. 采样点生成

## 3.1 基本原则
- 不依赖固定候选区域白名单
- 按行政区边界或目标 polygon 动态生成
- 面积越大，点数越多
- 第一轮点数可比第二轮更多

## 3.2 生成方式
推荐两种实现路径：

### 方案 A：polygon 内网格采样
1. 获取目标范围 polygon
2. 生成规则网格点
3. 过滤掉 polygon 外的点
4. 剩余点作为候选采样点

### 方案 B：边界框生成 + polygon 过滤
1. 先在 bounding box 生成点
2. 再做 point-in-polygon 过滤
3. 保留有效点

当前原型已经支持这一路径：
- 可通过 `--polygons-json` 或 `--polygons-file` 传入 GeoJSON/JSON
- 对匹配到的省份 / 区域执行精确 point-in-polygon 过滤
- 若未显式传入 polygon，原型会默认尝试加载 bundled 数据：`data/china-provinces-full.geojson`

当前打包版仅依赖 skill 内自带的省级 / prefecture 级边界数据，不在运行时动态下载额外 GeoJSON 资源。

---

## 3.5 多日晚间结果如何汇总成路线

当用户问的是假期 / 连续多天拍星时，不应直接把“每晚独立最优区域”串起来当作最终方案。

优先级应改为：
1. 先找 **连续可执行的主路线 / 主战区**
2. 再看这条路线在每一晚是否都还有可接受窗口
3. 最后才把“逐晚最优榜单”作为补充说明

轻量约束规则：
- 默认优先同一区域 / 同一走廊 / 相邻区域的连续方案
- 对明显跨大区的大跳跃方案做降权处理
- 若用户未明确表示愿意长距离转场，不要把跨省远跳组合直接当主方案输出
- 若某个远距离单晚冠军确实明显更强，应把它表述成“另一套独立方案”，而不是和本地连续路线硬拼

### 3.5.1 Phase 3.1 设计草案：路线组合评分

建议先做一个不依赖外部地图 API 的轻量路线层：

1. 每晚先产出 Top N 区域（建议 N=3）
2. 对多日晚的 Top N 做组合枚举
3. 对每条组合路线计算：

```text
trip_score = weather_score_total - travel_penalty_total + continuity_bonus
```

其中：
- `weather_score_total`：逐晚区域得分求和
- `travel_penalty_total`：相邻两晚之间的转场惩罚和
- `continuity_bonus`：同区域 / 同走廊 / 相邻区域的连续性奖励

### 3.5.2 转场惩罚的轻量实现

Phase 3.1 不需要真实导航，只需要基于区域代表点（如 cluster 中心点 / best point）计算相邻两晚之间的大致球面距离。

建议分档：
- `< 150km`：几乎不罚
- `150-350km`：轻罚
- `350-700km`：中罚
- `> 700km`：重罚
- `> 1200km`：默认不进入主路线，只能进入“独立远线方案”候选

### 3.5.3 默认假设

如果用户没有额外提供信息，默认假设：
- 未提供出发地
- 未说明交通方式
- 未明确接受长距离转场

在这个默认假设下，路线层应优先保证：
- 连续
- 可执行
- 不跨大区乱跳

### 3.5.4 预期输出层（设计）

路线层建议新增输出：
- `trip_plans`
- `trip_score`
- `weather_score_total`
- `travel_penalty_total`
- `continuity_level`
- `why`

面向用户时，应优先组织成：
- 主推荐路线
- 备选路线
- 独立远线方案
- 逐晚最优补充说明

### 3.5.5 默认交互建议（性能优先）

多日问题不应默认立刻进入重型路线规划。建议采用：

1. **先给逐晚最佳区域**（快速层）
2. 明确告知用户：如需连续路线规划，可继续启用 `trip mode`
3. 在启用前，先用自然语言说明大致快慢，让用户自行选择档位（Quick / Standard / Deep）

建议对用户只保留笼统说明：
- Quick：更快，先看路线草案
- Standard：更稳，适合认真规划
- Deep：更细，但会更慢

这样可以在保证响应速度的前提下，把多日路线规划作为第二层能力，而不是默认阻塞主回答。

## 4. 点数预算

## 4.1 不写死神圣数字
采样点数应视为可调参数，不是永远固定的常量。

应该遵循：
- 先设初值
- 用实测耗时与结果覆盖率校准
- 再动态调整

## 4.2 面积分档建议
### 单省模式
- 小省：低密度采样
- 中省：中密度采样
- 大省：高密度采样
- 超大省 / 自治区：更高密度采样

### 全国模式
- 第一轮：全国粗筛点数较多
- 第二轮：只对通过区域做更少但更有价值的精查

## 4.3 调优方向
- 如果粗筛太慢 → 降低第一轮点数
- 如果粗筛漏掉明显好区域 → 提高第一轮点数
- 如果第二轮太慢 → 缩减每个入围区域的精查点数

---

## 5. 第一轮云量粗筛

## 5.1 查询字段
第一轮只查：
- `cloud_cover`

不要在第一轮就查：
- 湿度
- 可见度
- 风速
- 月相外的额外复杂项

## 5.2 粗筛阈值
### 银河 / 星空 / 观星
- `<= 35%`：通过
- `35% ~ 50%`：边缘保留
- `> 50%`：淘汰

### 月景
- `<= 50%`：通过
- `50% ~ 65%`：边缘保留
- `> 65%`：淘汰

## 5.3 保底机制
如果粗筛后可用区域太少：
- 自动保留云量最低的一小批点继续进入第二轮
- 结果里明确提示：全国 / 全省整体条件一般，仅给出相对最优区域
- 单天模式下，若实际进入第二轮前的 survivor 数量 <= 10，则直接对这批 survivor 全量细筛，不再额外压 stage2 预算
- 多天 / trip mode 下，为了保留路线连续性候选，direct refine 阈值应自动放宽到至少 20；也就是说 survivor 数量 <= 20 时，直接对这批 survivor 全量细筛

---

## 6. 第二轮完整精查
第二轮只对粗筛通过点或其周边区域进行完整查询。

## 6.1 查询字段
第二轮查询：
- `cloud_cover`
- `humidity`
- `visibility`
- `wind_speed`
- `moon`（按时间计算或辅助获取）

## 6.2 第二轮预算
第二轮必须严格控量：
- 只查粗筛后相对最优的区域
- 不要把所有边缘保留点全部精查到底
- 通过 `max_stage2_points` 之类的预算参数，硬性限制第二轮参与评分的点数

## 6.3 并发策略
真实天气查询应使用**有上限的并发**，不要串行逐点慢查，也不要无限并发。

建议：
- 使用固定上限 worker 数，例如 `max_workers`
- 点数较多时并发抓取 Open-Meteo 数据
- 仍保留总体预算限制，避免第一轮或第二轮无限膨胀

---

## 7. 跨省近邻去重与合并规则
这是当前方案的关键新增规则。

## 7.1 为什么要去重
全国模式下，如果两个高分采样点：
- 分属不同省份
- 但空间距离非常近

那么它们很可能描述的是**同一片真实区域天气**，只是刚好落在行政边界两侧。

如果不去重，容易出现：
- 青海西北部 与 甘肃西北部 同时上榜，但其实是同一个区域团块
- 新疆边界附近两个点重复占用 Top 名额

## 7.2 基本规则
当两个高分点满足以下条件时，应视为同一区域候选：
1. 空间距离低于阈值
2. 云量与第二轮综合评分差异不大
3. 聚类后属于同一方向性区域描述

则：
- **按一个区域来算**
- 不要分成两个独立推荐项

## 7.3 距离阈值
第一版建议：
- 用一个可调的空间去重阈值
- 后续根据测试再调

原则：
- 太小 → 重叠区域去不掉
- 太大 → 不同区域被误合并

## 7.4 合并后的归属表达
若跨省相邻高分点被合并：
- 优先用更中性的区域表达
- 避免在同一结果中用两个几乎重叠的省份名称分别占位

例如：
- 可表达为"青海-甘肃交界西北部高地带"
- 或只保留综合分更高、代表性更强的一侧作为主标签

## 7.5 保留规则
当两个近邻重叠点合并时：
- 默认保留综合分更高者为主
- 如果分差极小，则保留空间中心更代表整体区域的点
- 被合并的点不再单独占用 Top 名额

---

## 8. 区域聚合规则
不要直接返回采样点列表。

应当：
1. 对高分点做空间聚类
2. 对近邻重叠点先做去重 / 合并
3. 再输出区域级描述
4. 优先输出 cluster 级结果，而不是逐点标签

当前原型已支持：
- 通过 `cluster_km` 控制聚类半径
- 单省 cluster 输出"某省 + 方位"
- 跨省 cluster 输出"省A-省B交界 + 方位"一类更自然的标签
- cluster 结果输出成员点、cluster 大小、平均云量与平均分，便于后续继续收敛优化
- 海拔辅助标注：基于 Open-Meteo elevation 数据 + 固定高度分档（低山/中山/中高海拔/高海拔/超高海拔）
- 地理特征优先命名：内置约 40 个知名地理参考点（城市、山脉、湖泊、沙漠），cluster 中心落在参考点附近时优先使用特征名而非纯方向描述
- 标签格式示例："甘肃敦煌周边高海拔区域"、"青海中高海拔区域·西部"、"甘肃安西盆地高海拔区域"

## 8.1 输出风格
优先输出：
- 省级 + 方位
- 或跨省中性区域描述

例如：
- 青海西北部
- 新疆东部
- 西藏西部
- 青海-甘肃交界西北高地带

## 8.2 输出数量
- 默认 Top 3
- 最多 Top 5

---

## 9. 超时与降级
如果查询耗时超出可接受范围：
- 优先保留第一轮粗筛结果
- 缩减第二轮点数
- 给用户返回相对粗粒度结论

不要为了等所有精查完成而让交互卡死。

---

## 10. 原型脚本
当前已提供一个无外部依赖的原型脚本：
- `scripts/dynamic_sampling_prototype.py`

当前能力：
- 生成 bounding box 内采样点
- 支持第一轮云量粗筛（mock 或真实 Open-Meteo 云量）
- 支持第二轮轻量评分（humidity / visibility / wind + moon factor）
- 支持基础查询预算（第二轮点数上限）
- 支持有上限的并发抓取
- 做跨省近邻去重
- 生成方向性区域标签

当前限制：
- 当前已支持 GeoJSON polygon 过滤，但还没有接入正式全国/省级行政边界数据源
- 还没有做缓存优化
- 当前月相仍是原型级近似处理，不是正式天文引擎

## 11. 调参原则
后续测试时重点观察：
1. 全国模式总耗时
2. 省级模式总耗时
3. 粗筛是否漏掉明显好区域
4. 跨省边界区域是否重复上榜
5. 聚合后的区域描述是否自然

调参方向：
- 太慢 → 降低第一轮或第二轮点数
- 漏得太多 → 提高第一轮点数或放宽粗筛阈值
- 重叠太多 → 增大跨省近邻合并阈值
- 误合并太多 → 减小合并阈值

本文件是当前版本的实现约束；具体数值以后以实测为准。
