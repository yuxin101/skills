# 更具体地名输出层方案

## 目标
在不立刻全国接入县级/区级 polygon 的前提下，让 `go-stargazing` 的 refinement 结果从“工程格子名”升级成更可执行的人类地名。

例如：
- `阿拉善左旗东南缘西南格`
- 升级为：`阿拉善左旗南部`
- 或：`阿拉善左旗东南部，靠腾格里南缘`

---

## 为什么先做这个
直接全国接入县级/区级 polygon 的问题：
1. 数据量大，边界数量暴增
2. 采样/过滤成本明显上升
3. 在中远期预报下，过度精确会制造“精确地错”的假象

因此先做：
- refinement 继续使用现有网格/区域细分
- 输出层增加“更具体地名反查”

---

## 第一阶段：轻量地名反查（优先）
输入：
- refinement 最优子区中心点（lat/lng）
- 当前所属省/大区
- 当前区域标签

输出：
- 更具体的人类地名：旗 / 县 / 区 / 附近地理单元

### 实现思路
1. 先保留当前 refinement 网格逻辑不变
2. 对最佳子区中心点做地名反查
3. 输出层拼成：
   - 行政区名（如阿拉善左旗）
   - 方位（南部 / 东南部）
   - 可选附近地理特征（腾格里南缘 / 靠乌海方向）

### 最低可用版本（MVP）
- 只做：
  - 省 / 地级市 / 旗县级名称反查
  - 方位词拼接
- 暂不做：
  - 复杂 POI 命名
  - 具体机位级别推荐

---

## 第二阶段：县级/区级 polygon 下钻（后续）
当 refinement 流程稳定后，再做：
1. 接入全国县级 / 区级行政边界数据
2. 候选区域进入 Top N 后，再做县级精确过滤
3. 在县级边界内继续 refinement

### 使用边界
- 全国粗筛阶段：不使用县级 polygon
- 候选区域精修阶段：才加载县级 polygon

这样可以避免：
- 全国级计算爆炸
- token / 请求 / latency 失控

---

## 输出规范（未来）
全国报告后，如果进入 refinement：
- 不再输出“西南格 / 东北格”这类工程名
- 优先输出：
  - `阿拉善左旗南部`
  - `阿拉善左旗东南部`
  - `乌海西北过渡带`
  - `德令哈周边`

如果地名反查不够稳定：
- 允许输出：
  - `阿拉善左旗南部（子区）`
  - `柴达木盆地东缘（子区）`
作为降级命名

---

## 当前建议顺序
1. 先做更具体地名反查输出层
2. 让 refinement 结果不再暴露网格名
3. 后续再接县级/区级 polygon
