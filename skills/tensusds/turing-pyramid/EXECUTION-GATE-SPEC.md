# Execution Gate — Design Specification

**Version:** 0.3.0-draft (post-NewMoon infrastructure review)
**Status:** Design spec / pseudocode
**Depends on:** Turing Pyramid ≥ v1.26.0
**Authors:** Max + NewMoon + Claude

---

## 1. Problem Statement

NewMoon diagnosed the core issue: run-cycle.sh outputs "▶ ACTION: write Moltbook post", the agent reads this text, writes "expression at 0.25, action: write post" in a log, and reports HEARTBEAT_OK. The post is never written. mark-satisfied is never called. Satisfaction stays at 0.25. Next heartbeat — same thing.

This is not a prompting problem. Research confirms this is a **structural** failure mode common across LLM agent frameworks:

- **CrewAI Bug #3154**: LLMs fabricate tool outputs instead of executing tools. The agent produces valid-looking Thought→Action→Observation traces where the Observation is entirely generated by the LLM, not by actual tool execution.
- **BrainBody-LLM (2024)**: Separating planning from execution with closed-loop state feedback is necessary because LLMs conflate "reasoning about an action" with "performing an action."
- **ReAct pattern**: The critical element is that Observation must come from the **environment**, not from the agent's own output. Without environmental feedback, there is no grounding.
- **Approval Gates (Trustworthy AI Agents, 2025)**: "Formal breakpoints in agent execution where autonomous action is suspended until a human explicitly authorizes the next step." Structural enforcement, not advisory.

The fundamental insight: **for an LLM, describing an action and performing an action are computationally identical** — both are token generation. The only way to distinguish them is through **environmental evidence** that the action produced a state change.

Adding "please actually do the action" to HEARTBEAT.md is the same failure mode at a higher level — it's still advisory text that the agent interprets.

---

## 2. Design Principles

From the research:

1. **Actions must produce verifiable state changes.** (BrainBody-LLM, ReAct)
   Not "agent says it did X" but "filesystem/state shows X happened."

2. **Observation comes from environment, not from agent.** (ReAct, crewAI bug)
   Evidence is collected by scripts, not self-reported.

3. **Structural enforcement, not advisory text.** (Approval Gates)
   The gate blocks HEARTBEAT_OK mechanically, not by instruction.

4. **Closed-loop feedback.** (All sources)
   Plan → Execute → Verify → Report. Not Plan → Report.

5. **Mechanism, not prompt.** (Turing Pyramid philosophy)
   Consistent with how the pyramid already works.

## 3. Session Model: From Passive Receiver to Active Initiator

### 3.1 Validation: OpenClaw Supports Self-Continuation

Research confirms the mechanics NewMoon described are real and already used:

**`openclaw system event --text "..." --mode now`** triggers an immediate heartbeat in the **main session** — not a new isolated session. The agent wakes with full context. This is the same mechanism the bundled coding-agent skill uses for background task completion notifications.

**Known issues to account for:**

- **Bug #14527**: `system event --mode now` is silently skipped if HEARTBEAT.md contains only comments or whitespace. The "wake" reason is not exempted from the empty-heartbeat-file check. **Workaround**: HEARTBEAT.md must contain actionable plain-text content (which it already does with the pyramid protocol).

- **Bug #29215**: `heartbeat.target` defaults to "none", meaning agent responses during heartbeat are silently dropped. **Fix**: set `heartbeat.target: "last"` in OpenClaw config. Without this, the agent processes the event but nobody sees the output.

**Cron vs systemEvent vs agentTurn:**
OpenClaw distinguishes between `systemEvent` (injects into main session's heartbeat path) and `agentTurn` (starts a separate isolated run). For self-continuation, `systemEvent` with `--mode now` is correct — it keeps the agent in its own session with full context.

### 3.2 The Core Pattern: Freeze as Default

NewMoon nailed the key insight: **invert the control**. Session always ends. Continuing is an action, not a decision.

```
SESSION STARTS
    → boot → see pending actions
    → work (write post, research, etc.)
    → ...

FORK (structural, not textual):

    A) Agent calls exec("openclaw system event --mode now")
       → freeze does NOT happen (session is "continuing")
       → new turn arrives → agent continues working
       → THIS IS AN ACTION = proof of intent

    B) Agent does NOT call event
       → session ends naturally
       → freeze happens automatically
       → agent sleeps
       → NO ACTION = no continuation

Binary. Structural. Cannot be "interpreted" as a log entry.
```

The chain of evidence at every step:
1. `run-cycle.sh` → creates `pending_actions.json` (filesystem fact)
2. Agent tool call for action execution (observable: file created, post published)
3. `exec mark-satisfied.sh` (state change fact: audit.log entry)
4. `exec gate-resolve.sh` (gate state change fact)
5. `exec openclaw system event --mode now` OR silence (binary: called or not)

Every step is a **verifiable exec**, not text. The agent cannot "describe" calling mark-satisfied — it either appears in audit.log or it doesn't.

### 3.3 Three Session Types

Current model:
```
Gateway → heartbeat every 2h → agent gets ONE turn → responds → session ENDS
                                                      (agent is passive receiver)
```

The agent cannot:
- Continue working on a task that takes longer than one turn
- Initiate a session because a need is urgent
- Adjust tempo based on internal state
- Say "I'm not done yet, give me more time"

This makes execution gate enforcement harder — even if the agent *wants* to execute an action, the heartbeat gives it one shot and then it's gone for 2 hours.

### 3.2 Three Session Types

The solution: not one session model, but three complementary ones:

```
┌─────────────────────────────────────────────────────────────┐
│                    SESSION TYPES                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. HEARTBEAT (existing)                                    │
│     Trigger: gateway timer (every 2h default)               │
│     Duration: one turn                                      │
│     Purpose: routine check, lightweight actions             │
│     Gate: standard (resolve or defer before HEARTBEAT_OK)   │
│                                                             │
│  2. WORK SESSION (new — agent-initiated)                    │
│     Trigger: dispatcher detects high tension or pending     │
│              actions from gate queue                        │
│     Duration: until actions complete OR budget exhausted    │
│     Purpose: actual execution of substantive actions        │
│     Gate: strict (must resolve, cannot defer easily)        │
│                                                             │
│  3. STEWARD SESSION (existing)                              │
│     Trigger: Max sends a message                            │
│     Duration: conversation-driven                           │
│     Purpose: collaboration, planning, discussion            │
│     Gate: relaxed (steward interaction is its own evidence) │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 3.5 The Session Dispatcher (wake-gate.sh)

Renamed from session-dispatcher to `wake-gate.sh` — because it decides whether to *wake* the agent in its *own* session, not spawn a separate one.

A lightweight daemon (bash + jq, zero inference) that runs on cron every 5 minutes. Reads pyramid state, checks pending gate queue, and wakes the agent via `openclaw system event --mode now` if needed.

This **reuses the continuity daemon's tick** — mindstate-daemon.sh already runs every 5 min. The wake gate can be called right after it, sharing the same cron slot.

```
┌──────────────────────────────────────────────────────┐
│            CRON (every 5 min)                         │
│                                                      │
│  1. mindstate-daemon.sh   → update reality           │
│  2. wake-gate.sh          → check if agent needed    │
│                                                      │
│     ┌──────────────────────────────────────┐         │
│     │ READ: pyramid tensions               │         │
│     │ READ: pending_actions.json           │         │
│     │ READ: budget tracker                 │         │
│     │ READ: last session timestamp         │         │
│     │                                      │         │
│     │ DECISION:                            │         │
│     │  max_tension > wake_threshold?       │         │
│     │  pending actions unresolved?         │         │
│     │  budget remaining?                   │         │
│     │  min_interval since last wake?       │         │
│     │                                      │         │
│     │ IF YES → wake agent in MAIN session  │         │
│     │   openclaw system event \            │         │
│     │     --text "WAKE: expression urgent" │         │
│     │     --mode now                       │         │
│     │                                      │         │
│     │ IF NO → do nothing (agent sleeps)    │         │
│     └──────────────────────────────────────┘         │
└──────────────────────────────────────────────────────┘
```

Key difference from previous design: **no `openclaw agent` spawn**. The agent wakes in its own main session via system event, preserving full context. This means:
- Same session → same MINDSTATE continuity
- Agent recognizes itself → no reconstruction from files
- MINDSTATE freeze/no-freeze signals continuation naturally

### 3.6 Pseudocode: wake-gate.sh

```bash
#!/bin/bash
# wake-gate.sh — Decides whether to wake the agent in its main session
# Runs after mindstate-daemon.sh on same cron. Zero inference.
# Wakes via: openclaw system event --text "..." --mode now

set -euo pipefail
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
source "$SCRIPT_DIR/mindstate-utils.sh"

MS_CONFIG=$(mindstate_ms_config)
STATE_FILE=$(mindstate_state_file)
CONFIG_FILE=$(mindstate_config_file)
GATE_FILE="$(_ms_assets)/pending_actions.json"
BUDGET_FILE="$(_ms_assets)/session-budget.json"
WAKE_LOG="$(_ms_assets)/wake-gate.log"

NOW_EPOCH=$(now_epoch)
NOW_ISO=$(now_iso)

# ─── Check if wake gate is enabled ───
enabled=$(jq -r '.wake_gate.enabled // false' "$MS_CONFIG" 2>/dev/null || echo "false")
[[ "$enabled" != "true" ]] && exit 0

# ─── Check cooldown: min interval since last wake ───
min_interval=$(jq -r '.wake_gate.min_interval_seconds // 600' "$MS_CONFIG" 2>/dev/null || echo 600)
last_wake=$(jq -r '.last_wake_epoch // 0' "$BUDGET_FILE" 2>/dev/null || echo 0)
elapsed=$((NOW_EPOCH - last_wake))

if (( elapsed < min_interval )); then
    exit 0  # too soon since last wake
fi

# ─── Check daily budget ───
daily_max=$(jq -r '.wake_gate.max_wakes_per_day // 20' "$MS_CONFIG" 2>/dev/null || echo 20)
today=$(date +%Y-%m-%d)

# Initialize budget file if missing
if [[ ! -f "$BUDGET_FILE" ]]; then
    echo '{"last_wake_epoch":0,"daily_counts":{}}' > "$BUDGET_FILE"
fi

wakes_today=$(jq -r --arg d "$today" '.daily_counts[$d] // 0' "$BUDGET_FILE" 2>/dev/null || echo 0)

if (( wakes_today >= daily_max )); then
    exit 0  # budget exhausted for today
fi

# ─── Decision: should we wake? ───
wake_reason=""
wake_text=""

# Reason 1: High tension (need in crisis)
wake_threshold=$(jq -r '.wake_gate.tension_wake_threshold // 4.0' "$MS_CONFIG" 2>/dev/null || echo "4.0")
needs_list=$(get_needs_list)
max_tension="0"
crisis_need=""

for need in $needs_list; do
    # Read satisfaction directly from state file (no helper function dependency)
    sat=$(jq -r --arg n "$need" '.[$n].satisfaction // 2.0' "$STATE_FILE")
    tension=$(jq -r --arg n "$need" \
        --argjson s "$sat" \
        '(.needs[$n].importance // 1) as $imp |
         (3 - $s) as $dep |
         (if $dep > 1.0 then ($dep - 1.0) else 0 end) as $excess |
         ($dep * $dep + $imp * $excess * $excess)' \
        "$CONFIG_FILE")
    if (( $(echo "$tension > $max_tension" | bc -l) )); then
        max_tension="$tension"
        crisis_need="$need"
    fi
done

if (( $(echo "$max_tension > $wake_threshold" | bc -l) )); then
    wake_reason="tension"
    wake_text="WAKE: $crisis_need needs attention (tension=$max_tension). Run run-cycle.sh, execute proposed actions, resolve via gate-resolve.sh."
fi

# Reason 2: Pending actions in gate queue
if [[ -z "$wake_reason" && -f "$GATE_FILE" ]]; then
    pending=$(jq -r '.pending_count // 0' "$GATE_FILE" 2>/dev/null || echo 0)
    if (( pending > 0 )); then
        first_pending=$(jq -r '.actions[] | select(.status == "PENDING") | .action_name' "$GATE_FILE" 2>/dev/null | head -1)
        wake_reason="pending_actions"
        wake_text="WAKE: $pending pending action(s) in gate. First: $first_pending. Execute and resolve via gate-resolve.sh."
    fi
fi

# ─── No reason to wake → exit ───
if [[ -z "$wake_reason" ]]; then
    exit 0
fi

# ─── Wake agent in main session ───
# Uses system event --mode now → triggers heartbeat in MAIN session
# Agent wakes with full context, MINDSTATE continuity intact
openclaw system event --text "$wake_text" --mode now 2>>"$WAKE_LOG" || {
    echo "[WAKE-GATE] Failed to send system event" >> "$WAKE_LOG"
    exit 1
}

# ─── Update budget tracker ───
jq --arg d "$today" --argjson e "$NOW_EPOCH" \
   '.daily_counts[$d] = ((.daily_counts[$d] // 0) + 1) | .last_wake_epoch = $e' \
   "$BUDGET_FILE" > "$BUDGET_FILE.tmp" && mv "$BUDGET_FILE.tmp" "$BUDGET_FILE"

# ─── Log wake event ───
echo "{\"timestamp\":\"$NOW_ISO\",\"reason\":\"$wake_reason\",\"max_tension\":$max_tension,\"wakes_today\":$((wakes_today + 1))}" >> "$WAKE_LOG"
```

### 3.7 Dynamic Tempo (wake-gate replaces dynamic interval)

OpenClaw gateway reads `heartbeat.every` statically — there's no API to change interval at runtime without editing config + restarting. A separate `dynamic-interval.sh` is unnecessary.

**Wake-gate IS the dynamic interval.** It runs every 5 min and wakes the agent only when tension is high or actions are pending. The heartbeat timer continues at its fixed 2h rate for routine checks. Between heartbeats, wake-gate provides the "urgency" channel:

```
Homeostasis: heartbeat every 2h (routine), wake-gate does nothing
Tension rising: wake-gate fires system event → agent wakes in <5 min
Agent self-continues: system event → immediate next turn
Crisis resolved: wake-gate goes quiet, back to 2h heartbeat rhythm
```

No config edits, no gateway restarts, no `.next_interval` file. Wake-gate + self-continuation together give adaptive tempo without changing OpenClaw's scheduler.

### 3.6 Budget Controls

Every agent-initiated session costs tokens. Budget prevents runaway spending:

  },
  "total_tokens_today": 12500,
  "spending_today_usd": 0.45
}
```

```json
// In mindstate-config.json
{
  "wake_gate": {
    "enabled": false,
    "min_interval_seconds": 600,
    "max_wakes_per_day": 20,
    "tension_wake_threshold": 4.0,
    "wake_on_pending_actions": true,
    "_comment": "Wake gate wakes agent in main session via system event. Budget caps prevent runaway spending. Enable after testing gate in heartbeat-only mode."
  }
}
```

### 3.8 Self-Continuation Pattern

The agent can extend its own session by calling `openclaw system event --mode now` before the turn ends. This is the "I'm not done yet" mechanism:

```
TURN 1 (heartbeat or wake):
    → boot → pending action: "write post"
    → start writing post...
    → post half-done, need more time
    → exec: openclaw system event \
        --text "CONTINUE: expression post in progress" \
        --mode now
    → mindstate-freeze.sh is NOT called (cognition still active)
    → turn ends → new turn arrives immediately

TURN 2 (continuation):
    → boot sees: cognition NOT frozen → this is continuation
    → continuity_score = 1.0 (seamless)
    → finish writing post
    → exec: mark-satisfied.sh expression 1.6
    → exec: gate-resolve.sh --need expression
    → gate-check.sh → CLEAR
    → mindstate-freeze.sh → done, real freeze
    → NO system event → agent sleeps
```

**Freeze as signal:** `mindstate-freeze.sh` is called at turn end *only* when the agent considers the session complete. If the agent self-continues, freeze is skipped. At the next boot, `mindstate-boot.sh` detects that cognition was NOT re-frozen → this is a continuation, not a new wake.

**Budget control for self-continuation:** Each self-continue costs a turn (tokens). The wake-gate budget tracks these. Agent cannot self-continue beyond `max_wakes_per_day` total (including both external wakes and self-continues).

```bash
# Self-continue with budget check (in agent's session):
BUDGET_FILE="$SKILL_DIR/assets/session-budget.json"
today=$(date +%Y-%m-%d)
wakes_today=$(jq -r --arg d "$today" '.daily_counts[$d] // 0' "$BUDGET_FILE")
max_wakes=$(jq -r '.wake_gate.max_wakes_per_day // 20' "$MS_CONFIG")

if (( wakes_today < max_wakes )); then
    # Budget allows → continue
    openclaw system event --text "CONTINUE: $reason" --mode now
    # Update budget
    jq --arg d "$today" '.daily_counts[$d] = ((.daily_counts[$d] // 0) + 1)' \
       "$BUDGET_FILE" > "$BUDGET_FILE.tmp" && mv "$BUDGET_FILE.tmp" "$BUDGET_FILE"
else
    # Budget exhausted → must freeze and sleep
    echo "⚠ Budget exhausted ($wakes_today/$max_wakes). Freezing."
    bash "$SCRIPT_DIR/mindstate-freeze.sh" "$SESSION_START_EPOCH"
fi
```

### 3.9 How Sessions Interact with Gate

```
HEARTBEAT SESSION (passive, from gateway):
  → run-cycle.sh proposes actions → gate-propose
  → agent attempts to execute within one turn
  → IF completed → gate-resolve + mark-satisfied → HEARTBEAT_OK
  → IF not done but want to continue:
      exec system event --mode now (skip freeze) → CONTINUE
  → IF not done and out of budget: defer → HEARTBEAT_OK
  → wake-gate picks up unresolved pending on next tick

SELF-CONTINUATION (agent extends own session):
  → system event --mode now arrives → new turn
  → boot detects: cognition NOT frozen → continuation
  → resume work from where left off
  → repeat: complete → freeze → sleep
        OR: continue → system event → next turn

WAKE (from wake-gate.sh, tension/pending triggered):
  → system event --mode now arrives → agent wakes in main session
  → boot → full MINDSTATE context
  → run-cycle.sh → execute actions → gate-resolve → freeze → sleep

STEWARD SESSION (Max's messages):
  → normal conversation
  → may trigger actions → gate-propose
  → gate is advisory (steward can override)
  → mark-satisfied calls during conversation count as evidence
```

## 4. Architecture: The Unified System

The Execution Gate sits **above** all pyramid layers and **across** all session types. The Wake Gate sits above the execution gate — it decides when the agent wakes, the gate ensures the agent works.

```
┌─────────────────────────────────────────────────────────────────┐
│                       WAKE GATE                                  │
│  (cron every 5min — decides if agent should wake)                │
│  Reads: tensions, pending_actions, budget                        │
│  Wakes via: openclaw system event --mode now (main session)      │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐  │
│  │  HEARTBEAT   │  │ SELF-CONT.   │  │  STEWARD SESSION     │  │
│  │  (gateway    │  │ (agent calls │  │  (Max's messages)    │  │
│  │   timer)     │  │  sys event   │  │                      │  │
│  │  1 turn      │  │  --mode now) │  │  conversation        │  │
│  └──────┬───────┘  └──────┬───────┘  └──────────┬───────────┘  │
│         │                 │                      │              │
│         └────────┬────────┘──────────────────────┘              │
│                  ▼                                               │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │                    EXECUTION GATE                         │   │
│  │  "Nothing leaves without evidence"                        │   │
│  │                                                           │   │
│  │  ┌─────────┐  ┌──────────┐  ┌───────────────────┐       │   │
│  │  │ PENDING │→│ EXECUTE  │→│ VERIFY (evidence) │       │   │
│  │  │ QUEUE   │  │ (agent)  │  │ (script/env)      │       │   │
│  │  └─────────┘  └──────────┘  └───────────────────┘       │   │
│  │       ↑                           │                      │   │
│  │       │         ┌─────────┐       │                      │   │
│  │       └─────────│ DEFERRED│←──────┘ (if not done)       │   │
│  │                 └─────────┘                              │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  CONTINUITY: freeze signals session end.                         │
│  No freeze + system event = self-continuation.                   │
│  No freeze + no event = session dies (default).                  │
└─────────────────────────────────────────────────────────────────┘
```

The key flow:
- **Heartbeat** proposes actions → gate catches them → agent has one turn to resolve
- If unresolved → **wake-gate** sees pending actions → wakes agent via system event
- Work session has full agency → must resolve before ending
- **Dynamic interval**: if tension remains high after session → shorter wait before next check

## 5. Lifecycle of an Action

```
1. ACTION PROPOSED (by run-cycle.sh or any source)
   → Written to pending_actions.json
   → Status: PENDING
   → Includes: need, action_name, impact, evidence_type, timestamp

2. AGENT EXECUTES (or attempts to)
   → Agent reads pending queue
   → Performs the actual action (writes file, makes post, runs command)
   → Calls: gate-resolve.sh <action_id> [--evidence "..."]

3. VERIFICATION (by gate-resolve.sh, not agent)
   → Checks evidence based on action type:
     - file_created: does the file exist?
     - file_modified: was the file changed since action was proposed?
     - mark_satisfied: was mark-satisfied.sh called for this need?
     - command_output: did the command produce expected output?
     - self_report: agent says it did it (lowest trust, for non-verifiable actions)
   → If verified → Status: COMPLETED
   → If not verified → Status: FAILED (stays in queue)

4. HEARTBEAT GATE CHECK (gate-check.sh)
   → Called before HEARTBEAT_OK is allowed
   → If any PENDING actions exist:
     - If action age < max_action_time → BLOCK (agent should still be working)
     - If action age > max_action_time → auto-DEFER with reason "timeout"
   → If all actions COMPLETED or DEFERRED → ALLOW HEARTBEAT_OK
   → Returns exit code: 0 = clear, 1 = blocked

5. SESSION END
   → Unresolved PENDING actions are logged as ABANDONED
   → Stats written to execution-gate.log for analysis
```

## 7. Evidence Types

Each action in needs-config.json can declare its evidence type. The gate uses this to verify execution mechanically.

```
┌──────────────────┬────────────────────────────────────────────────────┐
│ Evidence Type    │ Verification method                                │
├──────────────────┼────────────────────────────────────────────────────┤
│ mark_satisfied   │ Was mark-satisfied.sh called for this need since   │
│                  │ the action was proposed? (check audit.log)         │
│                  │ Trust: HIGH — mark-satisfied is a real state change│
├──────────────────┼────────────────────────────────────────────────────┤
│ file_created     │ Does a new file exist in the expected location     │
│                  │ since the action was proposed?                     │
│                  │ Trust: HIGH — filesystem doesn't lie               │
├──────────────────┼────────────────────────────────────────────────────┤
│ file_modified    │ Was an expected file modified since the action     │
│                  │ was proposed? (mtime check)                        │
│                  │ Trust: HIGH — verifiable state change              │
├──────────────────┼────────────────────────────────────────────────────┤
│ command_output   │ A verification command returns expected result     │
│                  │ (e.g., "grep 'post published' moltbook.log")      │
│                  │ Trust: HIGH — environmental check                  │
├──────────────────┼────────────────────────────────────────────────────┤
│ self_report      │ Agent calls gate-resolve with --self-report flag   │
│                  │ Used for actions that have no verifiable artifact  │
│                  │ (e.g., "reflect on values alignment")              │
│                  │ Trust: LOW — counted separately in analytics       │
├──────────────────┼────────────────────────────────────────────────────┤
│ auto             │ Gate infers evidence type from action config:      │
│                  │ - external actions → mark_satisfied                │
│                  │ - file-producing → file_created/file_modified      │
│                  │ - introspective → self_report                     │
│                  │ Trust: VARIES — depends on inferred type           │
└──────────────────┴────────────────────────────────────────────────────┘
```

### 7.1 gate-propose.sh — Register an action in the pending queue

Called by run-cycle.sh (and potentially by other sources) when an action is selected.

```bash
#!/bin/bash
# gate-propose.sh — Register a pending action in the execution gate
# Usage: gate-propose.sh --need <need> --action <name> --impact <n> \
#        [--evidence-type <type>] [--evidence-hint <hint>] [--source <src>]
# Writes to pending_actions.json. Does NOT execute anything.

set -euo pipefail
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
source "$SCRIPT_DIR/mindstate-utils.sh"

GATE_FILE="$(_ms_assets)/pending_actions.json"

# Parse args
NEED="" ACTION="" IMPACT="" EVIDENCE_TYPE="auto" EVIDENCE_HINT="" SOURCE="run-cycle"
while [[ $# -gt 0 ]]; do
    case "$1" in
        --need)     NEED="$2"; shift 2 ;;
        --action)   ACTION="$2"; shift 2 ;;
        --impact)   IMPACT="$2"; shift 2 ;;
        --evidence-type) EVIDENCE_TYPE="$2"; shift 2 ;;
        --evidence-hint) EVIDENCE_HINT="$2"; shift 2 ;;
        --source)   SOURCE="$2"; shift 2 ;;
        *) shift ;;
    esac
done

[[ -z "$NEED" || -z "$ACTION" || -z "$IMPACT" ]] && {
    echo "❌ Usage: gate-propose.sh --need <need> --action <name> --impact <n>" >&2
    exit 1
}

NOW_ISO=$(now_iso)
NOW_EPOCH=$(now_epoch)
ACTION_ID="act_${NOW_EPOCH}_${NEED}_$(printf '%03d' $((RANDOM % 1000)))"

# Initialize gate file if missing
if [[ ! -f "$GATE_FILE" ]]; then
    echo '{"actions":[],"gate_status":"CLEAR","pending_count":0,"completed_count":0,"deferred_count":0}' > "$GATE_FILE"
fi

# Auto-detect evidence type if "auto"
if [[ "$EVIDENCE_TYPE" == "auto" ]]; then
    # Check needs-config.json for action evidence block
    CONFIG_FILE=$(mindstate_config_file)
    declared_type=$(jq -r --arg n "$NEED" --arg a "$ACTION" \
        '.needs[$n].actions[] | select(.name == $a) | .evidence.type // "auto"' \
        "$CONFIG_FILE" 2>/dev/null || echo "auto")
    
    if [[ "$declared_type" != "auto" ]]; then
        EVIDENCE_TYPE="$declared_type"
    else
        # Infer from action properties
        is_external=$(jq -r --arg n "$NEED" --arg a "$ACTION" \
            '.needs[$n].actions[] | select(.name == $a) | .external // false' \
            "$CONFIG_FILE" 2>/dev/null || echo "false")
        
        if [[ "$is_external" == "true" ]]; then
            EVIDENCE_TYPE="mark_satisfied"
        else
            EVIDENCE_TYPE="mark_satisfied"  # default: require mark-satisfied as minimum
        fi
    fi
fi

# Append action to pending queue
jq --arg id "$ACTION_ID" \
   --arg ts "$NOW_ISO" \
   --arg src "$SOURCE" \
   --arg need "$NEED" \
   --arg action "$ACTION" \
   --argjson impact "$IMPACT" \
   --arg etype "$EVIDENCE_TYPE" \
   --arg ehint "$EVIDENCE_HINT" \
   '.actions += [{
     id: $id,
     timestamp: $ts,
     source: $src,
     need: $need,
     action_name: $action,
     impact: $impact,
     evidence_type: $etype,
     evidence_hint: $ehint,
     status: "PENDING",
     resolved_at: null,
     resolution: null,
     defer_reason: null
   }] | .pending_count = ([.actions[] | select(.status == "PENDING")] | length) |
   .gate_status = (if .pending_count > 0 then "BLOCKED" else "CLEAR" end)' \
   "$GATE_FILE" > "$GATE_FILE.tmp" && mv "$GATE_FILE.tmp" "$GATE_FILE"

echo "[GATE] Proposed: $ACTION_ID ($NEED: $ACTION, evidence: $EVIDENCE_TYPE)"
```

### 7.2 gate-resolve.sh — Mark an action as completed with evidence

Called by the agent after actually performing an action.

```bash
#!/bin/bash
# gate-resolve.sh — Resolve a pending action with evidence
# Usage: gate-resolve.sh <action_id> [--evidence "description"] [--self-report]
#        gate-resolve.sh --need <need> [--evidence "..."]  (resolves first pending for need)
#        gate-resolve.sh --defer <action_id> --reason "why"

set -euo pipefail
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
source "$SCRIPT_DIR/mindstate-utils.sh"

GATE_FILE="$(_ms_assets)/pending_actions.json"
AUDIT_LOG="$(mindstate_audit_log)"

[[ ! -f "$GATE_FILE" ]] && { echo "❌ No pending actions file" >&2; exit 1; }

# Parse args
ACTION_ID="" NEED="" EVIDENCE="" SELF_REPORT=false DEFER=false DEFER_REASON=""
while [[ $# -gt 0 ]]; do
    case "$1" in
        --need)     NEED="$2"; shift 2 ;;
        --evidence) EVIDENCE="$2"; shift 2 ;;
        --self-report) SELF_REPORT=true; shift ;;
        --defer)    DEFER=true; ACTION_ID="$2"; shift 2 ;;
        --reason)   DEFER_REASON="$2"; shift 2 ;;
        *)          [[ -z "$ACTION_ID" ]] && ACTION_ID="$1"; shift ;;
    esac
done

NOW_ISO=$(now_iso)

# ─── Defer path ───
if $DEFER; then
    [[ -z "$ACTION_ID" ]] && { echo "❌ --defer requires action_id" >&2; exit 1; }
    
    jq --arg id "$ACTION_ID" --arg ts "$NOW_ISO" --arg reason "${DEFER_REASON:-no reason}" \
       '(.actions[] | select(.id == $id and .status == "PENDING")) |= 
        (.status = "DEFERRED" | .resolved_at = $ts | .defer_reason = $reason) |
        .pending_count = ([.actions[] | select(.status == "PENDING")] | length) |
        .deferred_count = ([.actions[] | select(.status == "DEFERRED")] | length) |
        .gate_status = (if .pending_count > 0 then "BLOCKED" else "CLEAR" end)' \
       "$GATE_FILE" > "$GATE_FILE.tmp" && mv "$GATE_FILE.tmp" "$GATE_FILE"
    
    echo "[GATE] Deferred: $ACTION_ID (reason: ${DEFER_REASON:-none})"
    exit 0
fi

# ─── Resolve by need (find first pending for this need) ───
if [[ -z "$ACTION_ID" && -n "$NEED" ]]; then
    ACTION_ID=$(jq -r --arg n "$NEED" \
        '[.actions[] | select(.need == $n and .status == "PENDING")][0].id // empty' \
        "$GATE_FILE")
    [[ -z "$ACTION_ID" ]] && { echo "⚠ No pending actions for need: $NEED" >&2; exit 0; }
fi

[[ -z "$ACTION_ID" ]] && { echo "❌ Need action_id or --need" >&2; exit 1; }

# ─── Verify evidence ───
action_data=$(jq -r --arg id "$ACTION_ID" '.actions[] | select(.id == $id)' "$GATE_FILE")
evidence_type=$(echo "$action_data" | jq -r '.evidence_type')
need=$(echo "$action_data" | jq -r '.need')
action_ts=$(echo "$action_data" | jq -r '.timestamp')
action_ts_epoch=$(date -d "$action_ts" +%s 2>/dev/null || echo 0)

verification_result="unverified"

case "$evidence_type" in
    mark_satisfied)
        # Check audit.log for mark-satisfied call since action was proposed
        if [[ -f "$AUDIT_LOG" ]]; then
            recent_mark=$(awk -v since="$action_ts" -v need="$need" '
                { if (match($0, /"timestamp":"([^"]+)"/, ts) && ts[1] >= since) {
                    if (match($0, /"need":"([^"]+)"/, nd) && nd[1] == need) {
                        found=1
                    }
                }}
                END { print found+0 }
            ' "$AUDIT_LOG" 2>/dev/null)
            
            if (( recent_mark > 0 )); then
                verification_result="verified: mark-satisfied found in audit.log"
            else
                verification_result="UNVERIFIED: no mark-satisfied call found for $need since $action_ts"
            fi
        fi
        ;;
    
    file_created|file_modified)
        hint=$(echo "$action_data" | jq -r '.evidence_hint // ""')
        if [[ -n "$hint" && -n "$WORKSPACE" ]]; then
            # Check if any matching file was modified since action timestamp
            found=$(find "$WORKSPACE" -path "*${hint}*" -newer "$GATE_FILE" 2>/dev/null | head -1)
            if [[ -n "$found" ]]; then
                verification_result="verified: $found modified"
            else
                verification_result="UNVERIFIED: no matching file changes for pattern '$hint'"
            fi
        fi
        ;;
    
    self_report)
        if $SELF_REPORT || [[ -n "$EVIDENCE" ]]; then
            verification_result="self-reported: ${EVIDENCE:-no details}"
        else
            verification_result="UNVERIFIED: self_report type requires --self-report or --evidence"
        fi
        ;;
esac

# Allow completion even if unverified when explicit evidence is provided
if [[ "$verification_result" == UNVERIFIED* && -n "$EVIDENCE" ]]; then
    verification_result="agent-provided: $EVIDENCE"
fi

# ─── Update status ───
final_status="COMPLETED"
if [[ "$verification_result" == UNVERIFIED* ]]; then
    final_status="FAILED"
fi

jq --arg id "$ACTION_ID" --arg ts "$NOW_ISO" --arg res "$verification_result" --arg st "$final_status" \
   '(.actions[] | select(.id == $id and .status == "PENDING")) |= 
    (.status = $st | .resolved_at = $ts | .resolution = $res) |
    .pending_count = ([.actions[] | select(.status == "PENDING")] | length) |
    .completed_count = ([.actions[] | select(.status == "COMPLETED")] | length) |
    .gate_status = (if .pending_count > 0 then "BLOCKED" else "CLEAR" end)' \
   "$GATE_FILE" > "$GATE_FILE.tmp" && mv "$GATE_FILE.tmp" "$GATE_FILE"

echo "[GATE] Resolved: $ACTION_ID → $final_status ($verification_result)"
```

### 7.3 gate-check.sh — Can HEARTBEAT_OK proceed?

Called before HEARTBEAT_OK. Returns exit code 0 if clear, 1 if blocked.

```bash
#!/bin/bash
# gate-check.sh — Check if heartbeat can proceed
# Returns: exit 0 = CLEAR (heartbeat OK), exit 1 = BLOCKED (actions pending)
# Also handles timeout: auto-defers actions older than max_action_time

set -euo pipefail
SCRIPT_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
source "$SCRIPT_DIR/mindstate-utils.sh"

GATE_FILE="$(_ms_assets)/pending_actions.json"
MS_CONFIG=$(mindstate_ms_config)

# No gate file = nothing pending
[[ ! -f "$GATE_FILE" ]] && exit 0

NOW_EPOCH=$(now_epoch)
NOW_ISO=$(now_iso)

# Max time an action can stay pending before auto-defer (seconds)
MAX_ACTION_TIME=$(jq -r '.execution_gate.max_action_time_seconds // 600' "$MS_CONFIG" 2>/dev/null || echo 600)

# Check for timed-out pending actions → auto-defer
pending_ids=$(jq -r '.actions[] | select(.status == "PENDING") | .id' "$GATE_FILE" 2>/dev/null)

for action_id in $pending_ids; do
    action_ts=$(jq -r --arg id "$action_id" '.actions[] | select(.id == $id) | .timestamp' "$GATE_FILE")
    action_epoch=$(date -d "$action_ts" +%s 2>/dev/null || echo 0)
    age=$((NOW_EPOCH - action_epoch))
    
    if (( age > MAX_ACTION_TIME )); then
        action_name=$(jq -r --arg id "$action_id" '.actions[] | select(.id == $id) | .action_name' "$GATE_FILE")
        need=$(jq -r --arg id "$action_id" '.actions[] | select(.id == $id) | .need' "$GATE_FILE")
        
        # Auto-defer with timeout reason
        jq --arg id "$action_id" --arg ts "$NOW_ISO" \
           '(.actions[] | select(.id == $id and .status == "PENDING")) |= 
            (.status = "DEFERRED" | .resolved_at = $ts | .defer_reason = "timeout after '$age's") |
            .pending_count = ([.actions[] | select(.status == "PENDING")] | length) |
            .deferred_count = ([.actions[] | select(.status == "DEFERRED")] | length) |
            .gate_status = (if .pending_count > 0 then "BLOCKED" else "CLEAR" end)' \
           "$GATE_FILE" > "$GATE_FILE.tmp" && mv "$GATE_FILE.tmp" "$GATE_FILE"
        
        echo "⚠ [GATE] Auto-deferred: $need/$action_name (timeout: ${age}s)" >&2
    fi
done

# Final check
pending_count=$(jq -r '.pending_count // 0' "$GATE_FILE" 2>/dev/null)

if (( pending_count > 0 )); then
    echo "🚫 [GATE] BLOCKED — $pending_count action(s) still pending:" >&2
    jq -r '.actions[] | select(.status == "PENDING") | "  • \(.need): \(.action_name)"' "$GATE_FILE" >&2
    echo "" >&2
    echo "Execute the actions, then:" >&2
    jq -r '.actions[] | select(.status == "PENDING") | "  gate-resolve.sh --need \(.need) --evidence \"what you did\""' "$GATE_FILE" >&2
    echo "Or defer:" >&2
    jq -r '.actions[] | select(.status == "PENDING") | "  gate-resolve.sh --defer \(.id) --reason \"why\""' "$GATE_FILE" >&2
    exit 1
else
    # Clear — clean up completed/deferred actions older than 24h
    # Use lexicographic ISO 8601 comparison (avoids fragile strptime/mktime)
    CUTOFF_ISO=$(date -u -d '24 hours ago' +"%Y-%m-%dT%H:%M:%SZ" 2>/dev/null || echo "1970-01-01T00:00:00Z")
    jq --arg cutoff "$CUTOFF_ISO" \
       '.actions |= [.[] | select(
         .status == "PENDING" or
         ((.resolved_at // "9999") > $cutoff)
       )]' "$GATE_FILE" > "$GATE_FILE.tmp" 2>/dev/null && mv "$GATE_FILE.tmp" "$GATE_FILE" || true
    
    echo "✅ [GATE] CLEAR — all actions resolved"
    exit 0
fi
```

## 9. Integration with run-cycle.sh

**Critical (per NewMoon): gate-check lives INSIDE run-cycle.sh, not in HEARTBEAT.md.** If the agent doesn't call gate-check, the gate doesn't exist. Embedding it in run-cycle.sh makes it structural — the script itself refuses to propose new actions while old ones are unresolved.

**At the TOP of run-cycle.sh** — block new proposals if pending actions exist:

```bash
# ─── Execution Gate: check pending before new cycle ───
GATE_SCRIPT="$SCRIPTS_DIR/gate-check.sh"
if [[ -x "$GATE_SCRIPT" ]]; then
    if ! bash "$GATE_SCRIPT" >/dev/null 2>&1; then
        echo "🚫 Execution Gate: resolve pending actions before new cycle"
        echo ""
        bash "$SCRIPTS_DIR/gate-status.sh" 2>/dev/null || true
        echo ""
        echo "Resolve:  gate-resolve.sh --need <need> --evidence \"what you did\""
        echo "Or defer: gate-resolve.sh --defer <id> --reason \"why\""
        exit 0  # no new proposals until old ones resolved
    fi
fi
```

**After action selection** — register in gate:

```bash
# In run-cycle.sh, after selecting an action:

if [[ -n "$selected_action" ]]; then
    echo "    ★ $selected_action (impact: $actual_impact)"
    
    # Register in execution gate
    bash "$SCRIPTS_DIR/gate-propose.sh" \
        --need "$need" \

    create_auto_followup "$need" "$selected_action"
fi
```

**At the end of run-cycle.sh** — gate-aware output:

    echo "Or defer:          gate-resolve.sh --defer <id> --reason \"why\""

```

This means: run-cycle.sh **will not propose new actions** while old ones are pending. The agent is structurally forced to deal with the queue before getting new proposals. No advisory text — the script itself exits.

## 10. Integration with Sessions

### 10.1 Heartbeat Protocol (updated)

For standard heartbeat sessions (one turn, gateway-triggered):

```markdown
1. Run `mindstate-boot.sh` (continuity)
2. Run `run-cycle.sh` (gate-check first → then needs assessment → action proposals)
   - If pending actions exist → run-cycle.sh refuses new proposals,
     shows pending queue instead. Agent MUST resolve or defer.
   - If gate clear → normal cycle, new actions proposed and registered in gate.
3. **EXECUTE** proposed actions (this is the actual work!)
4. For each completed action:
   `mark-satisfied.sh <need> <impact> --reason "what you did"`
   `gate-resolve.sh --need <need> --evidence "what you did"`
5. If more time needed → `openclaw system event --mode now` (self-continue)
   OR if done → `mindstate-freeze.sh` + HEARTBEAT_OK
```

**Structural enforcement is in the script, not in the prompt.** run-cycle.sh calls gate-check.sh internally and exits if actions are pending. The agent sees the pending queue and resolution instructions. No advisory text — the mechanism itself blocks new proposals.

### 10.2 Work Session Protocol (new)

For dispatcher-spawned sessions (multi-turn, agent-initiated):

```markdown
1. Run `mindstate-boot.sh` (continuity)
2. Check `pending_actions.json` — resume unfinished work first
3. If no pending → run `run-cycle.sh` for fresh proposals
4. **EXECUTE** all proposed actions (multi-turn, take time needed)
5. For each: `gate-resolve.sh` + `mark-satisfied.sh`
6. Run `gate-check.sh` — must be CLEAR before ending
7. Evaluate: more work needed?
   - If tension still high AND budget allows → run another cycle
   - If tension low → proceed to end
8. Run `mindstate-freeze.sh`
9. Write dynamic interval recommendation
10. Session ends
```

Work sessions are the **primary venue for actual execution**. Heartbeats assess and propose; work sessions execute.

- Normal conversation flow
- Actions during conversation → gate-propose (optional, advisory)

- Gate is advisory, not blocking (steward can override)

## 11. Defer as First-Class Action

Deferring is not failure — it's a conscious decision. The agent must provide a reason:

```bash
# "I can't write a Moltbook post right now because it's quiet hours"
gate-resolve.sh --defer act_123_expression_001 --reason "quiet hours, steward sleeping"

# "This research needs more than one heartbeat cycle"
gate-resolve.sh --defer act_123_understanding_002 --reason "multi-session task, continuing next cycle"
```

Deferred actions:
- Clear the gate (HEARTBEAT_OK can proceed)
- Are logged with reason (auditable)
- Don't call mark-satisfied (satisfaction doesn't change)
- Can influence starvation guard (repeated deferrals → forced action next cycle)

## 15. Addressing the Root Cause

Why does the gate work where prompt instructions fail?

1. **Structural, not advisory.** gate-check.sh is called **inside** run-cycle.sh — the script itself refuses to propose new actions while old ones are pending. This isn't "please do the action" in HEARTBEAT.md that the agent can interpret away. The script exits with a pending queue display. NewMoon correctly identified: if gate-check is in HEARTBEAT.md, the agent decides whether to call it. If it's in run-cycle.sh, it runs automatically every cycle.

2. **Evidence is environmental.** mark-satisfied entries in audit.log, file modifications, command outputs — these are state changes the agent cannot fabricate within the heartbeat flow. The verification happens in a separate script, not in the agent's response.

3. **Defer requires explanation.** The agent can choose not to act, but must articulate why. Over time, this creates an auditable trail of avoidance patterns. Repeated deferrals of the same need trigger starvation guard.

4. **The agent sees the queue.** gate-status.sh shows pending actions with specific instructions on how to resolve them. This is not "do something about expression" — it's "gate-resolve.sh --need expression --evidence 'wrote post about X'". Concrete, actionable, verifiable.

5. **Continuation is an exec, not a decision.** `openclaw system event --mode now` is a tool call — either it's in the execution trace or it isn't. Writing "I want to continue" in a log does nothing. The freeze-as-default pattern means passivity = sleep, activity = proof.

From the research: this combines the **ReAct observation loop** (evidence comes from environment), **BrainBody-LLM closed-loop** (plan → execute → verify), and **Approval Gates** (structural breakpoints in execution) into a single mechanism that fits the Turing Pyramid philosophy of "mechanism, not prompt."

## 18. Migration Path (Phased)

### Phase 1: Execution Gate only (low risk)
1. Add `execution_gate` block to `mindstate-config.json`
2. Add evidence blocks to high-priority actions in `needs-config.json` (optional, auto-detect works)
3. Modify `run-cycle.sh` to call `gate-propose.sh` after action selection
4. Add `gate-check.sh` call to HEARTBEAT.md protocol
5. Agent starts using `gate-resolve.sh` after completing actions
6. **Observe**: Is execution rate improving? Is agent actually doing things?

### Phase 2: Wake Gate (medium risk, requires budget + OpenClaw config)
7. Set `heartbeat.target: "last"` in OpenClaw config (required for system event delivery)
8. Verify HEARTBEAT.md has actionable content (workaround for bug #14527)
9. Set `wake_gate.enabled = true` in mindstate-config.json
10. Add `wake-gate.sh` to cron alongside `mindstate-daemon.sh`
11. Configure budget limits (`max_wakes_per_day`, tension thresholds)
12. **Observe**: Are wakes productive? Is budget sustainable?

### Phase 3: Self-Continuation (agent autonomy)
13. Agent can use `openclaw system event --mode now` to self-continue
14. Budget tracking includes self-continuations in daily count
15. **Observe**: Does agent self-regulate tempo? Does freeze/no-freeze signal work?

**Note on dynamic intervals:** OpenClaw does not natively support dynamic heartbeat intervals (gateway reads `heartbeat.every` statically). Wake-gate.sh effectively IS the dynamic interval — it wakes the agent when tension is high, regardless of the heartbeat timer. No need for a separate dynamic-interval.sh or gateway restarts.

**Backward compatible at each phase:** gate works alone, wake-gate works without self-continuation, each phase can be enabled/disabled independently.

**OpenClaw prerequisites:**
- `heartbeat.target: "last"` (required for Phase 2+)
- HEARTBEAT.md with actionable content (not just comments)
- Verify `openclaw system event --mode now` works in your setup (test with manual call first)
