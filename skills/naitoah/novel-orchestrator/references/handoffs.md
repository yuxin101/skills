# 最小协作骨架

## 默认流程

### 从想法或目标落成章节

默认流程：`planner → writer → checker`

适用场景：
- 用户给的是想法、目标、方向或模糊需求
- 需要先把需求整理成可执行方案，再落成正文

### 明确写作任务

默认流程：`writer → checker`

适用场景：
- 写作目标已经足够明确
- 不需要额外先做结构拆解

### 返修任务

默认流程：`checker → writer → checker`

适用场景：
- 已有审查意见
- 需要按问题清单返修并复审

## planner 介入原则

- 当任务方向、章节目标、结构路径尚不清晰时，优先让 `planner` 介入。
- 当任务本身明确涉及规划、拆解、分卷或长线推进时，让 `planner` 先行。
- 当任务进入大修、救文、长线修整或多章连续产出时，`planner` 默认优先介入。
- `planner` 负责决定写什么、先写哪块、这一章承担什么任务。
- `writer` 负责把方案写成正文。
- 不要机械地把 `planner` 插入到每一次写作任务之前。

## manager 介入原则

当任务涉及以下情况时，再按需启用 `manager`：

- 跨阶段统筹
- 跨章节推进
- 结果汇总
- 角色冲突处理
- 阶段收口与下一步判断

## 返修上限

- 同一章节中，`writer` 最多进行 2 次返修；连同初稿在内，单章最多写 3 次。
- 如果单章在 2 次返修后仍未通过，不再继续让 `writer` 返修。
- 达到返修上限后，由 `checker` 输出最终审查报告，并提示用户介入审查与决策。
