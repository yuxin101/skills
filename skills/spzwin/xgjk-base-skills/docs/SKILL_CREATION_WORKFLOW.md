# Skill 包构建流程（5 步详细指南）

> 本文档是按 XGJK 协议模板从零构建 Skill 包的完整操作手册。
> 由 `SKILL.md` 路由至此，协议规范详见 `XGJK_SKILL_PROTOCOL.md`。

---

## Step 1：意图理解与需求确认（多轮对话）

> **核心目标**：搞清楚要创建什么 Skill — 它是干什么的、有哪些能力、对应哪些 API、每个 API 需要哪些字段。

### 1.1 理解业务场景

用户说"我要构建一个 Skill"时，AI 需要先了解：

1. **这个 Skill 是做什么的？**
   - 业务场景、解决什么问题
   - 比如：管理 IM 机器人、发送消息、管理会话组...

2. **谁会用？怎么触发？**
   - AI Agent 自动调用？还是人手动触发？
   - 什么场景下会用到这个 Skill？

3. **核心能力范围**
   - 用户期望这个 Skill 能做哪些事情
   - 这决定了后面筛选 API 的依据

### 1.2 确认 Skill 基本声明

在对话中自然确认：
- **Skill 名称**（英文，如 `im-robot`）
- **Skill 描述**（一句话说明做什么）
- **模块划分**（如果 API 跨多个业务域，划分为多个模块）

### 1.3 获取接口文档

引导用户提供接口文档，**支持以下任意方式**：

**方式 A：提供接口文档 URL**（推荐）

```bash
python3 create-xgjk-skill/scripts/fetch_api_doc.py "<url>" [--output-dir ./temp]
```

脚本自动识别：
- **Markdown 文档**（普通 URL）→ 直接获取内容
- **Swagger 链接**（含 `doc.html#/...`）→ 解析 Swagger JSON，提取目标接口完整子集
- **支持混合使用**：一次传多个 URL

**方式 B：用户直接描述接口**

用户直接告诉 AI 接口信息（路径、方法、参数、响应），跳过脚本。

### 1.4 筛选 API（需要用户确认）

接口文档里的 API 很多，**不是全部都要**。AI 根据业务需求动态筛选：

1. 展示文档中所有 API 概览（按功能分组）
2. 根据业务需求推荐（✓ 建议纳入 / ? 可选 / ✗ 建议不纳入）
3. **用户确认**选中的 API 列表

### 1.5 精简字段（需要用户确认）

对选中的每个 API，精简输入和输出字段：

**默认保留**：`required: true` 的字段、核心业务标识、有 `enum` 约束的字段、驱动业务流程的字段
**默认移除**：元数据字段、纯前端展示字段、内部中转字段
**不确定的**：直接问用户

精简完成后，**用户最终确认**，确认后进入 Step 2。

---

## Step 2：按协议逐步生成 Skill 包

> **核心规则**：严格按 `XGJK_SKILL_PROTOCOL.md` 的规范执行，每个子步骤顺序进行，不跳不并。

### 2.1 创建目录骨架 + 写入固定文件

**第一步永远是搭骨架。**

1. 创建目标 Skill 的完整目录结构：

```bash
mkdir -p <skill-name>/common
mkdir -p <skill-name>/openapi/common
mkdir -p <skill-name>/openapi/<module>    # 每个模块
mkdir -p <skill-name>/examples/<module>
mkdir -p <skill-name>/scripts/<module>
```

2. 写入 3 个固定文件（**不改一个字**）：

| 文件 | 来源 | 写入路径 |
|---|---|---|
| 认证鉴权规范 | `XGJK_SKILL_PROTOCOL.md` 附录 A | `<skill>/common/auth.md` |
| 通用约束规范 | `XGJK_SKILL_PROTOCOL.md` 附录 B | `<skill>/common/conventions.md` |
| Token 交换接口 | `XGJK_SKILL_PROTOCOL.md` 附录 C | `<skill>/openapi/common/appkey.md` |

> **⚠️ 危险红线**：这 3 个文件是系统底层基石，**严禁做任何修改**。不准重构、不准精简、不准改变量名。

**写入后检查**：确认 3 个文件全部存在，缺一不可。

> **✅ 自检关卡**：按协议第三章步骤 1 自检关卡逐项核对，全部通过才进入 2.2。

### 2.2 生成 SKILL.md（定义工作流与编排）

根据 Step 1 确认的业务信息，按**协议 §2.1** 的格式生成目标 Skill 的 `SKILL.md`：

- **能力宪章**：定义这个 Skill 的规则边界
- **能力概览**：列出各模块的能力摘要
- **工作流**：定义 AI 使用这个 Skill 时的标准流程
- **意图路由表**：用户说什么话 → 路由到哪个模块 → 打开哪个文档 → 执行哪个脚本
- **能力树**：完整的目录结构（与实际文件一一对应）

> **关键**：这不是模板填充。要结合用户的业务场景重新组织语言，写用户实际会说的话、实际的触发条件。

> **✅ 自检关卡**：按协议第三章步骤 2 自检关卡逐项核对，全部通过才进入 2.3。

### 2.3 逐个 API 循环生成（核心步骤）

> **一个 API 一个 API 来，每个 API 完成全套再进入下一个。**

假设 Step 1 确认了 N 个 API，按以下循环执行：

```
for 第 i 个 API (i = 1 到 N):
  │
  ├─ ① 写接口文档 openapi/<module>/<endpoint>.md
  │    按协议 §2.3：作用、Headers、参数表、请求 Schema、响应 Schema、脚本映射
  │    字段只写 Step 1.5 精简确认后保留的字段
  │
  ├─ ② 写执行脚本 scripts/<module>/<endpoint>.py
  │    按协议 §2.6 标准模式：
  │    - 读取环境变量 XG_USER_TOKEN（仅需鉴权的接口）
  │    - API_URL 硬写完整 URL（与 ① 的接口文档一致）
  │    - 构造请求（Headers、Body）
  │    - 调 API，获取原始 JSON
  │    - print(json.dumps(result, ensure_ascii=False))
  │    - 入参字段必须与 ① 的参数表完全一致
  │
  └─ ③ 补充触发场景到 examples/<module>/README.md
       什么时候触发这个接口、标准流程是什么
```

**每个 API 完成后的迷你自检**（2 项全过才进入下一个 API）：
- endpoint.md 的 URL 与 endpoint.py 的 `API_URL` 是否完全一致？
- endpoint.md 的参数表字段与 endpoint.py 的请求 body 字段是否完全一致？

> **✅ 自检关卡**：所有 API 完成后，按协议第三章步骤 3 自检关卡逐项核对，全部通过才进入 2.4。

### 2.4 生成索引文件

所有 API 都完成后，生成各模块的索引：

1. **`openapi/<module>/api-index.md`**（按协议 §2.2）
   - 列出本模块所有接口的清单 + 文档路径 + 脚本映射

2. **`scripts/<module>/README.md`**（按协议 §2.5）
   - 脚本清单 + 共享依赖 + 使用方式 + 环境变量（仅需鉴权的脚本需要 XG_USER_TOKEN）

> 索引放最后写，因为此时所有 API 已确定，索引内容最完整最准确。

> **✅ 自检关卡**：按协议第三章步骤 4 自检关卡逐项核对，全部通过才进入 Step 3。

---

## Step 3：三轮反思检查 + 修复

> **⚠️ 强制步骤，不可跳过。无论生成过程是否顺利，三轮都必须完整执行。**
> 对应协议第三章步骤 5（5a/5b/5c），Step 4 对应 5d。

### 3.1 第一轮：逐项核对验证清单

按 `SKILL_VALIDATION_CHECKLIST.md`（A-H 全部 8 节），**逐条核对每一项**：

1. **逐项打勾**：对每一条给出 ✅ 或 ❌ 的明确判定，**不允许笼统地说"全部通过"**
2. **发现问题立即修复**：不通过的项，立即修改对应文件，修改后重新验证
3. **输出检查报告**：将 A-H 各节的检查结果汇总输出

### 3.2 第二轮：交叉验证一致性（必须输出证据）

> **⚠️ 不能只说"通过"，必须列出具体证据。** 协议第三章步骤 5b 定义了每项检查的证据输出格式，必须遵循。

1. **文件计数核对**：用公式 `3 + 1 + M×3 + N×2` 计算期望值，逐一列出实际文件，比对是否一致
2. **1:1 绑定验证**：列出每一对 `endpoint.md ↔ endpoint.py` 的对应关系
3. **SKILL.md 能力树 vs 实际目录**：逐行比对，列出差异
4. **占位符检查**：全文搜索 `<module>`、`<endpoint>`、`<skill-name>` 等占位符，列出搜索结果
5. **入参校验**：列出每个脚本的请求字段，与对应 endpoint.md 的参数表逐一比对

如有不通过项，**立即修复后重新检查该项**，直到全部通过。

### 3.3 第三轮：与参考示例结构比对

对照 `XGJK_SKILL_PROTOCOL.md` **附录 E 的完整迷你示例**，检查生成的 Skill 包结构特征：

1. 目录层级是否与示例一致？
2. SKILL.md 是否包含示例中的所有必备节（宪章、概览、工作流、路由表、能力树）？
3. 每个 endpoint.md 是否包含所有必备节（作用、Headers、参数表、请求 Schema、响应 Schema、脚本映射）？
4. 每个 endpoint.py 是否遵循示例中的脚本结构（docstring → import → API_URL → call_api → main → 守卫）？
5. api-index.md 和 scripts/README.md 是否与示例格式一致？

---

## Step 4：最终确认

确认 Step 3 三轮反思中发现的所有问题已全部修复，无遗留项：

1. 回顾 3.1 的检查报告，确认所有 ❌ 项已变为 ✅
2. 回顾 3.2 的证据输出，确认无矛盾
3. 回顾 3.3 的结构比对，确认无差异
4. 如仍有未解决项，**重新执行对应的反思轮次**，直到全部清零

---

## Step 5：完成

全部确认通过后，输出最终总结：

```
✅ Skill 构建完成
- Skill 名称：<name>
- 模块数量：M 个
- 接口数量：N 个
- 脚本数量：N 个
- 总文件数：公式值 X = 实际值 X
- 三轮反思：全部通过
  - 第一轮验证清单 A-H：全部通过
  - 第二轮交叉验证：全部通过（附证据）
  - 第三轮结构比对：与参考示例一致
- 最终确认：无遗留问题
```
