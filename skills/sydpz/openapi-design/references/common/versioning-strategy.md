# API Versioning Strategy

## 版本策略概述

### 推荐的版本策略：URL 路径版本

```
✅ /v1/users
✅ /v2/users
```

这是最直观、最广泛使用的 API 版本控制方式。

---

## URL 版本控制

### 基本格式

```
/v{version}/{resource}

示例：
/v1/users
/v1/products/123
/v2/orders
```

### 版本号规则

1. **使用主版本号**（v1, v2, v3）
2. **不使用次版本号**（v1.2, v1.3）
3. **版本号只增不减**

### 优势

| 优势 | 说明 |
|------|------|
| 直观 | 路径明确显示版本 |
| 可缓存 | URL 是缓存 key |
| 可发现 | 通过 URL 即可知道版本 |
| 简单 | 无需复杂的 Header 处理 |

## 何时需要升级版本

### 需要升级版本

| 场景 | 示例 |
|------|------|
| 破坏性变更 | 删除字段、改变字段类型、改变端点行为 |
| 删除端点 | 移除某个 API |
| 改变必填参数 | 原可选参数变为必填 |
| 改变认证方式 | 改变 token 格式 |

### 不需要升级版本

| 场景 | 示例 |
|------|------|
| 新增端点 | 添加 `/v1/webhooks` |
| 新增可选参数 | 添加可选查询参数 |
| 新增字段（响应） | 在响应中增加字段 |
| 修复 bug | 不改变接口行为的 bug 修复 |
| 性能优化 | 内部实现优化 |

## 版本迁移策略

### 生命周期阶段

```
v1 (当前) → v2 (新版本) → v3 (废弃) → 关闭
   |           |           |
   |      新功能       遗留系统
   |      迁移指南     迁移完成
   |
主版本支持期（通常 12-18 个月）
```

### 迁移步骤

1. **发布新版本** — v2 与 v1 同时运行
2. **提供迁移指南** — 文档说明从 v1 到 v2 的变更
3. **设置废弃警告** — 在响应 Header 中提示新版本
4. **废弃旧版本** — 宣布旧版本下线时间
5. **关闭旧版本** — 停止服务

### 废弃 Header

```
# 响应 Header 提示新版本
X-API-Warning: Deprecated API version. Please migrate to /v2/users.
```

## 多版本共存策略

### 共存期示例

| 时间 | v1 | v2 | v3 |
|------|----|----|-----|
| 2024-01 | ✅ 活跃 | | |
| 2024-06 | ✅ 活跃 | ✅ 新增 | |
| 2024-12 | ⚠️ 废弃 | ✅ 活跃 | |
| 2025-06 | ❌ 关闭 | ⚠️ 废弃 | ✅ 活跃 |

### 响应 Header 示例

```http
HTTP/1.1 200 OK
X-API-Version: v1
X-API-Deprecated: true
X-API-Sunset: Sat, 01 Jun 2025 00:00:00 GMT
X-API-Successor: /v2/users
```

| Header | 说明 |
|--------|------|
| `X-API-Version` | 当前请求的版本 |
| `X-API-Deprecated` | 是否已废弃 |
| `X-API-Sunset` | 废弃时间 |
| `X-API-Successor` | 推荐的替代版本 |

## 内部版本管理

### 细粒度版本（可选）

对于需要更精细控制的场景：

```http
# Header 方式（可选）
API-Version: 2024-01-15

# 或
Accept: application/vnd.myapi.v1+json
```

### 适用场景

1. **频繁变更的 API** — 每月多个小版本
2. **灰度发布** — A/B 测试不同实现
3. **实验性功能** — 功能开关控制

### 不适用场景

1. **公开 API** — 客户端难以跟踪 Header 版本
2. **简单场景** — 过度工程

## 版本选择建议

### 推荐：URL 版本

```bash
GET /v1/users
GET /v2/users
```

### 可选：Header 版本（补充）

```bash
# 结合 URL 版本使用
GET /users
API-Version: 2024-01-15

# 或
GET /users
Accept: application/vnd.myapi.v2+json
```

### 不推荐

```
❌ Query 参数版本（?version=2）— 难以缓存
❌ 子域名版本（v2.api.example.com）— 管理复杂
❌ 日期版本（?api_date=20240115）— 版本语义不清晰
```

## 常见问题

### Q: 能否跳过版本？

**A**: 可以但要谨慎。例如从 v1 直接到 v3，v2 可以不发布。

### Q: 旧版本能运行多久？

**A**: 建议至少支持 12 个月，给调用方足够迁移时间。

### Q: 版本升级需要客户端重新测试吗？

**A**: 通常需要。破坏性变更会影响客户端行为。
