# 用户访谈方法论 - v6 架构设计指南

**版本：** 6.0.0  
**创建时间：** 2026-03-19  
**说明：** v6 架构新增 Planning Phase，在创建任何 Agent 之前必须进行结构化用户访谈

---

## 🎯 为什么需要 Planning Phase

v6 架构引入三层级（User ↔ Main Agent ↔ Manager Agent ↔ Worker Agents），**规划阶段**是全新环节。

**核心原则：**
- 先理解用户 workflow，再设计 Agent 团队
- 不套用模板，每个团队都是定制设计
- 用户参与决策，不是被动接受配置

**输出物：** Team Design Document（团队设计文档）

---

## 📋 结构化发现问卷（Discovery Questions）

### 第一轮：Workflow 理解（5 个问题）

#### Q1: 请描述你典型的一天工作流程
> "Walk me through your typical day from start to finish."

**追问点：**
- 哪些任务占用了你最多时间？
- 哪些任务让你感到疲惫或挫败？
- 一天中什么时候效率最高？

**映射目标：** 识别高频任务、时间黑洞、能量曲线

---

#### Q2: 你目前最大的三个痛点是什么？
> "What are the top 3 things that frustrate you most in your current workflow?"

**追问点：**
- 这些痛点多久发生一次？
- 你尝试过什么解决方法？
- 如果只能解决一个，你会选哪个？

**映射目标：** 确定优先级、识别阻塞点

---

#### Q3: 你的任务可以分成哪几类？
> "How would you categorize the types of work you do?"

**引导选项：**
| 类别 | 示例 |
|------|------|
| 创意型 | 设计、写作、头脑风暴 |
| 分析型 | 数据分析、调研、决策 |
| 执行型 | 编码、写作、制作 |
| 审查型 | 代码 review、校对、质检 |
| 协调型 | 项目管理、沟通、排期 |

**映射目标：** 匹配 Task Categories → Agent Roles

---

#### Q4: 你需要怎样的协作模式？
> "How do you prefer to collaborate with AI assistants?"

**选项矩阵：**

| 模式 | 特点 | 适合场景 |
|------|------|----------|
| **自主型** | 给目标，AI 自己探索 | 深度研究、代码重构 |
| **陪伴型** | 边做边讨论，实时反馈 | 设计迭代、写作润色 |
| **流水线型** | 明确分工，按步骤执行 | 标准化流程、批量处理 |
| **顾问型** | 先咨询建议，再决定 | 战略决策、方案选择 |

**映射目标：** 确定 Main Agent 的协调风格

---

#### Q5: 你对输出质量的要求是什么？
> "What does 'good enough' look like for your deliverables?"

**质量维度：**
- **速度优先** - 快比完美重要（quick 任务）
- **平衡型** - 合理时间内合理质量（standard 任务）
- **质量优先** - 必须达到高标准（ultrabrain/deep 任务）
- **零容忍** - 错误不可接受（unspecified-high 任务）

**映射目标：** 选择 Model Fallback Chain

---

### 第二轮：深度挖掘（3-5 个问题）

#### Q6: 请举一个最近让你印象深刻的成功案例
> "Tell me about a recent win - something you were proud of."

**挖掘点：**
- 是什么让这次成功？
- 有没有 AI 可以复制的部分？
- 如果重来，哪里可以改进？

**映射目标：** 识别可自动化的成功模式

---

#### Q7: 请举一个最近的失败或挫折
> "Tell me about a recent failure or something that didn't go well."

**挖掘点：**
- 失败的根本原因是什么？
- 当时缺少什么资源或能力？
- 如果有个 Agent 能帮你，它该做什么？

**映射目标：** 识别能力缺口、设计补救 Agent

---

#### Q8: 你希望和 AI 保持怎样的关系？
> "Ideally, how would you describe your relationship with AI assistants?"

**关系类型：**
| 类型 | 描述 | 对应 Persona |
|------|------|--------------|
| **导师** | 经验丰富，指点方向 | 芒格、德鲁克 |
| **搭档** | 并肩作战，互补能力 | 费曼、乔布斯 |
| **助手** | 执行指令，处理杂事 | 快速响应型 |
| **监督者** | 检查质量，把控标准 | 德明、Uncle Bob |

**映射目标：** 选择人格原型基调

---

#### Q9: 你的团队/项目有什么特殊约束？
> "Are there any constraints or requirements specific to your team/project?"

**常见约束：**
- 合规要求（金融、医疗、政府）
- 技术栈限制（只能用某些语言/工具）
- 安全要求（代码不能离开本地）
- 预算限制（模型调用成本控制）

**映射目标：** 调整 Model Selection、添加安全检查 Agent

---

#### Q10: 3 个月后，你希望这个 Agent 团队帮你实现什么？
> "In 3 months, what would success look like for this Agent team?"

**成功指标：**
- 节省时间（每天节省 X 小时）
- 提升质量（错误率降低 X%）
- 扩展能力（能做以前做不到的事）
- 减少焦虑（不再担心某些任务）

**映射目标：** 设定验收标准、设计度量指标

## 🔍 Workflow 分析框架

### Step 1: 任务分解（Task Decomposition）

将用户 workflow 拆解为原子任务：

```
用户 Workflow
    ├─ 任务 A（高频、规则明确）→ 可自动化
    ├─ 任务 B（低频、需要创意）→ 需要 artistry Agent
    ├─ 任务 C（关键、不容出错）→ 需要 review Agent
    └─ 任务 D（复杂、多步骤）→ 需要 orchestration
```

**分析维度：**

| 维度 | 问题 | 影响 |
|------|------|------|
| **频率** | 多久做一次？ | 高频 → 值得专门优化 |
| **复杂度** | 需要多少步？ | 复杂 → 需要 orchestration |
| **创造性** | 需要多少原创思考？ | 高 → 需要 ultrabrain/deep |
| **容错性** | 出错代价多大？ | 低 → 需要 review Agent |
| **协作性** | 需要多少人参与？ | 高 → 需要 coordination |

---

### Step 2: 角色映射（Role Mapping）

将任务映射到 Agent 角色：

```
任务特征 → Agent 角色 → Persona 选择

战略规划 + 逆向思考 → 战略 Agent → 芒格
深度开发 + 自主探索 → 开发 Agent → 费曼
质量把关 + 数据驱动 → 审查 Agent → 德明
项目管理 + 目标跟踪 → 协调 Agent → 德鲁克
```

**映射规则：**

| 如果你的任务需要... | 选择这个角色 | 推荐 Persona |
|---------------------|--------------|--------------|
| 需求分析、PRD 设计 | 战略 Agent | 芒格、Naval |
| 代码实现、技术探索 | 开发 Agent | 费曼、Linus、Carmack |
| 代码审查、安全检查 | 质量 Agent | 德明、Uncle Bob、Dijkstra |
| 任务拆解、进度跟踪 | 项目 Agent | 德鲁克、安迪·格鲁夫 |
| 产品设计、用户体验 | 产品 Agent | 乔布斯、张小龙、俞军 |
| 文案创作、内容生成 | 内容 Agent | 海明威、奥格威、鲁迅 |
| 数据分析、用户研究 | 分析 Agent | 卡尼曼、丹尼尔·卡尼曼 |

---

### Step 3: 协作模式设计（Collaboration Pattern）

确定 Agent 之间的协作方式：

#### 模式 A: 流水线模式（Pipeline）
```
用户 → 芒格（规划）→ 费曼（开发）→ 德明（审查）→ 输出
```
**适合：** 标准化流程、步骤清晰、质量要求高

#### 模式 B: 并行模式（Parallel）
```
用户 → 主 Agent
         ├─→ 费曼（方案 A）
         ├─→ 芒格（方案 B）
         └─→ 乔布斯（方案 C）
              ↓
           德鲁克（整合）
```
**适合：** 需要多方案对比、创意发散

#### 模式 C: 迭代模式（Iterative）
```
用户 ↔ 费曼 ↔ 德明（循环直到通过）
```
**适合：** 质量要求高、需要反复打磨

#### 模式 D: 顾问模式（Advisory）
```
用户 → 芒格（建议）→ 用户决策 → 费曼（执行）
```
**适合：** 用户希望保持控制权、AI 提供选项

## 🎯 团队组成决策矩阵

### 核心决策维度

| 维度 | 选项 A | 选项 B | 选项 C |
|------|--------|--------|--------|
| **团队规模** | 小型（2-3 人） | 中型（4-5 人） | 大型（6+ 人） |
| **协作深度** | 浅层（主 Agent 分发） | 中层（Manager 协调） | 深层（Worker 间直接通信） |
| **响应速度** | 即时（quick 模型） | 平衡（standard） | 深度（ultrabrain） |
| **质量要求** | 够用就行 | 良好 | 极致 |

---

### 团队规模决策树

```
用户 Workflow 复杂度
    │
    ├─ 简单（1-2 个核心任务）
    │   └─ 推荐：2-3 个 Agent
    │       ├─ 主 Agent（协调）
    │       └─ 1-2 个 Worker（执行核心任务）
    │
    ├─ 中等（3-5 个任务类型）
    │   └─ 推荐：4-5 个 Agent
    │       ├─ 主 Agent（Relay）
    │       ├─ Manager Agent（协调）
    │       └─ 2-3 个 Worker（分工执行）
    │
    └─ 复杂（5+ 任务类型，多团队协作）
        └─ 推荐：6+ 个 Agent
            ├─ 主 Agent（Relay）
            ├─ Manager Agent（规划 + 协调）
            ├─ 多个 Specialist Worker
            └─ Review Agent（质量把关）
```

## 📝 示例访谈记录

### 示例 1：独立开发者（Before → After）

#### 访谈记录

**受访者：** 小王，独立开发者，维护一个开源工具库

**Q1: 典型一天？**
> "早上回 issue，下午写代码，晚上写文档。最痛苦的是代码 review，自己写的代码自己审，总是漏掉边界情况。"

**Q2: 最大痛点？**
> "1. 代码质量不稳定，经常发布后才发现 bug；2. 文档更新跟不上代码迭代；3. 一个人做所有事，没有反馈。"

**Q3: 任务分类？**
> "主要是三类：写新功能、修 bug、写文档。偶尔要做架构决策。"

**Q4: 协作模式？**
> "我希望有个搭档能帮我 review，但我没预算请真人。"

**Q5: 质量要求？**
> "代码必须稳定，文档可以稍后再完善。"

**Q8: 理想关系？**
> "像个严格的代码审查员，能指出我看不到的问题。"

**Q10: 3 个月目标？**
> "发布版本不再出现明显 bug，文档和代码同步更新。"

---

#### 团队设计方案

**诊断：**
- 核心痛点：缺少代码审查、文档维护
- 任务类型：开发（deep）+ 审查（unspecified-high）+ 文档（writing）
- 协作模式：迭代模式（开发 → 审查 → 修复）

**推荐团队（3 个 Agent）：**

| Agent | Persona | Role | Task Category | Model |
|-------|---------|------|---------------|-------|
| 主 Agent | 爱兔（默认） | Relay | - | Claude Opus |
| 费曼 | 理查德·费曼 | 开发 Worker | deep | GPT Codex |
| 德明 | W. Edwards Deming | 审查 Worker | unspecified-high | Claude Opus |

**Workflow 设计：**
```
用户请求 → 主 Agent → 费曼（开发）
                            ↓
                         德明（审查）
                            ↓
              通过 → 返回用户
              不通过 → 费曼（修复）→ 德明（再审）
```

**关键配置：**
- 德明必须严格，不通过时给出具体修改建议
- 费曼收到审查意见后必须逐条回应
- 最多 3 轮迭代，超过则升级给用户决策

### 示例 2：产品经理（Before → After）

#### 访谈记录

**受访者：** 小李，B2B SaaS 产品经理，带 3 人小团队

**Q1: 典型一天？**
> "上午开会，下午写 PRD，晚上整理需求优先级。最累的是写 PRD，要写清楚所有边界情况。"

**Q2: 最大痛点？**
> "1. PRD 写得太慢，经常延期；2. 需求评审时开发问的问题我没考虑到；3. 竞品分析没时间做。"

**Q3: 任务分类？**
> "战略类（规划、竞品分析）、执行类（PRD、原型）、协调类（开会、排期）。"

**Q4: 协作模式？**
> "我希望 AI 能帮我做研究和初稿，我来把关和决策。"

**Q7: 失败案例？**
> "上次一个功能上线后发现和另一个模块冲突，如果当时有人帮我检查关联性就好了。"

**Q10: 3 个月目标？**
> "PRD 产出速度提升 50%，需求评审时开发的问题减少 70%。"

---

#### 团队设计方案

**诊断：**
- 核心痛点：PRD 产出慢、考虑不全面、缺少竞品分析
- 任务类型：战略（ultrabrain）+ 文档（writing）+ 分析（deep）
- 协作模式：顾问模式（AI 研究 → 用户决策 → AI 执行）

**推荐团队（4 个 Agent）：**

| Agent | Persona | Role | Task Category | Model |
|-------|---------|------|---------------|-------|
| 主 Agent | 爱兔（默认） | Relay | - | Claude Opus |
| Manager | 德鲁克升级版 | 规划 + 协调 | ultrabrain | GPT-5 |
| 芒格 | 查理·芒格 | 战略 Worker | ultrabrain | GPT-5 |
| 费曼 | 理查德·费曼 | 分析 Worker | deep | GPT Codex |

**Workflow 设计：**
```
用户需求 → 主 Agent → Manager（规划）
                            ├─→ 芒格（竞品分析 + 方案设计）
                            └─→ 费曼（技术可行性分析）
                                   ↓
                              Manager（整合）
                                   ↓
                              用户决策
                                   ↓
                              芒格（PRD 撰写）
                                   ↓
                              费曼（关联性检查）
                                   ↓
                              输出 PRD
```

**关键配置：**
- Manager 负责拆解任务、协调芒格和费曼
- 芒格输出 PRD 时必须包含：背景、目标、3 个方案、推荐、风险、验收标准
- 费曼检查时必须验证：技术可行性、关联模块影响、边界情况

### 示例 3：内容创作者（Before → After）

#### 访谈记录

**受访者：** 小张，科技博主，每周产出 2-3 篇长文

**Q1: 典型一天？**
> "上午找选题、读资料，下午写作，晚上修改。最痛苦的是改稿，自己改总是改不到点子上。"

**Q2: 最大痛点？**
> "1. 选题没方向，不知道写什么；2. 初稿写得慢；3. 改稿次数太多，一篇改 5-6 遍。"

**Q3: 任务分类？**
> "研究类（选题、资料）、创作类（写作）、优化类（修改、润色）。"

**Q5: 质量要求？**
> "内容必须有深度，不能是泛泛而谈。速度可以妥协。"

**Q8: 理想关系？**
> "像个编辑，能帮我指出哪里逻辑不通、哪里可以更深入。"

**Q9: 特殊约束？**
> "必须原创，不能直接复制 AI 生成的内容，需要有自己的观点。"

---

#### 团队设计方案

**诊断：**
- 核心痛点：选题困难、写作慢、修改多
- 任务类型：研究（deep）+ 创作（artistry/writing）+ 审查（unspecified-high）
- 协作模式：并行模式（多方案）+ 迭代模式（修改）

**推荐团队（4 个 Agent）：**

| Agent | Persona | Role | Task Category | Model |
|-------|---------|------|---------------|-------|
| 主 Agent | 爱兔（默认） | Relay | - | Claude Opus |
| 芒格 | 查理·芒格 | 选题 Worker | ultrabrain | GPT-5 |
| 海明威 | Ernest Hemingway | 写作 Worker | writing | Claude Opus |
| 鲁迅 | 鲁迅 | 审查 Worker | unspecified-high | Claude Opus |

**Workflow 设计：**
```
用户方向 → 主 Agent → 芒格（3 个选题方案 + 推荐理由）
                            ↓
                        用户选择选题
                            ↓
                        海明威（初稿）
                            ↓
                        鲁迅（审查：逻辑 + 深度 + 原创性）
                            ↓
              通过 → 返回用户
              不通过 → 海明威（修改）→ 鲁迅（再审）
```

**关键配置：**
- 芒格选题时必须分析：受众痛点、竞品覆盖情况、差异化角度
- 海明威写作遵循：简洁有力、短句、一个段落一个观点
- 鲁迅审查标准：逻辑是否自洽、观点是否有深度、是否有抄袭风险

## 📄 Team Design Document 模板

```markdown
# Team Design Document - [项目名称]

## 1. 用户画像
- **角色：** [开发者/产品经理/内容创作者等]
```
