# 混合架构设计文档

## 概述

大明核心系统采用**古代中国官僚体系与现代软件架构**的混合设计，将传统三省六部制度与现代微服务、事件驱动架构相结合，形成独特的混合架构模式。

## 架构原则

### 1. 职责分离原则
- **三省**：决策、审议、执行三层分离
- **六部**：按功能领域划分职责
- **锦衣卫**：独立监察，不参与具体业务

### 2. 事件驱动原则
- 所有操作都通过事件触发
- 事件在各部门间流转
- 异步处理，提高系统吞吐量

### 3. 数据完整性原则
- 所有操作都有完整记录
- 支持反查审计
- 数据不可篡改

## 架构层次

### 第一层：决策层（中书省）
- **职责**：接收圣旨，创建任务档案
- **技术实现**：API网关 + 任务编排
- **数据模型**：TaskDraft

### 第二层：审议层（门下省）
- **职责**：审核任务，分配资源
- **技术实现**：审批工作流 + 资源管理器
- **数据模型**：AuditRecord

### 第三层：执行层（尚书省 + 六部）
- **职责**：协调六部，执行任务
- **技术实现**：调度器 + 微服务
- **数据模型**：DispatchRecord

### 第四层：监察层（锦衣卫）
- **职责**：巡查系统，发出预警
- **技术实现**：监控代理 + 告警系统
- **数据模型**：PatrolReport

## 六部职责映射

### 户部 (HuBu) - 资源管理
- **现代对应**：资源管理器 + 配额系统

- **职责**：权限验证、安全审计、威胁检测

### 刑部 (XingBu) - 审计监察
- **现代对应**：审计系统 + 合规检查
- **职责**：操作审计、违规检测、处罚执行

### 吏部 (LiBu) - 人事管理

### 礼部 (LiBu) - 规范制定

## 数据流设计

### 正向流程（任务执行）
```
圣旨 → 中书省 → 户部 → 门下省 → 工部 → 民夫团队 → 结果
```

### 逆向流程（审计反查）
```
结果 → 工部 → 门下省 → 户部 → 中书省 → 原始圣旨
```

### 监察流程（实时监控）
```
系统状态 → 锦衣卫巡查 → 预警 → 皇上
```

## 技术实现

### 核心组件
1. **事件总线**：基于消息队列的事件传递
2. **状态机**：任务状态流转管理
3. **数据湖**：统一数据存储和查询
4. **监控代理**：分布式系统监控

### 通信协议
- **内部通信**：gRPC + Protobuf
- **外部通信**：REST API + JSON
- **事件通知**：WebSocket + SSE

### 数据存储
- **关系型数据库**：PostgreSQL（核心数据）
- **文档数据库**：MongoDB（日志和审计）
- **缓存层**：Redis（会话和状态）
- **文件存储**：MinIO（文件和媒体）
