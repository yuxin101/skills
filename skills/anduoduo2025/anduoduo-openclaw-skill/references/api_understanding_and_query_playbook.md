# 安多多平台 API 理解、数据建模与查数路径推演

## 1. 文档定位与总体判断
该 API 本质上不是通用 CRUD 接口集，而是一套围绕**云资产、风险、规则、合规、扫描任务**构建的安全数据分析接口。

从数据分析视角看，它最适合承载三类任务：
1. **风险总览与分层钻取**：从聚合总览到风险明细，再到单条风险详情；
2. **资产与账号排查**：从云厂商、云账号、资产分类、资产列表逐步下钻；
3. **合规框架分析与报告流转**：从合规框架枚举到摘要，再到异步报告生成与状态轮询。

默认认证方式不是 Bearer，而是直接在请求头中传入：
- `Authorization: <API_KEY>`

统一响应采用：
- `code`
- `msg`
- `data`
- `requestId`
- `timestamp`

因此在所有 SOP 中，都应把“HTTP 状态 + 业务 code”联合判定作为默认规则，不能只看 HTTP 200。

## 2. 数据域建模

### 2.1 核心实体
可以把接口抽象成以下实体：
- **CloudType（云厂商）**：系统支持的云平台类型
- **CloudAccount（云账号）**：某个用户绑定的云账号
- **Asset / CloudInstance（资产）**：云平台实例、资源对象
- **Rule（规则）**：检测逻辑、风险定义、合规映射
- **RiskMapping（风险映射）**：某资产命中了某规则后形成的风险记录
- **ComplianceFramework（合规框架）**：规则归属的合规标准集合
- **ScanTask / ReportTask（任务）**：扫描或报告生成的异步任务

### 2.2 关键关系
1. 一个云账号属于一个云厂商；
2. 一个资产属于某个云账号、某个云厂商、某种实例类型、某个分类；
3. 一条风险记录本质上是“资产 × 规则”的映射结果；
4. 一个规则会映射到严重度、类型、标签、基础原因、合规框架；
5. 合规摘要是规则在某合规框架下的统计投影；
6. 报告导出是基于合规框架集合触发的异步任务。

## 3. 查数任务的第一性原理拆解
一个标准查数请求，通常可分解为：
- **指标**：查数量、通过率、严重度分布、风险数、规则数
- **维度**：云厂商、云账号、实例类型、规则类型、规则标签、规则 ID、基础原因、合规框架
- **时间/版本**：扫描任务时间、统计版本时间、实例版本
- **粒度**：总览、聚合桶、分页明细、单条详情
- **过滤**：按严重度、规则类型、规则 ID、账号、云厂商等约束
- **下钻**：从聚合结果定位到明细，再到个体对象
- **追溯**：由风险回溯资产、规则、任务与报告

## 4. 默认查数主干（建议优先级最高）

### 4.1 风险总览 -> 下钻 -> 明细 -> 详情
这是默认查数需求下最优先的主干链路：
1. 先用聚合接口获取风险总览；
2. 再选择一个维度（严重度 / 规则类型 / 云账号 / 规则 ID）做下钻；
3. 用风险分页接口承接筛选后的明细；
4. 取明细中的 `riskMapping.id` 查询单条风险详情；
5. 需要解释规则时，再补查规则列表或使用明细中携带的规则字段。

适合回答的问题：
- 当前风险总体分布如何？
- 哪类风险最多？
- 哪个云账号或实例类型最突出？
- 某个高风险规则具体落在哪些资产上？
- 某条风险的实例、规则、状态、消息是什么？

### 4.2 资产总览 -> 资产清单 -> 关联风险
适合从资产面排查：
1. 云厂商/云账号枚举；
2. 资产分类树查询（适合面板/图谱入口）；
3. 资产分页查询拿到具体资产；
4. 结合资产 ID、实例 Hash ID 或实例类型回查风险分页。

适合回答的问题：
- 某账号下有哪些资产类型？
- 某类资产数量和分布如何？
- 某台实例当前有哪些风险？

### 4.3 合规框架 -> 摘要 -> 报告异步轮询
适合从合规视角查数：
1. 先查合规框架列表；
2. 选择一个或多个框架查摘要；
3. 需要正式下载物时发起合规报告导出；
4. 通过报告详情接口轮询状态并获取下载地址。

适合回答的问题：
- 某框架下总规则数、通过数、通过率如何？
- 多个框架哪个通过率更低？
- 是否需要导出正式合规报告给外部使用？

## 5. 推荐的分析型接口分层

### A. 探针与前置枚举层
- `GET /cloudType/list`
- `GET /cloudAccounts/list`
- `GET /riskSeverity/list`
- `GET /ruleType/list`

用途：
- 连通性验证
- 枚举值收集
- 为后续筛选参数提供合法输入集合

### B. 总览聚合层
- `POST /group-by/severity`
- `POST /group-by/rule-type`
- `POST /group-by/instance-type`
- `POST /group-by/basic-reason`
- `POST /group-by/rule-label`
- `POST /group-by/rule-id`
- `POST /group-by/account-id`

用途：
- 做管理看板
- 决定下钻方向
- 找到最高风险维度

### C. 分页明细层
- `POST /risks/list`
- `POST /assets/list`
- `POST /rules/list`

用途：
- 生成明细表
- 承接筛选与分页
- 作为 HTML 报告中的主要明细数据来源（默认最多展示前 300 条）

### D. 单体详情层
- `GET /risks/{riskId}`
- `GET /scans/{taskId}`
- `GET /reports/{reportId}`

用途：
- 对某个对象做深入解释
- 补全单条记录上下文
- 追踪异步任务状态

### E. 异步任务/扩展能力层
- `POST /scans`
- `POST /reports/compliance`
- `POST /cloudAccounts/add`

用途：
- 发起扫描
- 发起报告导出
- 新增账号

默认查数不应把这些接口作为主干起点，但可在需要“刷新数据”或“导出正式文件”时作为扩展路径。

## 6. 首批候选 SOP（待真实接口验证后再决定是否标记为已验证）

### SOP-A：风险管理总览查数
目标：快速得到当前风险全貌并找出最值得排查的方向。
建议路径：
1. 云厂商或云账号枚举；
2. 严重度聚合；
3. 规则类型聚合；
4. 账号维度聚合；
5. 挑选最高风险桶进入 `risks/list`。

### SOP-B：高风险规则下钻查数
目标：从高风险总数定位到具体规则和受影响资产。
建议路径：
1. 严重度聚合或规则 ID 聚合；
2. 选择某个规则 ID；
3. `risks/list` 按 `ruleId`、`severities` 过滤；
4. 对单条风险调用详情接口。

### SOP-C：资产暴露面排查
目标：从资产角度识别高风险资产簇。
建议路径：
1. 资产分类树；
2. 资产分页；
3. 选择实例类型、账号或资产 ID；
4. 回查风险列表；
5. 必要时补充规则列表说明风险含义。

### SOP-D：合规框架诊断
目标：对合规框架进行摘要对比并产出报告。
建议路径：
1. 合规框架列表；
2. 合规摘要；
3. 若需要对外交付，发起报告导出；
4. 报告状态轮询至完成。

## 7. 真实验证时的重点检查项
1. 探针接口是否可达、认证是否稳定；
2. 聚合接口是否接受空请求体或需要显式传筛选项；
3. 分页接口在 `page/pageSize` 变化时是否满足基本一致性；
4. 过滤条件（严重度、规则类型、规则 ID、云账号）是否能让结果形成子集关系；
5. `risks/list` 中的 `riskMapping.id` 是否确实可用于 `risks/{riskId}`；
6. 合规摘要与报告导出是否都要求非空 `standardAndRegulationsIds`；
7. 真实返回中是否含敏感字段，需要在技能示例中统一 mock 化。

## 8. 报告生成建议（写入技能时使用）
在默认查数需求下，应同步生成 HTML 报告。

若任务属于风险总览、风险资产概览、整体风险情况、高风险分布等概览性查询，必须固定使用 `assets/anduoduo_risk_report.html` 作为 HTML 彩页报告骨架。固定的是结构、样式分区和图表位置，不是模板里的静态示例值；生成报告时必须用当前查询结果替换模板中的示例统计、案例、日期和图表数据。

概览类固定模板报告至少应包含：
- 指标总览卡片
- 风险严重度分布图
- 规则类型分布图
- 关键文本结论
- 明细数据表（若需要展示原始明细，默认仅展示前 300 条）

若数据支持，应优先补充：
- 受影响实例类型 TOPN
- 云账号维风险分布
- 高风险规则 TOPN
- 风险根因/主因分布
- AI 洞察：总体态势、0 值严重度解释、主导风险类型、重点对象和优先修复建议

展示边界：
- 严重度必须完整展示“严重 / 高危 / 中危 / 低危”四档，即使某一档为 0；
- 类型分布中的 `Issue` / `issues` / `issus` / `问题` 应统一对外显示为 `风险`；
- 合规正式导出文件保持其专用导出链路，不要求强制改造成该模板。

## 9. 当前结论
从文档层面判断，这套 API 非常适合构建“自上而下”的查数技能：
- 先枚举与探针
- 再做总览聚合
- 再做维度下钻
- 再拿分页明细
- 最后补详情或导出报告

后续必须通过真实请求验证以上路径中哪些可以进入“已验证 SOP”，哪些只能保留为“候选 SOP”。
