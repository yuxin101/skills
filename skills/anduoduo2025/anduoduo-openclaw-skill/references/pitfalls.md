# 安多多平台查数避坑说明

## 1. 不要假设分页结果稳定
在真实环境中，`pageSize=5` 返回的前 5 条记录并不是 `pageSize=10` 第一页前 5 条记录的稳定前缀。

这意味着：
- 不要把不同 `pageSize` 下的首屏记录当作同一序列的稳定截断；
- 如果需要比较不同页大小的结果，应按集合意义或 ID 去重后的交集关系来理解，而不是按顺序比较。

## 2. 不要默认账号过滤可靠
在锚点验证中，按已知账号 ID 过滤风险列表虽然请求成功，但结果既未包含锚点风险，也不能保证返回记录全部匹配该账号。

这意味着：
- `cloudAccountId` 不能直接作为默认风险定位主键；
- 如果用户提出“按账号查风险”，应先通过聚合接口做账号维总览，再谨慎落到明细验证，而不是直接承诺结果严格受控于该字段。

## 3. 不要从任意资产直接正向反查风险
使用随机资产 ID 回查风险时，当前样本出现了返回为空的情况。

这意味着：
- “资产一定有风险”不是可默认成立的前提；
- 更稳妥的方式是先从风险明细中获得锚点资产，再反推该资产的风险集合。

## 4. 不要把 HTTP 200 当作业务成功
所有接口都必须同时检查 HTTP 状态码和响应体中的 `code` 字段。

这意味着：
- HTML 报告或文本结论中，应记录业务失败信息，而不是只根据网络层状态做成功判断；
- 对异步任务类接口尤其要防止“请求提交成功”被误判为“结果生成完成”。

## 5. 不要在技能示例中泄露真实数据
当前真实返回已经暴露出账号名称、实例名称、规则描述等业务信息。

这意味着：
- 技能包中所有示例必须用 mock 假数据；
- 原始验证证据只能保留在内部工作目录中，不应进入最终对外交付压缩包。

## 6. 不要一开始就做重查询
真实环境下宽范围大页查询存在超时边界。

这意味着：
- 应优先采用“小页、窄范围、逐步下钻”的工作方式；
- 在默认场景下，聚合接口用于发现问题，明细接口用于解释问题，而不是用明细接口承担全局扫描职责。
