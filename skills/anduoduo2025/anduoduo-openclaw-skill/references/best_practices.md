# 安多多平台查数最佳实践

## 1. 默认查询节奏
面向默认查数需求时，优先采用“探针与枚举 → 总览聚合 → 小页明细 → 单条详情”的顺序。

推荐原因：
1. 真实环境已验证 `cloudType/list`、`cloudAccounts/list`、`riskSeverity/list`、`group-by/severity`、`risks/list`、`risks/{riskId}` 可组成稳定主干；
2. 直接大页宽范围查询存在超时风险；
3. 先聚合后明细更适合生成 HTML 报告中的总览、透视和解释层。

## 2. 明细查询的推荐节奏
对分页接口默认使用小页探测，建议从 `page=1`、较小 `pageSize` 开始，再根据需要逐步扩大。

推荐原因：
- 真实环境中，大页宽范围风险查询出现过超时；
- 小页明细在风险列表中已稳定返回，可满足定位、抽样和报告明细截取；
- 默认明细展示需求最多 300 条，因此没有必要一开始就追求大页或全量。

## 3. 已验证有效的风险过滤键
在风险列表场景下，以下过滤方式已经过真实环境验证，可优先使用：
- `severities`
- `ruleId`
- `cloudInstanceId`
- `instanceHashId`

使用方式建议：
1. 先用总览接口确定严重度或规则方向；
2. 再用 `ruleId` 或 `severities` 收窄到风险明细；
3. 若已有锚点风险对应的资产标识，可进一步使用 `cloudInstanceId` 或 `instanceHashId` 聚焦单资产风险集合；
4. 从风险明细中提取 `riskMapping.id`，最后调用详情接口补全单条风险上下文。

## 4. 详情查询主键
风险详情查询时，优先使用 `riskMapping.id` 作为详情主键。

推荐原因：
- 真实环境已验证从 `risks/list` 中提取的 `riskMapping.id` 可以稳定命中 `risks/{riskId}`；
- 不要使用规则 ID、资产 ID 或其它近似字段代替详情主键。

## 5. HTML 报告的默认组织方式
默认查数需求下，建议同步生成 HTML 报告。

其中，概览类风险查询应固定使用 `assets/anduoduo_risk_report.html` 作为版式骨架。固定的是结构与图表位置，不是模板中的示例数据。

报告至少应包含：
- 指标总览卡片
- 风险严重度分布图
- 规则类型分布图
- 关键文本结论
- 明细数据表（若需要展示原始明细，默认最多展示前 300 条）

若数据支持，优先补充：
- 受影响实例类型 TOPN
- 云账号维风险分布
- 高风险规则 TOPN
- 风险根因/主因分布

展示要求：
- 严重度必须完整展示四档，即使某一档为 0；
- `Issue` / `issues` / `issus` / `问题` 在最终报告中统一显示为 `风险`；
- 总览来自 `group-by/*`；明细来自 `risks/list`；单条解释来自 `risks/{riskId}`。

## 6. 数据示例与安全边界
在技能中的示例输出、示例报告、字段样例中，只能使用 mock 假数据。

原因：
- 真实环境返回中已观察到账号名、实例名、规则描述、参考文档链接等真实业务信息；
- 验证证据仅用于内部确认，不应直接进入面向使用者的技能示例。
