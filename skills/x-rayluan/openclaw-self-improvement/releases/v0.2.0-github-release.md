# OpenClaw Self-Improvement v0.2.0

**OpenClaw Self-Improvement v0.2.0** turns self-improvement from a vague idea into an operational loop.

This release is for OpenClaw / ClawLite operators and multi-agent teams that want to reduce repeated failures in a durable, inspectable way.

## Highlights

### New in v0.2.0
- **Experiment mode** for repeated failures
- **Binary eval loops** for testing new guardrails, SOPs, and workflow changes
- **Keep / partial_keep / discard** decision model
- **Practical examples** for:
  - Mission Control summary quality
  - deploy closeout / production verification
- **Experiment summary helper** for surfacing unresolved follow-up debt
- **Decision rules** for when to log, experiment, or promote
- **Daily routine integration** with heartbeat, Karen QA, AGENTS, and self-improving ops policy

## Why this release matters

Many agent systems can log lessons.
Far fewer can test whether a new rule, checklist, or guardrail actually reduced the repeated failure.

v0.2.0 adds a lightweight eval-driven improvement loop:
1. capture the repeated problem
2. define a baseline
3. change one thing at a time
4. evaluate with binary checks
5. classify the result as `keep`, `partial_keep`, or `discard`
6. promote proven improvements into durable operating rules

This makes self-improvement more operational and less hand-wavy.

## What problems this skill is built for

OpenClaw Self-Improvement is especially useful when your agent workflows keep repeating problems like:
- fake-complete states
- missing receipts or weak proof bundles
- back-end fixes that never reach operator-facing truth surfaces
- code-ready or build-ready states being confused with production-ready states
- new rules being added without evidence that they actually improve outcomes

## Why `partial_keep` matters

One of the most important additions in this release is the idea that a fix can be **directionally correct but still incomplete**.

Example:
- summary JSON becomes better
- but the front-end still fails to render the new fields

That should not be treated as “done.”
It should be tracked as `partial_keep` until the paired fix lands.
