# Plugin Layout and Registration

This reference describes the recommended shape for OpenClaw plugins, especially when developing inside the OpenClaw repo. The goal is stability, clarity, and easy future extension.

## Recommended directory layout

```text
extensions/<plugin-id>/
├── package.json
├── openclaw.plugin.json
├── index.ts
├── api.ts
├── runtime-api.ts            # optional; add only if runtime boundaries get complex
├── commands/                 # slash/native commands
├── hooks/                    # typed hooks or custom hooks
├── skills/                   # plugin-shipped skills
├── tools/                    # agent tools
├── src/                      # shared state, helpers, runtime logic
└── *.test.ts                 # scoped tests
```

`extensions/observability-lab/` is the best example to study directly.

## File responsibilities

### `package.json`

At minimum:

```json
{
  "name": "@openclaw/<plugin-id>",
  "type": "module",
  "openclaw": {
    "extensions": ["./index.ts"]
  }
}
```

Key rules:

- Keep runtime dependencies in the plugin's own `dependencies`.
- Do not casually move plugin runtime dependencies into the repo-root `package.json`.
- Do not use `workspace:*` in runtime dependencies.

### `openclaw.plugin.json`

Minimum required fields:

- `id`
- `configSchema`

Common optional fields:

- `name`
- `description`
- `skills`
- `uiHints`

If the plugin ships skills, declare the skills directory explicitly:

```json
{
  "id": "my-plugin",
  "skills": ["./skills"],
  "configSchema": {
    "type": "object",
    "additionalProperties": false,
    "properties": {}
  }
}
```

### `index.ts`

The entrypoint should assemble the plugin, not hold large blocks of business logic.

Recommended shape:

```ts
export default definePluginEntry({
  id: "my-plugin",
  name: "My Plugin",
  description: "Short summary",
  register(api) {
    api.registerTool(...);
    api.registerCommand(...);
    api.on(...);
  },
});
```

### `api.ts`

Use a local barrel to anchor official SDK imports. This keeps plugin code clean and prevents deep or unstable imports.

Forward only the official surfaces you actually need, for example:

- `openclaw/plugin-sdk/plugin-entry`
- `openclaw/plugin-sdk/plugin-runtime`
- other required official subpaths

## How to choose a registration API

### Tool

- Use `api.registerTool(tool, opts?)`
- `optional: true` means the tool is optional and must be explicitly allowed
- Without `optional`, the tool is registered as required at the registration layer, but runtime tool policy may still filter it out

### Command

- Use `api.registerCommand(def)`
- Best for deterministic plugin logic
- Commands do not go through model judgment
- If the real goal is “inject guidance or workflow,” prefer a plugin-shipped skill before adding a command

### Hook

- Use `api.on(hookName, handler, opts?)` for typed hooks
- Prefer typed hooks for observing or shaping prompt, model, tool, transcript, or session lifecycle behavior
- Only reach for `api.registerHook(...)` when you truly need a lower-level or more custom hook seam

### Plugin skill

- Put the skill directory under `skills/`
- Declare it in `openclaw.plugin.json`
- Skill command entrypoints are generated by OpenClaw, not manually registered in `index.ts`
- `user-invocable: true` exposes a skill command, but visibility in a specific surface still depends on that surface and on `commands.nativeSkills`

## Common boundary confusions

### `api.registerCommand(...)` and a plugin skill are not the same thing

- `api.registerCommand(...)` creates a plugin command that runs your handler directly
- a plugin skill is a packaged skill asset that enters model context and may also expose a skill command

### Skill commands are not tool dispatch by default

- By default, a skill command is rewritten into a normal agent request that uses the skill
- Only `command-dispatch: tool` creates a deterministic command → tool path

### “Registered” does not mean “available right now”

Tools and skills can still be affected by:

- tool policy
- skill snapshot behavior
- current session state
- current surface capabilities

## Import rules

Required guardrails:

- Import only from official `openclaw/plugin-sdk/<subpath>` surfaces
- Route shared plugin helpers and types through local `./api.ts` or `./runtime-api.ts`
- Do not import core `src/**` directly from plugin production code
- Do not cross from one plugin into another plugin by relative path

## Naming guidance

- Prefer aligning the folder name with the plugin id: `extensions/<id>`
- Keep `openclaw.plugin.json:id` aligned with the directory id
- Prefer package names like `@openclaw/<id>` or approved suffix variants
- Avoid tool, command, or skill names that collide with built-in host behavior

## Good in-repo examples

- `extensions/observability-lab/`

- `extensions/open-prose/`
