# Upgrade Rules - SOP-02-Lite 升级规则

## 概述

SOP-02-Lite 适用于快速、低风险的简单任务。当任务超出 lite 范围时，必须升级到 SOP-01 进行完整流程管理。

## 升级判断标准

满足以下 **任一** 条件时，提示用户升级到 SOP-01：

### 1. 时间维度
- **预计耗时 > 20 分钟**
  - 包含准备、执行、验证的总时间
  - 示例：需要编写复杂脚本、多轮调试

### 2. 系统维度
- **涉及 ≥ 2 个系统/服务**
  - 示例：同时修改数据库和应用配置
  - 示例：需要协调多个微服务

### 3. 操作维度
- **需要发布/重启操作**
  - 示例：修改生产环境代码
  - 示例：重启服务器/服务
  - 示例：部署新版本

### 4. 确认维度
- **需要多轮人工确认**
  - 示例：需要多个 stakeholder 审批
  - 示例：需要测试团队验证
  - 示例：需要安全团队审核

### 5. 范围维度
- **需要跨 gateway 操作**
  - 示例：同时操作开发和生产环境
  - 示例：需要跨云平台操作
  - 示例：涉及外部第三方系统

## 升级流程

### 1. 立即停止
当检测到任务超出范围时：
- 不再继续执行新操作
- 保持当前状态不变

### 2. 标记状态
```bash
python3 scripts/update_state.py \
  --instance-path [实例路径] \
  --status UPGRADED \
  --stage [当前阶段] \
  --reason "升级原因：[具体说明]"
```

### 3. 通知用户
使用标准话术：
> ⚠️ **检测到任务超出 SOP-02-Lite 范围**
>
> **升级原因**: [具体原因]
>
> **当前进度**: [已完成的工作]
>
> **建议操作**: 创建 SOP-01 实例继续执行
>
> 已将当前实例标记为 UPGRADED 状态，可以在 SOP-01 中引用此实例的记录。

### 4. 记录详情
在 LOG.md 中详细记录：
- 触发升级的具体条件
- 当前已完成的操作
- 升级时间点
- 建议 SOP-01 的后续步骤

### 5. 引导升级
提供下一步指引：
```
如需创建 SOP-01 实例，请提供：
1. 任务的详细背景
2. 当前已完成的工作
3. 预计后续步骤
```

## 判断示例

### 示例 1: 不需要升级
**任务**: 修改某个服务的日志级别（DEBUG → INFO）
- **评估**: 
  - 预计耗时：5 分钟 ✓
  - 涉及系统：1 个 ✓
  - 无需重启 ✓
  - 单次确认 ✓
- **结论**: 继续使用 SOP-02-Lite

### 示例 2: 需要升级（时间）
**任务**: 优化数据库查询性能
- **评估**:
  - 预计耗时：需要分析、测试、验证，约 1 小时 ✗
  - 涉及系统：数据库 + 应用 ✗
- **结论**: 升级到 SOP-01

### 示例 3: 需要升级（操作）
**任务**: 修改生产环境配置并重启服务
- **评估**:
  - 需要重启服务 ✗
  - 影响生产用户 ✗
- **结论**: 升级到 SOP-01

### 示例 4: 需要升级（确认）
**任务**: 调整 API 限流策略
- **评估**:
  - 需要产品确认 ✓
  - 需要测试验证 ✓
  - 需要运维审核 ✓
  - 多轮人工确认 ✗
- **结论**: 升级到 SOP-01

## 特殊情况

### 不确定时
如果无法确定是否需要升级：
1. **保守原则**: 倾向于升级
2. **咨询用户**: 向用户说明情况，由用户决定
3. **记录原因**: 无论选择哪种，都记录判断依据

### 中途发现
如果在执行过程中发现超出范围：
1. **立即停止**：不再继续执行
2. **评估风险**：评估已执行操作是否需要回滚
3. **完整记录**：记录所有已执行的操作和结果
4. **建议升级**：提供完整的上下文给 SOP-01
