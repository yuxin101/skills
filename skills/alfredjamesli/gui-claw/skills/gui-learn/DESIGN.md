# gui-learn 设计文档

> 最后更新：2026-03-24

## 核心原则

**每个 app/网站都要学。学过一次后模板匹配比重新检测快 10 倍以上。**

## 学习 vs 检测

| | 学习（learn） | 检测（detect） |
|---|---|---|
| 何时调用 | 第一次操作某 app/页面 | 每次操作前 |
| 做什么 | 检测 + 裁剪 + 命名 + 保存到 memory | 模板匹配已保存的组件 |
| 耗时 | ~4s（GPA + OCR + 裁剪 + 写文件） | ~0.3s（模板匹配） |
| 输出 | components.json + 组件图片 | 匹配到的组件名 + 坐标 |

学习的投入是一次性的，之后的所有操作都受益于模板匹配的速度。

## 组件命名策略

检测到的元素需要命名才能保存。命名优先级：

1. **OCR 文字**：如果元素本身有文字（"Search"、"Book"）→ 直接用文字作名字
2. **最近文字**：如果是图标，找最近的文字标签（60px 范围内）→ 用那个文字
3. **位置命名**：都找不到 → `unlabeled_{x}_{y}`

### 为什么不让 LLM 命名

早期尝试过把裁剪的组件图片发给 image tool 让 LLM 命名。问题：
- 每个组件一次 API 调用，50 个组件 = 50 次调用
- 慢（~5s/个）且贵
- LLM 命名不稳定（同一个按钮可能叫 "search_btn" 或 "magnifying_glass"）

现在：自动命名后，unlabeled 的组件在操作中遇到时 LLM 可以补充命名（rename 命令）。

## 去重机制

### 位置去重

新组件和已有组件坐标差 < 15px → 认为是同一个，用旧名字。

为什么 15px：UI 元素在不同状态下可能微移几像素（hover 效果、动画），但不会大幅移动。15px 能容忍这种微调。

### 视觉去重

`is_duplicate_icon()` 用模板匹配（阈值 0.92）比较新裁剪的组件图片和已保存的。

为什么 0.92：低于这个值的"重复"可能其实是外观相似但功能不同的组件（比如两个不同的蓝色按钮）。

## 过滤规则

- w < 25 或 h < 25 → 跳过（太小的元素通常是噪声或装饰性的分隔线）
- 长文本 > 15 字符 → 可能是内容文字而不是 UI 标签（`_find_nearest_text` 中过滤）

## 浏览器场景

浏览器学习分两层：
1. **浏览器级**：toolbar、设置页面 → `memory/apps/chromium/`
2. **网站级**：每个网站独立 → `memory/apps/chromium/sites/<domain>/`

`learn_from_screenshot(img_path, domain="united.com")` 通过 domain 参数决定存到哪里。

## auto_cleanup_dynamic

学习后自动清理动态内容（时间戳、聊天消息、通知数字）。

这个函数主要针对聊天应用（WeChat 等），对网站效果有限。网站的动态内容（广告、推荐）更依赖遗忘机制自动淘汰。
