# Adapted notes from surveyed skills

这份文件记录从外部 skill 吸收后的可复用点，只保留方法，不保留对外部脚本或外部生态的强依赖。

## 1. 来自 codebase-analysis

可借鉴：
- 在做非 trivial 判断前，必须先理解代码库，而不是直接规划
- 适合把分析拆成：探索 → 聚焦 → 深读 → 验证 → 综合
- 输出结论时应带证据，而不是只给印象

不采用：
- “一触发就立刻跑脚本”的强依赖模式
- 把流程绑定到固定 Python orchestrator

落地到我们的版本：
- 保留“先证据后结论”
- 改成工具无关流程，适配本地 repo / GitHub / 任意语言栈

## 2. 来自 codebase-recon

可借鉴：
- 明确区分 direct observation / docs / tests / history / inference
- 在分析过程中追踪 confidence
- 发现、模式、缺口、下一步，这四项很适合中间态汇报

不采用：
- 过重的显式 confidence bar 作为固定输出要求
- 对其他 micro-skills 的联动依赖

落地到我们的版本：
- 保留“事实 / 推测 / 待验证”分层
- 用更符合工程沟通的表达替代复杂置信度条

## 3. 来自 architecture-review

可借鉴：
- 先测量，再改结构
- 模块边界、依赖方向、循环依赖、god module、深层嵌套，都是结构分析关键点
- 适合把问题按严重度分类

不采用：
- 过于前置地把任务引向重构方案
- 只偏 TS/JS 架构审查的表述

落地到我们的版本：
- 强调模块边界、主链路、复杂度热点
- 默认先做“理解和评估”，不是直接建议重构

## 4. 来自 repo-research-analyst

可借鉴：
- 先看 README / CONTRIBUTING / ARCHITECTURE / CLAUDE 这类高信号文件
- 先识别 stack，再识别目录和入口点
- 适合产出接手 handoff 文档

不采用：
- 固定要求把结果写入 handoff 目录
- 过于模板化的仓库研究汇报

落地到我们的版本：
- 保留“最小高信号阅读集”
- 保留“给新接手工程师的阅读顺序和注意事项”

## 5. 来自 github-explorer

可借鉴：
- GitHub 项目分析不应只看 README
- 在需要时，可以补 repo 健康度、issue 质量、社区讨论、竞品视角
- 强调输出可追溯，不要空泛评价

不采用：
- 对 search-layer / content-extract / MinerU 的依赖
- 过重的外部社区情报模板作为默认输出

落地到我们的版本：
- 把“GitHub 外部信号”设为可选增强层，而不是默认必做项
- 默认优先做工程向仓库分析；当用户明确要“这个项目怎么样”时，再补项目健康度和 adoption 视角

## 6. 来自 github-deep-research

可借鉴：
- 研究可以分轮：官方信息优先，再逐步扩展到外部来源
- 强调 inline citation 和来源分级
- 对 timeline / release / issue / commit 维度的分析很有价值

不采用：
- 固定四轮深研究工作流
- 大而全报告作为默认输出

落地到我们的版本：
- 官方 repo / 官方文档优先
- 深研究只在用户明确要求时启用

## 汇总：我们自己的 repo-analysis 应该坚持什么

1. **工程优先**
   目标是帮助人更快接手、评估、修改项目，不是写百科。

2. **证据优先**
   优先使用仓库直接证据：目录、清单、入口、实现、测试、部署文件。

3. **先广后窄**
   先扫 repo 形态和 stack，再读高信号文件，再追主链路。

4. **明确边界**
   输出里要区分：
   - 已确认事实
   - 推测
   - 需要继续验证的点

5. **默认轻量，按需升级**
   默认给工程速读；如果用户要更深，再升级到架构版、接手评审版、GitHub 项目研判版。
