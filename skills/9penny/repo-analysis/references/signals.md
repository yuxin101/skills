# Signals for Repo Analysis

## Positive signals

- README 不只是 marketing，还写了运行方式、架构、目录、配置
- 模块边界清晰，目录命名一致
- 入口文件容易定位
- 依赖管理清晰，有锁文件
- 有 migration、seed、CI、lint、test、deploy 说明
- 有 auth、RBAC、审计、可观测性等平台化能力
- 插件/适配器层边界明确，不是到处硬编码

## Risk signals

- 命名混乱，同一概念出现多个名称
- 抽象层很多，但对应实现深度不够
- README 写得很大，代码里落地不完整
- 多服务协同时缺少测试、缺少运行说明
- 关键路径过深，调试困难
- 配置项过多但约束不清晰
- 安全、权限、部署逻辑分散在多个模块中
- CE/EE、多 runtime、多插件并存，但缺少统一收敛点

## Takeover cost heuristics

### 低
- 单体项目
- 入口清晰
- 文档完整
- 测试覆盖关键路径

### 中
- 多模块但边界清晰
- 有少量历史命名包袱
- 基础设施可本地启动

### 高
- 多服务 + 多 runtime + 事件流 + 插件系统
- 平台化抽象较重
- 文档与实现存在偏差
- 缺少集成测试 / 调试手册 / 时序说明

## Questions worth answering in deeper review

- 最核心的一条主链路是什么？
- 哪些模块是业务核心，哪些只是接入层？
- 新人第一周最容易踩坑的地方在哪？
- 哪些抽象是真有价值，哪些可能是过度设计？
- 如果线上出问题，最难排查的点会在哪？
