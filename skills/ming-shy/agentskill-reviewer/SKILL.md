# Skill Reviewer - Skill 质量评审工具

对 AgentSkills 进行全面质量评审，生成结构化的中文 Markdown 报告。

## 评审流程

### 1. 输入验证
- 接收 skill 目录路径（绝对路径或相对于 workspace 的路径）
- 验证目录是否存在
- 检查 SKILL.md 是否存在

### 2. 文件读取与分析
按以下顺序读取并分析：
1. **SKILL.md** - 主文档
2. **metadata.json** (如果存在)
3. **scripts/** 目录下的脚本
4. **references/** 目录下的参考文档
5. **assets/** 目录下的资源文件

### 3. 评审维度

#### 📋 结构完整性 (Structure)
- SKILL.md 是否存在且可读
- YAML frontmatter 格式是否正确
- 必需字段：`name`、`description`
- 目录结构是否合理（scripts/、references/、assets/）
- 是否包含不必要的文件（README.md、CHANGELOG.md 等）

#### 📝 文档质量 (Documentation)
**重点评审内容，这是核心！**

##### Description 质量
- **完整性**：是否包含 skill 的功能说明 + 触发条件
- **清晰度**：描述是否让人一眼就知道这个 skill 做什么
- **触发词覆盖**：是否列举了足够的使用场景和触发短语
- **长度适中**：不应过短（<50字）或过长（>500字）
- **示例**：
  - ❌ 差："处理 PDF 文件"（太模糊）
  - ✅ 好："读取、编辑、合并、分割 PDF 文件。当用户要求操作 PDF、提取文本、旋转页面、添加水印时使用。"

##### Body 质量
- **简洁性**：是否过度冗长（建议 <500 行）
- **必要性**：每段内容是否真的是 Codex 不知道的信息？
- **结构清晰**：是否使用标题、列表、代码块合理组织
- **示例充分**：是否有足够的示例演示用法
- **冗余检查**：
  - 是否重复描述 Codex 已知的常识（如"Python 是一种编程语言"）
  - 是否过度解释显而易见的步骤
  - 是否在 SKILL.md 和 references/ 中重复同样的内容

##### 渐进式披露 (Progressive Disclosure)
- 是否将详细内容合理拆分到 references/ 目录
- references/ 文件是否在 SKILL.md 中被引用
- 是否说明了何时加载哪个 reference 文件
- references/ 内容是否应该移回 SKILL.md（太少的话）

#### 🔧 技术实现 (Technical)
- **脚本质量**：
  - 脚本是否可执行（检查 shebang、权限）
  - 脚本是否有基本的错误处理
  - 是否有注释说明用途
- **工具调用**：是否正确指导 Codex 调用工具（read、write、exec 等）
- **路径处理**：相对路径是否基于 skill 目录正确解析

#### 🎯 用户体验 (UX)
- 是否易于理解和使用
- 错误情况是否有清晰的处理指引
- 是否提供了足够的使用示例

#### ⚠️ 常见问题识别
- **冗余信息**：是否包含 Codex 已知的常识
- **过度解释**：是否过度详细地描述显而易见的步骤
- **内容错位**：
  - 触发条件写在 body 而不是 description
  - 详细文档应该在 references/ 却放在 SKILL.md
- **不必要的文件**：README.md、INSTALLATION.md 等
- **命名不规范**：skill 名称是否符合 lowercase-with-hyphens
- **缺少必要的文件**：references/ 被引用但文件不存在

### 4. 生成报告

生成结构化的中文 Markdown 报告，包含：

## 使用方式

当用户提供 skill 路径后：
1. 读取所有相关文件
2. 逐项评审每个维度
3. 识别具体问题并给出行号和建议
4. 生成完整的中文 Markdown 报告
5. 将报告保存到 workspace（如 `reviews/skill-name-review-YYYYMMDD.md`）

## 评审原则

1. **客观公正**：基于 AgentSkills 规范评审，不主观臆断
2. **具体可操作**：指出具体问题位置和修改建议
3. **重点突出**：优先指出影响使用的严重问题
4. **建设性**：不仅指出问题，还提供解决方案
5. **尊重原意**：理解作者意图，避免过度批评

## 注意事项

- **相对路径解析**：skill 内的相对路径应基于 skill 目录解析
- **行号标注**：指出问题时尽量标注具体行号
- **示例对比**：用 ❌/✅ 对比展示问题和正确做法
- **优先级分级**：P0（必须）、P1（建议）、P2（可选）
