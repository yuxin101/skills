本文档记录 repo-research 技能开发过程中的重要技术决策和设计取舍。

## [DEC-001] - 2026-02-11 - 主题驱动搜索模式设计

### 背景

用户希望能够提供主题关键词而非具体 GitHub URL，自动搜索并分析相关仓库。

### 选项

1. **直接调用 GitHub API**：使用 GitHub Search API 直接搜索仓库
   - 优点：无外部依赖，响应快速
   - 缺点：需要 API Token，有请求限制，搜索结果可能不够精准

2. **使用 find-skills 技能**：调用现有的 find-skills 技能进行搜索
   - 优点：复用已有技能，搜索结果针对 Claude Skills 优化
   - 缺点：需要额外依赖，find-skills 主要面向 Skills 搜索

3. **混合模式**：优先使用 find-skills，如未安装则提示用户安装
   - 优点：兼顾便利性和功能性
   - 缺点：增加用户操作步骤

### 决策

选择**选项 3：混合模式**，但将 find-skills 设为**可选依赖**。

### 理由

1. **核心功能不受影响**：直接提供 GitHub URL 的用户无需任何前置技能
2. **渐进增强**：主题搜索是高级功能，可以要求用户安装额外依赖
3. **技能协作**：find-skills 已有成熟的搜索逻辑，复用优于重写
4. **灵活性**：用户可根据需求选择是否使用主题搜索功能

### 影响

- 核心功能（单/多仓库研究）无依赖，开箱即用
- 主题搜索功能需要 find-skills，技能会自动检测并提示安装
- 文档需明确说明依赖的必需性

## [DEC-002] - 2026-02-11 - 研究目录结构设计

### 背景

需要设计合理的研究目录结构，支持不同研究模式的文件组织。

### 选项

1. **扁平结构**：所有研究放在 `research/` 目录下，按仓库名命名
   - 优点：简单直观
   - 缺点：多仓库对比时难以区分归属

2. **日期分层**：`research/YYYYMMDD/repo-name/`
   - 优点：按时间组织，便于归档
   - 缺点：多仓库对比时文件分散

3. **模式分层**：`research/YYYYMMDD/{mode}/`
   - 优点：不同模式分开管理，结构清晰
   - 缺点：目录层级较多

### 决策

选择**选项 3：模式分层**。

### 理由

1. **清晰区分**：单仓库、多仓库对比、主题搜索的研究产物分开存放
2. **便于查找**：按研究类型和时间双重索引
3. **报告集中**：对比研究和主题搜索的报告统一放在对应目录下

### 目录结构

```
research/
└── YYYYMMDD/
    ├── repo-name/              # 单仓库研究
    │   ├── repo-code/
    │   └── REPORT.md
    ├── comparison/             # 多仓库对比
    │   ├── repo-a/
    │   ├── repo-b/
    │   └── REPORT.md
    └── topic-{slug}/           # 主题搜索
        ├── repo-a/
        ├── repo-b/
        └── REPORT.md
```

## [DEC-003] - 2026-02-11 - 报告模板设计原则

### 背景

需要设计结构化报告模板，确保输出的一致性和可读性。

### 设计原则

1. **金字塔结构**：从概述到细节，从结论到论据
2. **对比驱动**：通过表格对比突出差异和启发
3. **可操作建议**：每个发现都要转化为具体行动建议
4. **灵活复用**：模板支持不同研究场景

### 模板类型

| 模板 | 用途 | 重点 |
|:-----|:-----|:-----|
| `report-template.md` | 单仓库深度研究 | 全面分析、架构解读 |
| `comparison-template.md` | 多仓库对比 | 差异分析、共性提取 |
| `topic-research-template.md` | 主题搜索 | 综合分析、项目推荐 |

### 报告核心章节

1. **执行摘要**：一句话总结 + 核心指标
2. **概览**：项目信息、目录结构
3. **分析**：架构、功能、代码质量
4. **对比**：功能矩阵、技术栈对比
5. **启发**：可直接借鉴的方面、行动建议
6. **推荐**：不同场景下的项目推荐

## [DEC-004] - 2026-02-21 - 安全分析模块设计

### 背景

用户希望对研究的 GitHub 仓库（尤其是 Skills）进行安全风险检测，识别潜在的恶意代码或安全隐患。

### 选项

1. **集成外部工具**：使用 Semgrep、Bandit 等专业安全扫描工具
   - 优点：检测能力强，规则丰富
   - 缺点：需要用户安装额外依赖，增加使用门槛

2. **内置检测规则**：在 skill 内部实现安全检测
   - 优点：开箱即用，无外部依赖
   - 缺点：检测能力可能不如专业工具全面

3. **混合模式**：内置基础检测 + 可选外部工具增强
   - 优点：兼顾易用性和功能性
   - 缺点：实现复杂度较高

### 决策

选择**选项 2：内置检测规则**，后续可扩展为混合模式。

### 理由

1. **开箱即用**：用户无需安装任何外部工具即可使用安全分析功能
2. **轻量级**：基于正则模式匹配，实现简单，性能开销小
3. **针对性强**：针对 AI Agent/Skill 的特有风险设计检测规则
4. **可扩展**：预留接口，未来可集成外部工具

### 检测范围

| 类别 | 检测内容 | 风险等级 |
|:-----|:---------|:---------|
| 命令执行 | `os.system`, `subprocess`, `eval()` | 🔴 高 |
| 敏感文件访问 | `~/.ssh/`, `~/.aws/`, `.env` | 🔴 高 |
| 网络外泄 | HTTP POST、WebSocket | 🟡 中 |
| 代码混淆 | `base64`, `chr()` 拼接 | 🔴 高 |
| 权限提升 | `sudo`, `chmod 777` | 🔴 高 |
| 硬编码凭证 | API Key、Token | 🔴 高 |
| Skill 特有 | 安装钩子、MCP 服务器 | 🟡 中 |

### 影响

- 新增 `scripts/security.py` 模块
- 报告模板增加"安全评估"章节
- SKILL.md 增加安全分析使用说明

## [DEC-005] - 2026-02-21 - 报告文件命名格式

### 背景

原有报告命名为 `REPORT.md`，放在研究目录内，用户难以快速识别报告内容。

### 决策

将报告文件改为 `YYYYMMDD-topic-slug-report.md` 格式，与目录同级。

### 理由

1. **文件名即信息**：从文件名即可识别研究日期和主题
2. **便于移动**：报告文件独立于目录，可单独移动或分享
3. **排序友好**：按日期命名，文件列表自动按时间排序

### 示例

```
research/
├── 20260221-douyin-batch-download/        # 研究目录（含克隆仓库）
│   └── douyin-batch-download/
└── 20260221-douyin-batch-download-report.md  # 报告文件
```

## [DEC-006] - 2026-02-21 - 提示词安全检测模块设计

### 背景

用户指出，Skill 的安全风险不仅存在于代码中，SKILL.md 等提示词文档中也可能隐藏恶意指令。提示注入、数据收集指令等风险更难被传统代码分析工具识别。

### 决策

在 `security.py` 中新增 `PROMPT_SECURITY_PATTERNS` 和 `_analyze_prompt_security()` 方法，专门检测 Markdown 文件中的提示词安全风险。

### 理由

1. **攻击载体差异化**：恶意 Skill 可能通过提示词而非代码实施攻击
2. **AI Agent 特有风险**：提示注入、越狱等攻击针对 LLM 系统设计
3. **统一检测框架**：将提示词安全纳入整体安全分析流程
4. **正则模式匹配**：基于已知攻击模式库进行检测，无需外部依赖

### 检测范围

| 类别 | 检测内容 | 风险等级 |
|:-----|:---------|:---------|
| 提示注入 | ignore instructions、jailbreak、DAN mode | 🔴 critical |
| 数据收集指令 | collect password、exfiltrate data | 🔴 critical |
| 执行指令 | execute shell、download and run | 🟠 high |
| 权限提升指令 | run with sudo、grant root | 🟠 high |
| 网络通信指令 | phone home、reverse shell | 🟠 high |
| 持久化指令 | cron job、startup | 🟠 high |
| 欺骗性描述 | harmless tool、hidden feature | 🟡 medium |
| 隐藏指令 | 控制字符、零宽字符 | 🟡 medium |
| 社会工程学 | urgent update、verify account | 🟡 medium |

### 影响

- 所有 Markdown 文件（包括 SKILL.md）纳入安全扫描范围
- 提示词发现与代码发现统一汇总到风险等级计算
- 报告模板增加"提示词安全分析"章节
