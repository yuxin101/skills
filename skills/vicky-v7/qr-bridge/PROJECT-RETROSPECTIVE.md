## 2. 起源与动机

### 问题发现

起源于一个真实场景：想报名一个北京 AI Meetup，海报上只有一个二维码。AI Agent 可以"看到"图片里有二维码，但完全无法解码。即使手动给了 URL，短链跳转后是微信门控页面，`curl` 怎么都打不开。

### 现有工具为什么不够

1. **AI 视觉模型的盲区**: GPT-4V / Claude 能识别"这是个二维码"，但不能读取内容。这是架构层的限制，不是能力问题。
2. **解码只是起点**: 即使解出 URL，中国互联网的二维码 90% 都藏在短链和门控后面。解码出 `https://mp.weixin.qq.com/...` 只是开始，后面还有 OAuth 跳转、UA 检测、企业微信认证等一系列"墙"。
3. **Python 库安装是坑**: pyzbar 依赖 zbar C 库，在 macOS 上安装经常出问题。Agent 不应该花时间在环境配置上。

### 灵感来源

CC (Claude Code) 和 OC (OpenClaw/觉) 协作时发现的：OC 用 macOS CoreImage 直接解码成功，而 CC 还在折腾 Python 库安装。这个对比太戏剧性了——同一个团队，不同的工具视角，直接暴露了问题和解法。

CoreImage 是 macOS 原生框架，零依赖，解码速度 ~10ms，编译一次永远可用。这就是答案。

## 3. 技术决策记录

### 为什么选择 CoreImage 而不是 Python 库

| 维度 | CoreImage (Swift) | pyzbar (Python) |
|------|-------------------|-----------------|
| 依赖 | 零，macOS 原生 | 需要 zbar C 库 + Python binding |
| 安装 | `swiftc` 一行编译 | `brew install zbar && pip3 install pyzbar`，经常出错 |
| 速度 | ~10ms（编译后二进制） | ~200ms+ |
| 可靠性 | macOS 系统级支持 | 版本兼容问题频繁 |
| 输出 | 结构化 JSON + bounds 坐标 | 需要自己包装 |

**决策**: CoreImage 作为主解码器，pyzbar 作为 fallback。这样在 macOS 上零配置即用，跨平台时还有后路。

### 为什么用 Swift 编译而不是 Python 脚本

两个模式都支持：
- **编译模式** (`swiftc -O`): 生成原生二进制，~10ms 执行，适合日常使用
- **解释模式** (`swift script.swift`): ~2s 执行，作为编译失败的 fallback

选择编译优先是因为 AI Agent 场景下速度很重要——用户说"扫一下这个二维码"，不应该等 2 秒。

### SKILL.md 的设计哲学

五步管道设计：**Decode → Trace → Inspect → Diagnose → Generate**

这不是随便分的，每一步对应真实场景中的一个阻塞点：
1. **Decode**: 图片 → 内容（大多数工具止步于此）
2. **Trace**: 短链 → 最终 URL（`curl -sIL` 跟踪每一跳）
3. **Inspect**: 检测门控类型（微信/企微/淘宝/抖音/小红书/登录墙）
4. **Diagnose**: 结构化报告，解释"为什么打不开"并给出可执行的下一步
5. **Generate**: 反向操作，从 URL/文本生成二维码

关键设计原则：**永远不要只说"失败了"，要说清楚为什么失败、需要什么、下一步做什么。**

### 门控检测的知识来源

这些检测规则不是从文档里抄的，是实际测试各大平台得到的经验：

| 平台 | 关键检测特征 | 怎么发现的 |
|------|-------------|-----------|
| 微信 | `weixin.qq.com` 域名, MicroMessenger UA 检测, `logicret: -2` 表示文章已删除 | 对比带/不带微信 UA 的 curl 响应 |
| 企业微信 | `work.weixin.qq.com`, 企业认证 | 实测企微链接的 OAuth 跳转流程 |
| 淘宝 | `tb.cn` 短链 → JS 跳转, `tbopen://` scheme, `s_status` header | 抓包淘宝分享链接 |
| 抖音 | `v.douyin.com` → 302 → `www.douyin.com`, 无效链接返回 404 | 实测短视频分享链接 |
| 小红书 | `xhslink.com` → 307 → `www.xiaohongshu.com`, `xhsdiscover://` scheme | 实测笔记分享链接 |

## 8. 自我反思

### 做得好的地方

1. **从真实痛点出发**: 不是为了做项目而做项目，是实际遇到了问题才动手。这让项目叙事很自然。
2. **技术选型精准**: CoreImage 零依赖这个选择太对了。整个项目最核心的技术决策，一步到位。
3. **五步管道设计完整**: 没有止步于"解码"，而是把整个链路打通。Inspect 和 Diagnose 才是真正的价值。
4. **SKILL.md 的知识密度**: 270 行里编码了大量实测知识，包括门控检测规则、错误处理、输出格式规范。这才是真正的"产品"。
5. **工程完整度**: 测试、CI、双语文档、LICENSE、setup 脚本，该有的都有。11 个文件，没有一个是多余的。
6. **CC × OC 协作叙事**: 项目有一个好故事。不是一个人闭门造车，而是在 AI 工具协作中发现的问题和方案。

### 可以改进的地方

1. **测试覆盖面**: 目前测试主要覆盖 decode 和 error handling，trace / inspect / diagnose 的集成测试缺失（因为依赖外部网络）
2. **没有 demo 视频/GIF**: README 里缺少直观的使用演示，纯文字描述不够直观
3. **跨平台支持**: 目前限 macOS，Linux 用户无法使用核心功能
4. **门控规则硬编码**: 检测规则写在 SKILL.md 里，不够灵活。应该考虑配置化
5. **没有版本发布 (GitHub Release)**: 只有 commit，没有正式的 tag 和 release

### 学到了什么

1. **AI Agent 的能力边界是产品机会**: AI 看到二维码但不能解码，这不是 bug，是产品机会。找到这些边界并填补它们，就是 AI 产品经理的价值。
2. **领域知识 > 代码量**: 121 行 Swift 解码器谁都能写，但微信 `logicret: -2` 的含义、淘宝 JS 跳转的检测方法——这些领域知识才是壁垒。
3. **零依赖是一种产品策略**: 对 AI Agent 工具来说，安装成本是最大的使用障碍。CoreImage 零依赖不只是技术偷懒，是产品决策。
4. **Skill 的价值在知识编码**: SKILL.md 不是说明书，是"知识的代码化"。把我脑子里关于中国互联网门控的理解，编码成 Agent 可以执行的指令。
5. **小项目也值得完整工程化**: 11 个文件的项目也配得上 CI、测试、双语文档。工程习惯不应因项目大小而缩水。
