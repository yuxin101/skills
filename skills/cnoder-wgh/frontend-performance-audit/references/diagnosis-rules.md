# 诊断规则

## 当 LCP 偏高时
可能原因：
- 服务端响应慢
- 首屏大图过大
- 首屏图片未预加载
- 阻塞渲染的 CSS 或 JS 过多
- 客户端渲染导致首屏内容出现过晚
- 关键资源优先级设置不合理

建议动作：
- 降低 TTFB
- 预加载首屏主图
- 压缩并按实际尺寸输出主图
- 内联关键 CSS
- 延迟非关键 JS
- 提高首屏关键资源优先级

## 当 INP 偏高时
可能原因：
- 主线程 JavaScript 负担过重
- 点击或输入事件处理函数过重
- Hydration 成本过高
- 第三方脚本阻塞交互
- 长任务过多
- 页面组件结构过于复杂

建议动作：
- 拆分长任务
- 减少不必要的 Hydration
- 延迟加载非关键组件
- 将重计算移出主线程
- 优化事件处理函数
- 延迟第三方脚本初始化

## 当 CLS 偏高时
可能原因：
- 图片未设置宽高
- 广告位或嵌入内容晚插入
- 字体切换导致布局抖动
- 首屏上方异步插入内容
- 动态组件加载时未预留空间

建议动作：
- 为图片和广告位预留空间
- 明确声明宽高
- 优化字体加载策略
- 避免在现有内容上方插入异步模块
- 为异步组件和广告模块保留占位区域

## 当 FCP 偏高时
可能原因：
- 阻塞渲染的 CSS 过多
- HTML 返回慢
- 首屏依赖脚本过多
- 首屏内容依赖客户端渲染
- 关键字体或关键样式加载过晚

建议动作：
- 精简首屏 CSS
- 内联关键 CSS
- 提前返回首屏 HTML
- 减少首屏依赖的脚本
- 优先加载关键字体和关键样式

## 当 TTFB 偏高时
可能原因：
- 服务端处理耗时高
- 缓存命中率低
- API 链路慢
- 边缘节点未生效
- 首次请求存在重定向或中间层阻塞

建议动作：
- 启用页面缓存或边缘缓存
- 优化服务端逻辑
- 缩短 API 链路
- 检查 CDN 和回源策略
- 减少不必要的重定向

## 当 DCL 偏高时
可能原因：
- HTML 体积过大
- 同步 JavaScript 阻塞解析
- 第三方脚本插入在 head 中阻塞 DOM 构建
- 过多内联脚本执行
- DOM 节点数量过多
- 首屏模板结构过深

建议动作：
- 减少 HTML 体积
- 将阻塞脚本改为 defer 或 async
- 延迟加载第三方脚本
- 减少首屏同步执行逻辑
- 精简 DOM 结构和层级
- 去掉不必要的首屏节点

## 当 TBT 偏高时
可能原因：
- JavaScript 体积大
- 初始化逻辑过重
- 大量第三方脚本执行
- 主线程长任务过多
- Hydration 范围过大
- 动画、轮播、埋点初始化集中执行

建议动作：
- 减少首屏 JavaScript 体积
- 进行代码拆分
- 延迟非关键逻辑执行
- 拆分长任务
- 缩小 Hydration 范围
- 控制第三方脚本初始化时机

## 当请求总数偏多时
可能原因：
- 静态资源拆分过碎
- 第三方资源过多
- 图片数量过多
- 历史遗留资源未清理
- 页面模块过于零散

建议动作：
- 合理合并静态资源
- 精简第三方依赖
- 减少首屏图片数量
- 清理无效资源引用
- 合理控制模块拆分粒度

## 当 JavaScript 总体积偏大时
可能原因：
- 业务代码拆分不足
- Vendor 包过大
- 重复依赖
- 引入了不必要的库
- 首屏加载了过多非关键逻辑

建议动作：
- 路由级或页面级代码拆分
- 精简依赖
- 检查重复依赖
- 用体积更小的替代库
- 将非关键功能改为懒加载

## 当 CSS 总体积偏大时
可能原因：
- 未清理无用样式
- 全量样式一次性加载
- 主题、组件库样式过重
- 页面只用到少量样式却加载了整包 CSS

建议动作：
- 清理未使用 CSS
- 拆分首屏与非首屏样式
- 优化组件库样式引入方式
- 提取关键 CSS

## 当图片总体积偏大时
可能原因：
- 首屏图片过大
- 使用了不合适的格式
- 没有按展示尺寸裁剪
- 图片数量过多
- 没有使用懒加载

建议动作：
- 压缩图片
- 优先使用 WebP 或 AVIF
- 输出与展示尺寸匹配的图片
- 为非首屏图片启用懒加载
- 控制首屏图数量

## 当长任务数量偏多时
可能原因：
- 大量同步 JavaScript 执行
- 第三方脚本阻塞
- 初始化阶段逻辑过重
- 页面交互和渲染集中在同一阶段执行

建议动作：
- 拆分长任务
- 延后非关键初始化
- 避免首屏集中执行复杂逻辑
- 检查第三方脚本是否可延迟或移除

## 诊断输出规则
- 必须先写证据，再写推断。
- 如果没有直接测量值，但基于截图、结构、瀑布图或资源列表进行判断，必须明确标注为“推断”。
- 建议优先给出“低改动、高收益”的动作。
- 如果问题与多个指标有关，应优先说明它影响的核心指标。
- 不要直接建议整体重构，除非当前问题无法通过局部优化解决。
