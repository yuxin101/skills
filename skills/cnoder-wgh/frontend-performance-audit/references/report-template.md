# 报告模板

请始终按以下结构输出前端性能分析结果。如果用户提供的信息不足，可以保留结构，但要明确说明缺失数据，不要假装已经测到。

## 一、概览
输出以下内容：
- 页面或项目名称
- 分析对象（页面 URL、Lighthouse 报告、截图、构建产物、代码片段等）
- 总体判断：优秀 / 需改进 / 较差
- 最值得优先处理的 3 个问题

## 二、性能指标
优先输出以下指标；没有的数据要明确写“未提供”：
- 最大内容绘制时间（LCP）
- 交互到下次绘制（INP）
- 累积布局偏移（CLS）
- 首次内容绘制（FCP）
- 首字节时间（TTFB）
- DOMContentLoaded（DCL）
- 总阻塞时间（TBT）
- 请求总数
- JavaScript 总体积
- CSS 总体积
- 图片总体积
- 长任务数量

每个指标需要包含：
- 指标名
- 数值
- 判断等级（优秀 / 需改进 / 较差 / 未提供）
- 一句简短解释

如果 DCL 偏高，需要重点分析：
- HTML 体积
- 阻塞脚本
- head 中第三方脚本
- DOM 节点数量

## 三、关键问题
至少输出 3 条关键问题；如果证据不足，输出已有问题并说明置信度。

每条问题都必须包含以下字段：
- 问题标题
- 证据
- 影响范围
- 可能原因
- 建议动作
- 优先级

字段要求如下：

### 问题标题
用一句话说明问题本质，例如：
- 首屏主图加载过慢
- 主线程 JavaScript 负担过重
- 布局抖动明显
- 阻塞渲染资源过多
- DOM 构建耗时偏高

### 证据
必须引用用户提供的数据、截图、报告或页面现象。
如果是推断，必须明确写“推断”。

### 影响范围
使用以下等级之一：
- 高
- 中
- 低

### 可能原因
先给最可能原因，再给次要原因。
不要只写抽象概念，要尽量落到可执行层面。

### 建议动作
必须写成可执行建议，例如：
- 为首屏图片补充 preload
- 将非关键脚本延迟加载
- 为图片和广告位补充固定尺寸
- 拆分首页大体积 bundle
- 延迟第三方埋点脚本执行
- 精简首屏 DOM 结构

### 优先级
使用以下等级之一：
- 高
- 中
- 低

优先级判断原则：
- 能明显改善核心指标且改动成本低 → 高
- 有一定收益但需要中等改动 → 中
- 收益有限或依赖较大改造 → 低

## 四、快速收益项
输出 3 到 5 条“投入小、收益高”的建议。
每条建议包含：
- 动作
- 预期收益
- 适用前提

例如：
- 预加载首屏主图：有助于降低 LCP；适用于首屏大图明确的页面
- 延迟加载聊天和统计脚本：有助于降低 TBT 和 INP；适用于第三方脚本较多的页面
- 精简首屏 DOM 结构：有助于降低 DCL；适用于页面模块很多、层级较深的页面

## 五、工程改造建议
按类别归纳建议，至少包含以下分类：
- 前端层
- 资源层
- 服务端层
- 第三方依赖层

每类下面给出 1 到 3 条建议。

## 六、结论
最后给出简短总结，包含：
- 当前最主要瓶颈是什么
- 最建议优先做哪一步
- 如果还缺数据，下一步最该补什么数据
