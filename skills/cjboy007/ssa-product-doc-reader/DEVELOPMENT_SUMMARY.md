## ⚠️ 踩过的雷（重要教训）

### 1. Google Drive 搜索分页陷阱
**问题：** `gog drive search "599"` 只返回 17 个结果，但实际有 58 个文件。

**原因：** Drive API 有分页限制，部分文件在根目录，部分在 `599` 文件夹内。

**解决：** 逐个搜索 `599-001` 到 `599-058` 才能找到完整列表。

**教训：** 批量文件处理时，不要依赖单一搜索，要分页或逐个获取。

---

### 2. 模具号识别错误
**问题：** 
- 599-030 识别成 `P`（单字符）
- 599-032 识别成 `DP-0.34A`（包含 `.`）

**原因：** 
- pdftotext 输出中模具号与产品名在不同行
- Vision API 的 OCR 把产品名当成了模具号

**解决：** 
- 添加数据校验规则（排除单字符、排除 `.` 格式）
- 优先使用 pdftotext 精确文本
- MODEL NO. 正则支持换行符匹配

**教训：** 关键字段必须校验，不能盲目相信 OCR 结果。

---

### 3. 产品名提取错误
**问题：** 提取到 BOM 词汇（OPEN/SHORT/MISS/CONTACT/RESISTANCE 等）。

**原因：** 产品名解析逻辑没有排除测试要求区域的词汇。

**解决：** 添加排除列表（50+ 个 BOM 常见词汇）。

**教训：** 产品名提取需要结合上下文位置，不能只看格式。

---

### 4. 漏产品（599-008）
**问题：** 599-008 有 3 个产品（30F/50F/100F），只提取到 1 个。

**原因：** 产品名和包装规范在不同行，解析逻辑没关联起来。

**解决：** 跨行收集产品信息（上下 5 行搜索）。

**教训：** PDF 布局多样，需要支持跨行数据关联。

---

### 5. Excel 文件位置搞错
**问题：** 在本地 workspace 处理 Excel，但实际文件在 Google Drive。

**原因：** 没有先确认文件位置就开始处理。

**解决：** 使用 `gog drive search` 确认文件 ID，下载→处理→上传。

**教训：** 先确认文件位置（本地 vs Drive），不要假设。

---

### 6. MODEL NO. 正则不匹配换行
**问题：** `MODEL NO.\n\n5004-6A` 格式无法匹配。

**原因：** 正则 `\s*` 只匹配空白字符，不包括换行符。

**解决：** 使用 `re.MULTILINE` 标志，或显式匹配 `\n`。

**教训：** PDF 文本提取后格式可能变化，正则要考虑换行。

## 📋 核心设计原则（v5.0）

### 1. pdftotext 优先
- 模具号/BOM/产品名优先用 pdftotext（字体嵌入，100% 准确）
- Vision API 仅作为兜底（当 pdftotext 失败时）

### 2. 数据校验
- 模具号必须包含字母 + 数字
- 排除单字符（如 P）
- 排除 BJ 开头（包装规范）
- 排除 599 开头（Drawing No.）
- 产品名长度 > 3 字符

### 3. 跨行关联
- 产品名/长度/包装规范可能在不同行
- 支持上下 5 行搜索关联

### 4. 300 DPI OCR
- Vision API 使用 300 DPI 高分辨率
- 提高小字识别准确率

## 🔧 已知不足与改进建议

### 1. 产品名提取还可以优化
**现状：** 依赖关键词排除列表，可能漏掉新词汇。

**改进：** 
- 用 Vision API 识别表格结构，精确定位产品名区域
- 添加机器学习分类器区分产品名 vs BOM 词汇

### 2. 批量处理无断点续传
**现状：** 58 个文件处理到一半失败，需要重来。

**改进：**
- 记录已处理文件列表
- 支持从断点继续

### 3. 特殊模板适配
**现状：** 客户图纸（如 C331 模板）字段位置不同，需要手动调整。

**改进：**
- 自动检测模板类型（福睿标准/客户模板）
- 根据模板自适应调整解析逻辑

### 4. 无自动校验机制
**现状：** 需要人工检查提取结果。

**改进：**
- 添加置信度阈值（<80% 标记为需人工审核）
- 与 OKKI 产品库交叉验证

### 5. 知识库归档格式单一
**现状：** 只生成 Markdown。

**改进：**
- 支持 Obsidian Dataview 格式
- 自动生成产品对比表
- 支持导出 Excel/CSV
